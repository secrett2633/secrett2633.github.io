9:Tc06,<h2>비관적 락 (Pessimistic Lock)</h2>
<h3>비관적 락이란?</h3>
<p>비관적 락은 <strong>데이터베이스에서 동시성 제어</strong> 를 위해 락을 사용하여 작업을 처리하는 방법입니다. 이 방식은 <strong>데이터 정합성</strong> 을 강하게 보장하지만, 락을 획득하고 있는 동안 다른 작업들이 대기하거나 영향을 받을 수 있는 단점이 있습니다. 이는 주로 충돌 가능성이 높거나 데이터의 정합성이 매우 중요한 상황에서 사용됩니다.</p>
<h3>Django에서 비관적 락 사용하기</h3>
<p>Django에서는 비관적 락을 <strong><code>select_for_update</code></strong> 메서드를 통해 구현할 수 있습니다. 이 메서드는 특정 레코드에 대해 <strong>락을 걸어두고</strong> 다른 트랜잭션이 해당 데이터를 수정하지 못하도록 합니다.</p>
<h4>예시 코드:</h4>
<pre><code class="language-python">from django.db import transaction

with transaction.atomic():
    # 데이터를 잠그고, id가 1인 Category 객체를 가져옵니다.
    category = Category.objects.select_for_update().get(id=1)
    
    # 데이터 수정
    category.name = "new_name"
    
    # 수정된 데이터 저장
    category.save()
</code></pre>
<h3>코드 설명</h3>
<ul>
<li>
<p><strong><code>select_for_update()</code></strong> : 이 메서드는 선택된 데이터베이스 레코드를 <strong>잠금 상태</strong> 로 만들며, 다른 트랜잭션이 해당 데이터에 접근할 수 없도록 합니다. 다른 트랜잭션은 해당 데이터에 대한 작업이 완료될 때까지 대기해야 합니다.</p>
</li>
<li>
<p><strong><code>transaction.atomic()</code></strong> : 이 컨텍스트 매니저는 트랜잭션의 <strong>원자성</strong> 을 보장합니다. 모든 작업이 완료된 후에 <strong>커밋</strong> 이 이루어지며, 만약 도중에 오류가 발생하면 모든 작업을 <strong>롤백</strong> 합니다. 따라서 위 예시에서 <code>save()</code>를 여러 번 수행하는 상황에서, 중간에 오류가 발생하면 <strong>첫 번째 작업</strong> 은 롤백되고, 두 번째 작업은 진행되지 않습니다.</p>
</li>
</ul>
<p><code>select_for_update</code>는 반드시 <code>transaction.atomic()</code> 블록 안에서 사용해야 합니다. 이 둘이 함께 사용되지 않으면 select_for_update 가 적용되지 않습니다.</p>
<h3>비관적 락의 장단점</h3>
<h4>장점:</h4>
<ul>
<li><strong>강력한 데이터 정합성 보장</strong> : 락을 사용하여 데이터가 동시에 수정되지 않도록 보장합니다.</li>
</ul>
<h4>단점:</h4>
<ul>
<li><strong>성능 저하</strong> : 락을 걸어둔 상태에서 다른 트랜잭션이 대기해야 하기 때문에, 성능이 저하될 수 있습니다.</li>
<li><strong>교착 상태(Deadlock)</strong> : 여러 트랜잭션이 상호 의존적인 락을 걸 경우, 교착 상태가 발생할 수 있습니다. 이럴 때는 적절한 락 해제 및 관리가 필요합니다.</li>
</ul>
