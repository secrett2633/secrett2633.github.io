<p><strong>원문 링크:</strong> <a href="https://peps.python.org/pep-3129/">PEP 3129 - Class Decorators</a></p>
<p><strong>상태:</strong> Final | <strong>유형:</strong> Standards Track | <strong>작성일:</strong> 01-May-2007</p>
</blockquote>
<h1>PEP 3129 – 클래스 데코레이터</h1>
<h2>개요 (Abstract)</h2>
<p>이 PEP(Python Enhancement Proposal)는 PEP 318에서 도입된 함수 및 메서드 데코레이터의 확장인 클래스 데코레이터를 제안합니다.</p>
<h2>도입 배경 (Rationale)</h2>
<p>함수 데코레이터가 Python 2.4에 포함될지에 대해 처음 논의되었을 때, 클래스 데코레이터는 메타클래스(metaclasses) 덕분에 모호하고 불필요하다고 여겨졌습니다. 그러나 Python 2.4.x 시리즈 릴리스를 통해 수년간 경험을 쌓고 함수 데코레이터와 그 활용법에 대한 친숙도가 높아지면서, BDFL(Benevolent Dictator For Life, 귀도 반 로썸)과 커뮤니티는 클래스 데코레이터를 재평가하여 Python 3.0에 포함할 것을 권고했습니다.</p>
<p>클래스 데코레이터의 주요 동기는 특정 구문들을 더 쉽게 표현하고 CPython 인터프리터의 구현 세부 사항에 덜 의존하도록 하는 것이었습니다. 메타클래스를 사용하여 클래스 데코레이터와 유사한 기능을 표현하는 것이 가능했지만, 그 결과는 일반적으로 불쾌했고 구현이 매우 취약했습니다. 또한, 메타클래스는 상속되는 반면, 클래스 데코레이터는 상속되지 않으므로 특정 단일 클래스에만 적용되는 클래스 데코레이터의 용도에는 메타클래스가 적합하지 않았습니다. Zope와 같은 대규모 Python 프로젝트들이 클래스 데코레이터와 유사한 기능을 달성하기 위해 복잡한 방식들을 사용하고 있다는 사실이 BDFL을 설득했습니다.</p>
<h2>의미론 (Semantics)</h2>
<p>클래스 데코레이터의 의미론(semantics)과 설계 목표는 함수 데코레이터(PEP 318)와 동일합니다. 유일한 차이점은 함수 대신 클래스를 데코레이트(decorate)한다는 것입니다. 다음 두 코드 스니펫은 의미론적으로 동일합니다.</p>
<pre><code class="language-python">class A:
    pass
A = foo(bar(A))
</code></pre>
<pre><code class="language-python">@foo
@bar
class A:
    pass
</code></pre>
<p>데코레이터에 대한 자세한 내용은 PEP 318을 참조하십시오.</p>
<h2>구현 (Implementation)</h2>
<p>클래스 데코레이터를 지원하기 위해 Python의 문법(grammar)을 조정하려면 두 가지 규칙을 수정하고 새로운 규칙을 추가해야 합니다.</p>
<p><code>funcdef: [decorators] 'def' NAME parameters ['->' test] ':' suite</code>
<code>compound_stmt: if_stmt | while_stmt | for_stmt | try_stmt | with_stmt | funcdef | classdef</code></p>
<p>위 규칙들을 다음으로 변경해야 합니다.</p>
<p><code>decorated: decorators (classdef | funcdef)</code>
<code>funcdef: 'def' NAME parameters ['->' test] ':' suite</code>
<code>compound_stmt: if_stmt | while_stmt | for_stmt | try_stmt | with_stmt | funcdef | classdef | decorated</code></p>
<p><code>decorated</code> 규칙을 추가하는 것은 문법의 모호성을 피하기 위해 필수적입니다.</p>
<p>Python AST(추상 구문 트리) 및 바이트코드(bytecode)도 이에 맞춰 수정되어야 합니다.</p>
<p>참조 구현은 Jack Diederich에 의해 제공되었습니다.</p>
<h2>수용 (Acceptance)</h2>
<p>이 PEP가 게시된 이후 사실상 아무런 논의도 없었으며, 이는 모든 사람이 이 제안이 수용되어야 한다는 데 동의했음을 의미합니다.</p>
