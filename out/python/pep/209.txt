<h1 id="pep-209-다차원-배열-multi-dimensional-arrays"><a href="#pep-209-다차원-배열-multi-dimensional-arrays">PEP 209 – 다차원 배열 (Multi-dimensional Arrays)</a></h1>
<ul>
<li><strong>저자:</strong> Paul Barrett, Travis Oliphant</li>
<li><strong>상태:</strong> 철회됨 (Withdrawn)</li>
<li><strong>유형:</strong> 표준 트랙 (Standards Track)</li>
<li><strong>생성일:</strong> 2001년 1월 3일</li>
<li><strong>Python 버전:</strong> 2.2</li>
<li><strong>Post-History:</strong> N/A (정보 없음)</li>
</ul>
<h2 id="요약-abstract"><a href="#요약-abstract">요약 (Abstract)</a></h2>
<p>이 PEP는 다차원 배열 모듈인 <code>Numeric</code>을 재설계하고 재구현하여 새로운 기능과 확장을 더 쉽게 추가할 수 있도록 제안합니다. <code>Numeric 2</code>의 주요 초점은 기가바이트를 초과하는 대규모 배열과 이기종 데이터 구조(inhomogeneous data structures) 또는 레코드(records)에 대한 효율적인 접근이었습니다. 제안된 설계는 네 가지 Python 클래스(<code>ArrayType</code>, <code>UFunc</code>, <code>Array</code>, <code>ArrayView</code>)와 효율적인 배열 연산을 처리하기 위한 저수준 C-확장 모듈 <code>_ufunc</code>를 사용합니다. 또한, 각 배열 타입은 자체 C-확장 모듈을 가지며, 이는 해당 타입의 강제 변환(coercion) 규칙, 연산 및 메서드를 정의합니다. 이 설계는 새로운 타입, 기능 및 확장을 모듈 방식으로 추가할 수 있도록 합니다. 새 버전은 기존 <code>Numeric</code>과의 일부 비호환성을 도입할 예정이었습니다.</p>
<h2 id="동기-motivation"><a href="#동기-motivation">동기 (Motivation)</a></h2>
<p>다차원 배열은 과학, 공학 및 컴퓨팅 분야에서 데이터를 저장하고 조작하는 데 일반적으로 사용됩니다. Python에는 현재 <code>Numeric</code> (이후 <code>Numeric 1</code>)이라는 확장 모듈이 있으며, 이는 중간 크기(약 10MB)의 동종(homogeneous) 데이터 배열을 조작하는 사용자에게 만족스러운 기능을 제공합니다. 그러나 더 큰 배열(100MB 이상)과 이기종 데이터에 접근하는 경우, <code>Numeric 1</code>의 구현은 비효율적이고 번거롭습니다. 미래에는 PEP 211, PEP 225에서 보듯이 Numerical Python 커뮤니티의 추가 기능 요청이 많아질 것으로 예상되었습니다.</p>
<h2 id="제안-proposal"><a href="#제안-proposal">제안 (Proposal)</a></h2>
<p>이 제안은 <code>Numeric 1</code>을 <code>Numeric 2</code>로 재설계하고 재구현할 것을 권장하며, 이를 통해 새로운 타입, 기능 및 확장을 쉽고 모듈화된 방식으로 추가할 수 있게 합니다. <code>Numeric 2</code>의 초기 설계는 다양한 타입의 배열을 조작하기 위한 일반적인 프레임워크를 제공하고, 새로운 배열 타입과 <code>UFunc</code>를 추가하는 간단한 메커니즘을 가능하게 하는 데 중점을 두어야 합니다. 다양한 분야에 더 특화된 기능적 메서드는 이 핵심 위에 계층화될 수 있습니다. 이 새로운 모듈은 여전히 <code>Numeric</code>이라고 불리며 <code>Numeric 1</code>에서 발견되는 대부분의 동작은 유지될 예정이었습니다.</p>
<p>제안된 설계는 <code>ArrayType</code>, <code>UFunc</code>, <code>Array</code>, <code>ArrayView</code>의 네 가지 Python 클래스와 배열 연산을 효율적으로 처리하는 저수준 C-확장 모듈을 사용합니다. 각 배열 타입은 해당 타입의 강제 변환 규칙, 연산 및 메서드를 정의하는 자체 C-확장 모듈을 가집니다. 핵심 기능이 안정화되면 일부 Python 클래스는 C-확장 타입으로 변환될 수 있습니다.</p>
<p>몇 가지 계획된 기능은 다음과 같습니다:</p>
<ul>
<li><strong>개선된 메모리 사용량 (Improved memory usage):</strong> 대규모 배열 처리 시 특히 중요하며, 성능 및 메모리 사용량을 크게 개선할 수 있습니다.
<ul>
<li><strong>로컬 강제 변환 모델 사용 (Use a local coercion model):</strong> <code>PEP 208</code>에 설명된 대로 Python의 전역 강제 변환 모델 대신 연산자에게 강제 변환 책임을 위임하는 로컬 모델을 구현합니다.</li>
<li><strong>임시 배열 생성 방지 (Avoid creation of temporary arrays):</strong> 복잡한 배열 표현식에서 임시 배열의 데이터 버퍼를 재사용하여 성능을 향상시킵니다.</li>
<li><strong>메모리 맵 파일의 선택적 사용 (Optional use of memory-mapped files):</strong> 사용 가능한 메모리보다 큰 데이터를 처리하거나 매우 큰 파일의 데이터에 접근하기 위해 메모리 맵 배열을 투명하게 사용할 수 있도록 합니다.</li>
</ul>
</li>
<li><strong>레코드 접근 (Record access):</strong> 이기종 데이터를 저장하는 레코드 또는 C-like 구조에 대한 접근을 지원합니다. 바이트 정렬 및 바이트 스와핑(byte swapping)과 같은 문제를 자동으로 처리하도록 설계되었습니다.</li>
<li><strong>추가 배열 타입 (Additional array types):</strong> <code>ushort</code>, <code>uint</code>, <code>ulong</code>과 같은 부호 없는 정수 타입이나 비트(bit) 타입과 같은 새로운 배열 타입을 쉽게 추가할 수 있도록 <code>Numeric</code>의 재설계가 필요합니다.</li>
<li><strong>비트 타입 (Bit type):</strong> 배열 간의 풍부한 비교(rich comparison) 결과로 생성되는 불리언(boolean) 값을 저장하는 데 <code>char</code> 타입 대신 비트 타입을 사용하여 메모리 사용량을 1/8로 줄입니다.</li>
<li><strong>향상된 배열 인덱싱 문법 (Enhanced array indexing syntax):</strong> 불규칙한 간격의 인덱스 목록이 필요한 경우 1차원 배열을 인수로 허용하는 향상된 배열 인덱싱 문법을 제안합니다.</li>
<li><strong>풍부한 비교 (Rich comparisons):</strong> <code>PEP 207</code>의 <code>Rich Comparisons</code> 기능을 <code>Numeric 2</code>에 구현하여 배열 조작 시 유연성을 높입니다.</li>
<li><strong>배열 브로드캐스팅 규칙 (Array broadcasting rules):</strong> 스칼라와 배열 간의 연산에서 스칼라 값이 배열의 형태를 가지는 새로운 배열로 확장되는 <code>array broadcasting</code> 동작을 구현합니다.</li>
</ul>
<h2 id="설계-및-구현-design-and-implementation"><a href="#설계-및-구현-design-and-implementation">설계 및 구현 (Design and Implementation)</a></h2>
<p><code>Numeric 2</code>의 설계는 네 가지 주요 클래스를 가집니다:</p>
<h3 id="arraytype"><a href="#arraytype">ArrayType</a></h3>
<p>배열 타입의 기본 속성(이름, 바이트 단위 크기, 다른 타입과의 강제 변환 관계 등)을 설명하는 간단한 클래스입니다.</p>
<ul>
<li><strong>속성 (Attributes):</strong> <code>.name</code>, <code>.typecode</code> (하위 호환성), <code>.size</code>, <code>.array_rules</code>, <code>.pyobj_rules</code>, <code>.doc</code>.</li>
<li><strong>메서드 (Methods):</strong> <code>__init__()</code>, <code>__del__()</code>, <code>__repr__()</code>.</li>
<li><strong>C-API:</strong> 추가 정의 필요.</li>
</ul>
<h3 id="ufunc"><a href="#ufunc">UFunc</a></h3>
<p><code>Numeric 2</code>의 핵심 클래스입니다. <code>ArrayType</code>과 유사하게, 이름, 총 인수 수, 입력 인수 수, 문서 문자열 및 비어 있는 <code>CFunc</code> 사전을 속성으로 가지는 단일 호출 가능 객체(<code>singleton callable object</code>)를 생성합니다. <code>__call__</code> 메서드는 배열에 연산이 적용될 때 호출되며, 인수의 타입을 확인하고 <code>CFunc</code> 사전을 통해 적절한 C 함수를 찾아 연산을 수행합니다.</p>
<ul>
<li><strong>속성 (Attributes):</strong> <code>.name</code>, <code>.nargs</code>, <code>.iargs</code>, <code>.cfuncs</code>, <code>.doc</code>.</li>
<li><strong>메서드 (Methods):</strong> <code>__init__()</code>, <code>__del__()</code>, <code>__repr__()</code>, <code>__call__()</code>, <code>initrule()</code>, <code>uninitrule()</code>, <code>register()</code>, <code>unregister()</code>.</li>
<li><strong>C-API:</strong> 추가 정의 필요.</li>
</ul>
<h3 id="array"><a href="#array">Array</a></h3>
<p>배열의 형상(shape), 타입, 데이터의 엔디안(endianness)과 같은 정보를 포함하는 클래스입니다. <code>+</code>, <code>-</code> 등의 연산자는 해당 <code>UFunc</code> 함수를 호출합니다.</p>
<ul>
<li><strong>속성 (Attributes):</strong> <code>.shape</code>, <code>.format</code>, <code>.real</code> (복소수만 해당), <code>.imag</code> (복소수만 해당).</li>
<li><strong>메서드 (Methods):</strong> <code>__init__()</code>, <code>__del__()</code>, <code>__repr__()</code>, <code>__str__()</code>, <code>__cmp__()</code>, <code>__len__()</code>, <code>__getitem__()</code>, <code>__setitem__()</code>, <code>__getslice__()</code>, <code>__setslice__()</code>, <code>copy()</code>, <code>aslist()</code>, <code>asstring()</code>.</li>
<li><strong>함수 (Functions):</strong> <code>fromlist()</code>, <code>fromstring()</code>, <code>array()</code>, <code>concat()</code>, <code>resize()</code>.</li>
<li><strong>C-API:</strong> 추가 정의 필요.</li>
</ul>
<h3 id="arrayview"><a href="#arrayview">ArrayView</a></h3>
<p><code>Array</code> 클래스와 유사하지만, <code>reshape</code> 및 <code>flat</code> 메서드는 예외를 발생시킵니다. 이는 비연속적인 배열은 포인터와 스텝(step-size) 정보만으로는 재형상화(reshape)하거나 평탄화(flatten)할 수 없기 때문입니다.</p>
<ul>
<li><strong>C-API:</strong> 추가 정의 필요.</li>
</ul>
<h3 id="c-확장-모듈-c-extension-modules"><a href="#c-확장-모듈-c-extension-modules">C-확장 모듈 (C-extension modules)</a></h3>
<p><code>Numeric 2</code>는 여러 C-확장 모듈을 가질 예정이었습니다.</p>
<ul>
<li><code>_ufunc</code>: 최소한의 기능, 즉 지정된 C 함수를 사용하여 배열을 반복(iterate)하는 역할을 담당합니다.</li>
<li><code>_int32</code>, <code>_real64</code> 등: 각 배열 타입별 C-확장 모듈로, <code>UFunc</code> 모듈에 의해 임포트될 때 자동으로 함수와 강제 변환 규칙을 등록합니다.</li>
</ul>
<h2 id="미해결-과제-open-issues"><a href="#미해결-과제-open-issues">미해결 과제 (Open Issues)</a></h2>
<p><code>PEP 209</code>는 다음과 같은 여러 미해결 과제를 언급했습니다:</p>
<ul>
<li><strong>슬라이싱 문법의 기본 동작 (Does slicing syntax default to copy or view behavior?):</strong> Python 리스트와 일관성을 위해 복사(copy) 동작을 할 것인지, <code>Numeric 1</code>처럼 성능을 위해 뷰(view) 동작을 할 것인지에 대한 논의가 있었습니다.</li>
<li><strong>아이템 문법의 기본 동작 (Does item syntax default to copy or view behavior?):</strong> 슬라이싱 문법과 유사하게, <code>a[0]</code>과 같은 아이템 접근 시 복사본을 반환할지 뷰를 반환할지에 대한 문제입니다.</li>
<li><strong>스칼라 강제 변환 구현 방식 (How is scalar coercion implemented?):</strong> Python의 <code>float</code> 타입이 <code>double</code>이기 때문에 <code>Numeric</code> 배열이 예상치 않게 <code>double</code>로 변환되는 문제를 해결하기 위한 스칼라 강제 변환 규칙 정의가 필요했습니다.</li>
<li><strong>정수 나눗셈 처리 방식 (How is integer division handled?):</strong> Python 3에서 정수 나눗셈의 동작이 변경됨에 따라 <code>Numeric 2</code>에서 <code>//</code> 연산의 일관성을 어떻게 유지할지에 대한 문제입니다.</li>
<li><strong>레코드 구현 방식 (How should records be implemented?):</strong> 레코드를 별도의 배열 클래스로 구현할지, 아니면 다양한 배열 타입 중 하나로 구현할지에 대한 접근 방식 논의가 있었습니다.</li>
<li><strong>레코드 시맨틱스 정의 및 구현 방식 (How are record semantics defined and implemented?):</strong> 레코드의 하위 필드에 접근하고 조작하는 문법 및 의미론에 대한 결정이 필요했습니다.</li>
<li><strong>마스크드 배열 구현 방식 (How are masked-arrays implemented?):</strong> <code>Numeric 1</code>에서는 별도의 배열 클래스였지만, <code>Numeric 2</code>에서는 새로운 배열 타입으로 구현될 가능성이 있었습니다.</li>
<li><strong>수치 오류 처리 방식 (How are numerical errors handled?):</strong> <code>divide-by-zero</code>, <code>underflow</code>, <code>overflow</code>와 같은 수치 오류를 어떻게 처리할지 (예: 메시지 출력, 오류 추적 등)에 대한 여러 옵션이 제시되었습니다.</li>
<li><strong>FORTRAN 라이브러리 및 코드 통합을 용이하게 하는 데 필요한 기능 (What features are needed to ease the integration of FORTRAN libraries and code?):</strong> FORTRAN 코드와의 통합을 용이하게 하는 기능에 대한 고려가 필요했습니다.</li>
</ul>
<h2 id="구현-단계-implementation-steps"><a href="#구현-단계-implementation-steps">구현 단계 (Implementation Steps)</a></h2>
<p>제안된 구현 단계는 다음과 같습니다:</p>
<ol>
<li><strong>기본 <code>UFunc</code> 기능 구현:</strong> 최소한의 <code>Array</code>, <code>ArrayType</code>, <code>UFunc</code> 클래스 및 <code>_UFunc</code> C-확장 모듈을 구현하여 1차원 배열의 기본 연산(예: <code>c = add(a, b)</code>)을 가능하게 합니다.</li>
<li><strong><code>UFunc</code> 이터레이터 및 <code>Array</code> 클래스 지속적으로 개선:</strong> <code>print</code>, <code>repr</code>, <code>getitem</code>, <code>setitem</code>과 같은 <code>Array</code> 클래스의 접근 메서드를 구현하고, 다차원 배열을 지원하며, <code>UFunc</code>를 사용하여 기본 배열 메서드를 구현합니다.</li>
<li><strong>표준 <code>UFunc</code> 및 <code>Array</code> 클래스 동작 완료:</strong> <code>getslice</code>, <code>setslice</code> 동작, 배열 브로드캐스팅 규칙, 레코드 타입 등을 구현합니다.</li>
<li><strong>추가 기능 추가:</strong> 더 많은 <code>UFunc</code>와 버퍼 또는 <code>mmap</code> 접근을 구현합니다.</li>
</ol>
<h2 id="비호환성-incompatibilities"><a href="#비호환성-incompatibilities">비호환성 (Incompatibilities)</a></h2>
<p><code>Numeric 1</code>과 <code>Numeric 2</code> 간의 행동 비호환성은 다음과 같았습니다:</p>
<ul>
<li><strong>스칼라 강제 변환 규칙 (Scalar coercion rules):</strong> <code>Numeric 1</code>의 단일 강제 변환 규칙 대신, <code>Numeric 2</code>는 배열과 Python 숫자 타입용, 그리고 배열 전용의 두 가지 강제 변환 규칙을 가질 예정이었습니다.</li>
<li><strong><code>savespace</code> 속성 없음 (No savespace attribute):</strong> <code>Numeric 1</code>의 <code>savespace</code> 속성은 <code>Numeric 2</code>에 없으며, 따라서 일반 배열 강제 변환 규칙이 적용될 예정이었습니다.</li>
<li><strong>슬라이싱 문법이 복사본 반환 (Slicing syntax returns a copy):</strong> <code>Numeric 1</code>의 슬라이싱 문법은 원본 배열의 뷰를 반환했지만, <code>Numeric 2</code>는 복사본을 반환할 예정이었습니다. 뷰를 얻으려면 <code>ArrayView</code> 클래스를 사용해야 할 것이었습니다.</li>
<li><strong>불리언 비교가 불리언 배열 반환 (Boolean comparisons return a boolean array):</strong> Python의 제약으로 인해 <code>Numeric 1</code>에서는 배열 간 비교 결과가 불리언 스칼라였지만, Python 2.1의 <code>Rich Comparisons</code> 도입으로 불리언 배열을 반환할 수 있게 될 예정이었습니다.</li>
<li><strong>타입 문자가 더 이상 사용되지 않음 (Type characters are deprecated):</strong> <code>Numeric 2</code>는 <code>Int8</code>, <code>Int16</code>, <code>Int32</code>와 같은 <code>ArrayType</code> 클래스를 가질 예정이었으며, <code>Numeric 1</code>의 <code>typecode</code> 체계는 하위 호환성을 위해 제공되지만 더 이상 사용되지 않을 예정이었습니다.</li>
</ul>

<p><code>J</code> 언어를 기반으로 <code>Numeric 1</code> 개발자들이 구현하려 했으나 완성하지 못했던 기능으로, 다차원 배열에서 하위 배열에 대한 연산을 암시적으로 수행하는 방식이었습니다. <code>Numeric 2</code>에서 배열 브로드캐스팅 규칙이 이 동작을 완전히 지원하지 않는다면 구현될 예정이었습니다.</p>
