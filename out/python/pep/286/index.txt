<p><strong>원문 링크:</strong> <a href="https://peps.python.org/pep-0286/">PEP 286 - Enhanced Argument Tuples</a></p>
<p><strong>상태:</strong> Deferred | <strong>유형:</strong> Standards Track | <strong>작성일:</strong> 03-Mar-2002</p>
</blockquote>
<hr>
<p>PEP 286: Enhanced Argument Tuples</p>
<h2>개요</h2>
<p>PEP 286은 <code>PyArg_ParseTuple</code> 함수가 인자 변환기(argument converter)를 통해 새로운 메모리를 생성할 때 발생하는 어려운 메모리 관리 문제를 해결하기 위해 제안되었습니다. 이 문제를 해결하기 위해 특수화된 인자 타입(argument type)의 도입이 제안되었습니다.</p>
<h2>PEP 연기 (PEP Deferral)</h2>
<p>이 PEP에서 다루는 개념에 대한 추가 탐색은 현재 이 PEP의 목표를 추진하고 피드백을 수집 및 통합하며, 이를 효과적으로 수행할 충분한 시간을 가진 '챔피언(champion)'의 부재로 인해 연기되었습니다. 이 PEP의 해결은 C API 인터페이스 코드의 일부 측면을 생성하기 위한 전처리 단계 사용을 제안하는 PEP 426의 해결에도 영향을 받을 수 있습니다.</p>
<h2>문제 설명 (Problem Description)</h2>
<p>현재, 인자 튜플(argument tuples)은 함수 인자에 대한 참조를 유지하며, 이 인자들은 최소한 함수 호출이 실행되는 동안 인자 튜플이 존재하는 한 유지됩니다.</p>
<p>어떤 경우, 인자를 파싱하는 과정에서 새로운 메모리를 할당하게 되는데, 이 메모리는 호출자(caller)가 해제해야 합니다. 여기서 두 가지 문제가 발생합니다.</p>
<ol>
<li><strong>실패 시 메모리 해제 문제:</strong> 파싱 실패 시, 애플리케이션은 어떤 메모리를 해제해야 하는지 알 수 없습니다. 대부분의 호출자는 자신이 해당 메모리를 해제할 책임이 있다는 사실조차 알지 못합니다. <code>N</code> 변환기(bug #416288)와 <code>es#</code> 변환기(bug #501716)가 이러한 사례에 해당합니다.</li>
<li><strong>성공 시 불편한 메모리 관리:</strong> 인자 파싱이 성공하더라도, 호출자가 메모리 해제를 담당하는 것은 여전히 불편합니다. 어떤 경우에는 불필요하게 비효율적입니다. 예를 들어, <code>es</code> 변환기는 올바른 내용을 가진 문자열 객체가 이미 존재함에도 불구하고 변환 결과를 메모리에 복사합니다.</li>
</ol>
<h2>제안된 해결책 (Proposed Solution)</h2>
<p>새로운 타입인 'argument tuple'이 도입됩니다. 이 타입은 <code>tuple</code>에서 파생되며, <code>__dict__</code> 멤버(<code>tp_dictoffset -4</code> 위치)를 추가합니다. 이 타입의 인스턴스는 다음 속성들을 가질 수 있습니다.</p>
<ul>
<li><code>'failobjects'</code>: 성공 시 할당 해제되어야 하는 객체들의 리스트입니다.</li>
<li><code>'okobjects'</code>: 인자 튜플이 해제될 때 해제될 객체들의 리스트입니다.</li>
</ul>
<p>이 타입을 관리하기 위해 다음과 같은 함수들이 추가되며, <code>ceval.c</code> 및 <code>getargs.c</code>에서 적절하게 사용될 것입니다.</p>
<ul>
<li><code>PyArgTuple_New(int);</code></li>
<li><code>PyArgTuple_AddFailObject(PyObject*, PyObject*);</code></li>
<li><code>PyArgTuple_AddFailMemory(PyObject*, void*);</code></li>
<li><code>PyArgTuple_AddOkObject(PyObject*, PyObject*);</code></li>
<li><code>PyArgTuple_AddOkMemory(PyObject*, void*);</code></li>
<li><code>PyArgTuple_ClearFailed(PyObject*);</code></li>
</ul>
<p>인자 파싱이 실패하면, 모든 <code>failobjects</code>는 <code>Py_DECREF</code>를 통해 해제되고, 모든 <code>fail memory</code>는 <code>PyMem_Free</code>를 통해 해제됩니다. 파싱이 성공하면, <code>fail objects</code> 및 <code>fail memory</code>에 대한 참조는 아무것도 해제하지 않고 제거됩니다.</p>
<p>인자 튜플이 해제될 때, 모든 <code>ok objects</code>와 메모리가 해제됩니다.</p>
<p>이 함수들이 다른 타입의 객체와 함께 호출되면 경고가 발생하고 추가 조치는 취해지지 않습니다. 인자 튜플을 사용하지 않고 영향을 받는 변환기를 사용하는 것은 더 이상 사용되지 않습니다(deprecated).</p>
<h2>영향을 받는 변환기 (Affected Converters)</h2>
<p>다음 변환기들은 <code>fail memory</code> 및 <code>fail objects</code>를 추가할 것입니다.</p>
<ul>
<li><code>N</code></li>
<li><code>es</code></li>
<li><code>et</code></li>
<li><code>es#</code></li>
<li><code>et#</code> (변환기에 메모리가 전달되지 않는 경우)</li>
</ul>
<h2>새로운 변환기 (New Converters)</h2>
<p>유니코드(Unicode) 변환을 단순화하기 위해, <code>e*</code> 변환기들은 <code>E*</code> 변환기들(예: <code>Es</code>, <code>Et</code>, <code>Es#</code>, <code>Et#</code>)로 복제됩니다. <code>E*</code> 변환기들의 사용법은 <code>e*</code> 변환기들과 동일하지만, 애플리케이션이 결과 메모리를 직접 관리할 필요가 없다는 점이 다릅니다. 이는 인자 튜플에 <code>Ok objects</code>를 등록함으로써 구현될 것입니다. <code>e*</code> 변환기들은 더 이상 사용되지 않습니다(deprecated).</p>
