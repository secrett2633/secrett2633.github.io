<h1>PEP 101 – Python 릴리스 수행 101</h1>
<p><strong>저자:</strong> Barry Warsaw, Guido van Rossum
<strong>상태:</strong> 활성 (Active)
<strong>유형:</strong> 정보 (Informational)
<strong>생성일:</strong> 2001년 8월 22일
<strong>이전 문서:</strong> PEP 102</p>
<h2>초록 (Abstract)</h2>
<p>Python 릴리스를 만드는 것은 흥미진진하면서도 복잡한 과정입니다. "고양이 떼를 모으는 것"이라는 표현을 들어보셨을 겁니다. 이 과정은 마치 그 고양이들을 안장 위에 태우고 도시로 데려가는 것과 같으며, 몇몇 고양이는 새로 날카롭게 간 발톱으로 등 뒤에 단단히 붙어 있는 상황을 상상해 볼 수 있습니다. 적어도 그들은 귀엽다고 스스로를 다독이게 될 것입니다.</p>
<p>사실, 이것은 약간의 과장입니다 😉. Python 릴리스 프로세스는 수년 동안 꾸준히 개선되어 왔으며, 이제는 놀라운 커뮤니티의 도움 덕분에 그리 어렵지 않습니다. 이 PEP는 Python 릴리스를 만드는 데 필요한 모든 단계를 한곳에 모으려고 합니다. 대부분의 단계는 현재 자동화되거나 자동화의 안내를 받으므로, 이 목록을 수동으로 따를 필요는 더 이상 없습니다.</p>
<h2>필요한 것들 (Things You'll Need)</h2>
<p>릴리스 관리자는 많은 리소스에 접근해야 합니다. 다음은 필요한 것들의 목록입니다.</p>
<ul>
<li><strong>GPG 키:</strong> Python 3.14 이전 릴리스는 GPG로 디지털 서명됩니다. 이를 위해서는 키가 필요하며, 이 키는 다른 릴리스 관리자 중 적어도 한 명과 "신뢰의 웹(web of trust)"에 포함되어야 합니다.
<ul>
<li><strong>참고:</strong> Python 3.14 이후부터는 이 PEP의 GPG 지침을 무시해도 됩니다. 자세한 내용은 <a href="https://peps.python.org/pep-0761/">PEP 761</a>을 참조하십시오.</li>
</ul>
</li>
<li><strong>소프트웨어:</strong>
<ul>
<li><code>python/release-tools</code> 저장소 체크아웃. 이 저장소에는 종속성을 먼저 설치해야 하는 <code>requirements.txt</code> 파일이 포함되어 있습니다. 그 후 이 PEP에서 나중에 다룰 스크립트를 실행할 수 있습니다.</li>
<li><code>blurb</code>, Misc/NEWS 관리 도구. <code>pip install blurb</code>로 설치할 수 있습니다.</li>
</ul>
</li>
<li><strong>파일을 업로드할 서버 접근 권한:</strong>
<ul>
<li><code>downloads.nyc1.psf.io</code>: 다운로드 파일을 호스팅하는 서버.</li>
<li><code>docs.nyc1.psf.io</code>: 문서를 호스팅하는 서버.</li>
</ul>
</li>
<li><code>python/cpython</code>에 대한 관리자 접근 권한.</li>
<li><code>www.python.org</code>에 대한 관리자 계정, "API 키" 포함.</li>
<li><code>python/peps</code> 저장소에 대한 쓰기 권한. (이 문서를 읽고 있다면 아마 이미 가지고 있을 것입니다. 릴리스 관리자의 첫 번째 임무는 릴리스 일정을 초안하는 것이기 때문입니다.)</li>
<li>Blogger에서 호스팅되는 웹로그인 <code>blog.python.org</code>에 대한 게시 권한. 이 블로그의 RSS 피드는 <code>www.python.org</code>의 'Python News' 섹션에 사용됩니다.</li>
<li><code>python-cabal</code>이라고 불릴 수도 있고 아닐 수도 있는 극비 릴리스 관리자 메일링 리스트 구독. Barry에게 문의하십시오.</li>
<li>릴리스 서명에 사용할 <code>@python.org</code> 이메일 주소. <code>postmaster@</code>에게 주소를 요청할 수 있습니다. 전체 계정을 받거나, 리디렉션 별칭과 SMTP 자격 증명을 받아 주요 이메일 제공업체에 합법적으로 보이는 이메일을 보낼 수 있습니다.</li>
<li>Python Security Response Team에 추가될 것.</li>
</ul>
<h2>릴리스 유형 (Types of Releases)</h2>
<p>만들어야 할 몇 가지 릴리스 유형이 있습니다. 여기에는 다음이 포함됩니다.</p>
<ul>
<li><code>alpha</code></li>
<li><code>begin beta</code> (beta 1 또는 new branch라고도 함)</li>
<li><code>beta 2+</code></li>
<li><code>release candidate 1</code></li>
<li><code>release candidate 2+</code></li>
<li><code>final</code></li>
<li><code>new branch</code></li>
<li><code>begin bugfix mode</code></li>
<li><code>begin security-only mode</code></li>
<li><code>end-of-life</code></li>
</ul>
<p>이러한 릴리스 유형 중 일부는 둘 이상의 릴리스 브랜치(branch)를 포함합니다. 특히, "새 브랜치(new branch)"는 새로운 기능 릴리스 사이클이 시작되는 릴리스 사이클의 지점입니다. 현재 CPython Git 저장소의 구성에 따르면 <code>main</code> 브랜치는 항상 새로운 기능의 대상이 됩니다. 다음 기능 릴리스의 릴리스 사이클의 어느 시점에서, 새로운 브랜치 릴리스가 만들어져 현재 진행 중인 기능 릴리스(예: <code>3.n.0</code>)의 안정화 및 향후 유지보수를 위한 새로운 별도 브랜치를 생성하고, <code>main</code> 브랜치는 새로운 버전(결국 <code>3.n+1.0</code>으로 릴리스될 버전)을 빌드하도록 수정됩니다.</p>
<p>새 브랜치 릴리스 단계는 릴리스 사이클의 여러 지점 중 한 곳에서 발생할 수 있지만, 현재 관행은 첫 번째 베타 릴리스로 예정된 릴리스의 기능 코드 마감 시점에 발생합니다.</p>
<p>아래 설명에서는 릴리스 유형에 따라 구체적인 단계가 레이블링되어 있으며, 현재는 <code>new branch</code>와 <code>final</code>이 있습니다.</p>
<h2>릴리스를 만드는 방법 (How To Make A Release)</h2>
<p>다음은 Python 릴리스를 만드는 단계입니다. 일부 단계는 자동화될 수 있는 부분이 거의 없기 때문에 (예: NEWS 항목 작성) 다른 단계보다 모호합니다. 일반적으로 "전문가(An Expert)"가 수행하는 단계의 경우 해당 전문가의 역할이 명시됩니다. 그렇지 않은 경우, 해당 단계는 릴리스를 수행하는 지정된 사람인 릴리스 관리자(RM)가 수행하는 것으로 가정합니다.</p>
<p>역할 및 현재 전문가:</p>
<ul>
<li><strong>RM (Release Manager):</strong>
<ul>
<li>Hugo van Kemenade</li>
<li>Thomas Wouters</li>
<li>Pablo Galindo Salgado</li>
<li>Łukasz Langa</li>
</ul>
</li>
<li><strong>WE (Windows Expert):</strong> Steve Dower</li>
<li><strong>ME (Mac Expert):</strong> Ned Deily</li>
<li><strong>DE (Docs Expert):</strong> Julien Palard</li>
</ul>
<p><strong>참고:</strong> RM은 릴리스 전날 전문가들에게 연락하는 것이 좋습니다. 전 세계가 다양한 시간대에 있기 때문에, RM은 전문가들이 바이너리 릴리스를 만들기에 충분한 시간에 릴리스 태그가 생성되도록 해야 합니다.</p>
<p>모든 전문가가 자신의 작업을 업데이트하기 전에 릴리스를 공개(웹사이트 업데이트 및 공지 전송)해서는 안 됩니다. Windows 또는 Mac 전문가가 부재중인 드문 경우, "(플랫폼) 바이너리는 곧 제공될 예정입니다"라는 메시지를 추가하고 진행할 수 있습니다.</p>
<p>아래 예시에서는 다음 규칙을 사용합니다. 릴리스 번호는 <code>3.X.YaN</code> 형식입니다. 예를 들어, Python 3.13.0 alpha 3의 경우 <code>3.13.0a3</code>이며, 여기서 "a"는 alpha, "b"는 beta, "rc"는 release candidate를 의미합니다.</p>
<p>릴리스 태그는 <code>v3.X.YaN</code>으로 명명됩니다. 마이너 릴리스 유지보수 브랜치의 이름은 <code>3.X</code>입니다.</p>
<p>가능한 한 릴리스는 <code>python/release-tools</code>라는 별도 저장소에 있는 <code>run_release.py</code> 스크립트에 의해 자동화되고 안내됩니다. 이는 다음 단계 중 많은 부분을 자동화하고, 일부 수동 단계를 수행하도록 안내합니다.</p>
<ol>
<li>
<p><strong>Discord에 로그인하여 Python Core Devs 서버에 가입합니다.</strong> Thomas 또는 Łukasz에게 초대장을 요청하십시오. 전 세계의 다른 사람들과 조율해야 할 가능성이 높으므로, 이 통신 채널이 만남을 위한 장소입니다.</p>
</li>
<li>
<p><strong>쇼스토퍼 버그(showstopper bugs)가 있는지 확인합니다.</strong> <code>https://github.com/python/cpython/issues</code>로 이동하여 이 릴리스를 차단할 수 있는 열린 버그를 찾습니다. 두 가지 관련 레이블을 확인합니다:</p>
<ul>
<li><code>release-blocker</code>: 릴리스를 즉시 중단시킵니다. 열린 <code>release-blocker</code> 버그가 있는 한 릴리스를 만들 수 없습니다.</li>
<li><code>deferred-blocker</code>: 이 릴리스를 차단하지는 않지만, 미래의 릴리스를 차단할 것입니다. 열린 <code>deferred-blocker</code> 버그가 있는 한 최종(final) 또는 후보(candidate) 릴리스를 만들 수 없습니다.
릴리스 블로커를 검토하고 해결하거나, <code>deferred</code>로 낮추거나, 릴리스를 중단하고 커뮤니티의 도움을 요청합니다. 최종 또는 후보 릴리스를 만들 경우, 열린 <code>deferred</code> 버그에 대해서도 동일하게 수행합니다.</li>
</ul>
</li>
<li>
<p><strong>안정적인 빌드봇(buildbots)을 확인합니다.</strong> <code>https://buildbot.python.org/all/#/release_status</code>로 이동합니다. 만들고 있는 릴리스의 빌드봇을 확인합니다. 오프라인 상태인 빌드봇은 무시합니다 (또는 커뮤니티에 알려 다시 시작하도록 합니다). 남은 빌드봇이 (대부분) 녹색이면 진행할 수 있습니다. 오프라인이 아닌 빨간색 빌드봇이 있다면, 수정될 때까지 릴리스를 보류하는 것이 좋습니다. 문제점을 검토하고, 알파, 베타 또는 최종 릴리스 중 어떤 것을 만들고 있는지 고려하여 판단을 내리십시오.</p>
</li>
<li>
<p><strong>릴리스 클론(release clone)을 만듭니다.</strong> GitHub의 CPython 저장소 포크(fork)에서 릴리스 브랜치를 생성합니다 (이제부터 "릴리스 클론"이라고 부릅니다). CPython 개발에 사용하는 동일한 GitHub 포크를 사용할 수 있습니다. Python 개발자 가이드에서 권장하는 표준 설정에 따라, 포크는 <code>origin</code>으로, 표준 CPython 저장소는 <code>upstream</code>으로 참조됩니다. 포크의 브랜치를 사용하여 릴리스 태그 지정 및 바이너리 생성을 위해 다른 전문가들과 공유하는 등 릴리스 엔지니어링 작업을 수행합니다.</p>
<p>최종 또는 릴리스 후보 2+ 릴리스의 경우, 마지막 릴리스 후보 이후 병합된 모든 변경 사항 중에서 다음 RC 또는 최종 릴리스를 위한 변경 사항의 하위 집합을 <code>cherry-pick</code>할 예정이라면, 가장 최근 릴리스 후보 태그(예: <code>v3.8.0rc1</code>)부터 시작하는 릴리스 엔지니어링 브랜치를 생성해야 합니다. 그런 다음 필요한 경우 표준 릴리스 브랜치에서 변경 사항을 릴리스 엔지니어링 브랜치로 <code>cherry-pick</code>하고 평소대로 진행합니다. 이전 RC 이후의 모든 변경 사항을 가져올 예정이라면 평소대로 진행할 수 있습니다.</p>
<p>릴리스 클론의 현재 브랜치가 릴리스하려는 브랜치인지 확인합니다 (<code>git status</code>).</p>
</li>
<li>
<p><strong><code>blurb release &#x3C;version></code>을 실행하여 버전 번호를 지정합니다.</strong> (예: <code>blurb release 3.4.7rc1</code>). 이는 모든 최근 뉴스 요약(news blurbs)을 이 릴리스의 버전 번호가 표시된 단일 파일로 병합합니다.</p>
</li>
<li>
<p><strong><code>Lib/pydoc-topics.py</code>를 재생성합니다.</strong> Doc 디렉토리에서 다음을 실행합니다.</p>
<pre><code class="language-bash">make pydoc-topics
cp build/pydoc-topics/topics.py ../Lib/pydoc_data/topics.py
</code></pre>
</li>
<li>
<p><strong><code>pydoc_topics.py</code> 변경 사항을 커밋합니다</strong> (및 문서에서 수정한 모든 내용).</p>
</li>
<li>
<p><strong><code>configure</code> 또는 기타 Autoconf 생성 파일이 이전 또는 이후 버전으로 마지막으로 커밋되어 잘못되거나 해로운 차이가 포함될 수 있는 경우, 현재 승인된 표준 버전을 사용하여 <code>autoconf</code>를 실행하는 것을 고려합니다.</strong> 현재 Autoconf 2.71이 사실상의 표준입니다. 차이가 있다면 커밋합니다.</p>
</li>
<li>
<p><strong><code>Doc/tools/extensions/pyspecific.py</code>의 <code>SOURCE_URI</code>가 Git 저장소의 올바른 브랜치(<code>main</code> 또는 <code>3.X</code>)를 가리키는지 확인합니다.</strong> 새 브랜치 릴리스의 경우, 파일의 브랜치를 <code>main</code>에서 새로 생성할 릴리스 브랜치(<code>3.X</code>)로 변경합니다.</p>
</li>
<li>
<p><strong>릴리스 스크립트를 통해 버전 번호를 업데이트합니다:</strong></p>
<pre><code class="language-bash">.../release-tools/release.py --bump 3.X.YaN
</code></pre>
<p><strong>알림:</strong> <code>X</code>, <code>Y</code>, <code>N</code>은 정수여야 합니다. <code>a</code>는 <code>a</code>, <code>b</code>, 또는 <code>rc</code> 중 하나여야 합니다 (예: <code>3.4.3rc1</code>). 최종 릴리스의 경우 <code>aN</code>을 생략합니다 (예: <code>3.4.3</code>). 새 버전의 첫 번째 릴리스의 경우 <code>Y</code>는 <code>0</code>이어야 합니다 (예: <code>3.6.0</code>).</p>
<p>이것은 다양한 릴리스 번호 업데이트를 자동화하지만, 몇몇 파일은 수동으로 수정해야 합니다. <code>$EDITOR</code> 환경 변수가 올바르게 설정되어 있으면 <code>release.py</code>가 편집해야 할 파일이 있는 편집기 창을 띄울 것입니다.</p>
</li>
<li>
<p><strong><code>blurb</code>가 생성한 <code>Misc/NEWS</code> 파일을 검토하고 필요에 따라 편집합니다.</strong> 모든 변경 사항이 커밋되었는지 확인합니다 (<code>release.py --bump</code>는 변경 사항을 커밋하지 않습니다). 최종 주요 릴리스의 경우, <code>Doc/whatsnew/3.X.rst</code>의 첫 번째 단락을 실제 릴리스 날짜를 포함하도록 편집합니다. 예를 들어, "Python 2.5 was released on August 1, 2003."과 같이 말입니다. 알파 또는 베타 릴리스의 경우 이를 편집할 필요는 없습니다. 이 디렉토리에서 <code>git status</code>를 수행합니다. 어떠한 파일도 보이지 않아야 합니다. 즉, 작업 디렉토리에 커밋되지 않은 변경 사항이 없어야 합니다.</p>
</li>
<li>
<p><strong><code>3.X.YaN</code>에 대한 릴리스 태그를 지정합니다:</strong></p>
<pre><code class="language-bash">.../release-tools/release.py --tag 3.X.YaN
</code></pre>
<p>이것은 <code>-s</code> 옵션과 함께 <code>git tag</code> 명령을 실행하여 저장소의 릴리스 태그가 GPG 키로 서명되도록 합니다. 릴리스 타볼(tarballs) 등에 서명하는 데 사용하는 개인 키를 선택하라는 메시지가 나타나면 선택합니다.</p>
<p><code>begin security-only mode</code> 및 <code>end-of-life</code> 릴리스의 경우, 두 파일을 검토하고 모든 활성 브랜치에서 버전을 accordingly 업데이트합니다.</p>
</li>
<li>
<p><strong>커밋을 GitHub 포크의 원격 릴리스 브랜치로 푸시합니다:</strong></p>
<pre><code class="language-bash"># 먼저 dry run을 수행합니다.
git push --dry-run --tags origin
# GitHub 포크로 푸시하는지 확인합니다.
# *메인 python/cpython 저장소가 아닙니다!*
git push --tags origin
</code></pre>
</li>
<li>
<p><strong><code>python/release-tools</code>에서 <code>build-release</code> 워크플로로 이동하여 "Run workflow"를 선택하고 방금 생성한 태그의 세부 정보를 입력합니다.</strong> 이는 다음 단계를 수행합니다.</p>
<ul>
<li>소스 gzip 및 xz 타볼을 생성합니다.</li>
<li>문서 tar 및 zip 파일을 생성합니다.</li>
<li>완전히 깨끗하고 순수한 빌드가 회귀 테스트를 통과하는지 확인하기 위해 소스 타볼을 검사합니다.</li>
<li>Android 바이너리를 빌드하고 테스트합니다 (Python 3.14 이상인 경우).
결과 아티팩트는 GitHub 워크플로의 요약 페이지에 첨부됩니다. 소스 타볼을 사용할 수 있게 되면, 다운로드하여 압축을 풀어 내용이 합리적인지, 불필요한 <code>.pyc</code> 파일 등이 없는지 확인합니다. 테스트가 통과하면 타볼이 괜찮다고 확신할 수 있습니다. 일부 테스트가 실패하거나, 새로 압축 해제된 디렉토리에 이상한 점이 있다면 즉시 중단하고 문제점을 파악해야 합니다.</li>
</ul>
</li>
<li>
<p><strong>전문가들에게 바이너리 빌드를 시작할 수 있다고 알립니다.</strong>
<strong>경고: 중단!</strong> 이 시점에서 릴리스를 생성하려면 다른 전문가들로부터 "녹색 신호(green light)"를 받아야 합니다. 기다리는 동안 할 수 있는 일들이 있으므로 다음 "중단" 지점까지 계속 읽으십시오.</p>
<p>WE는 <code>.azure-pipelines/windows-release/</code>의 Azure Pipelines 빌드 스크립트(현재 <code>https://dev.azure.com/Python/cpython/_build?definitionId=21</code>에 설정됨)를 사용하여 Windows 파일을 생성하고 게시합니다.</p>
<p>빌드 프로세스는 여러 단계로 실행되며, 각 단계의 출력은 다운로드 가능한 아티팩트로 제공됩니다. 단계는 다음과 같습니다.</p>
<ul>
<li>프로필 가이드 최적화(profile-guided optimization) 실행을 포함하여 모든 바이너리 변형(32비트, 64비트, debug/release)을 컴파일합니다.</li>
<li>Python 문서가 포함된 HTML Help 파일을 컴파일합니다.</li>
<li>PSF의 인증서로 모든 바이너리에 코드 서명을 합니다.</li>
<li><code>python.org</code>, <code>nuget.org</code>, 임베더블 배포판(embeddable distro) 및 Windows 스토어용 패키지를 생성합니다.</li>
<li>설치 프로그램에 대한 기본 검증을 수행합니다.</li>
<li><code>python.org</code> 및 <code>nuget.org</code>에 패키지를 업로드하고, 다운로드 캐시를 제거하며, 테스트 다운로드를 실행합니다.
업로드가 완료되면 WE는 빌드 로그에서 생성된 해시(hash)를 복사하여 RM에게 이메일로 보냅니다. Windows 스토어 패키지는 WE가 수동으로 <code>https://partner.microsoft.com/dashboard/home</code>에 업로드합니다.</li>
</ul>
<p>ME는 Mac 설치 프로그램 패키지를 빌드하고 GPG 서명 파일과 함께 <code>downloads.nyc1.psf.io</code>에 업로드합니다.</p>
</li>
<li>
<p><strong><code>build-release</code> 워크플로에 의해 빌드된 모든 파일을 서명, SBOM(Software Bill of Materials) 등과 함께 <code>downloads.nyc1.psf.io</code>의 홈 디렉토리로 <code>scp</code> 또는 <code>rsync</code>를 사용하여 전송합니다.</strong> 파일을 업로드하는 동안 나머지 작업을 계속할 수 있습니다. 또한 Discord 및/또는 <code>discuss.python.org</code>의 사람들에게 파일 업로드가 완료되면 다운로드하여 각자의 플랫폼에서 테스트해 달라고 요청할 수 있습니다.</p>
</li>
<li>
<p><strong>이제 <code>downloads.nyc1.psf.io</code>로 이동하여 모든 파일을 제자리에 옮겨야 합니다.</strong> 우리의 정책은 모든 Python 버전이 자체 디렉토리를 갖지만, 각 디렉토리에는 해당 버전의 모든 릴리스가 포함되는 것입니다.
<code>downloads.nyc1.psf.io</code>에서 <code>/srv/www.python.org/ftp/python/3.X.Y</code>로 이동하고 필요한 경우 생성합니다. <code>downloads</code> 그룹이 소유하고 그룹 쓰기 가능하도록 확인합니다. 위에서 홈 디렉토리에 업로드한 파일을 릴리스 디렉토리로 이동합니다. Windows/Mac 바이너리는 일반적으로 전문가들이 직접 그곳에 배치합니다.</p>
<p>파일이 전 세계적으로 읽기 가능하도록 확인합니다. 또한 그룹 쓰기 가능하고 <code>downloads</code> 그룹이 소유해야 합니다. <code>gpg --verify</code>를 사용하여 파일이 손상 없이 업로드되었는지 확인합니다.</p>
<p>이것이 최종(final) 또는 RC(Release Candidate) 릴리스인 경우: <code>doc</code> zip 파일과 타볼을 <code>/srv/www.python.org/ftp/python/doc/3.X.Y[rcA]</code>로 이동하고, 필요한 경우 디렉토리를 생성한 다음 <code>.../doc</code>의 "current" 심볼릭 링크(symlink)를 해당 디렉토리를 가리키도록 조정합니다. 그러나 이전 버전의 유지보수 릴리스를 릴리스하는 경우 "current" 링크를 변경하지 마십시오.</p>
<p>이것이 최종 또는 RC 릴리스인 경우 (유지보수 릴리스라도), HTML 문서도 <code>docs.nyc1.psf.io</code>의 <code>/srv/docs.python.org/release/3.X.Y[rcA]</code>에 압축 해제합니다. 파일이 <code>docs</code> 그룹에 속하고 그룹 쓰기 가능하도록 확인합니다. DE가 문서가 제대로 빌드되고 작동하는지 확인하도록 합니다. 문서와 다운로드 모두 캐싱 CDN 뒤에 있습니다. 웹사이트를 통해 다운로드한 후 아카이브를 변경하면 CDN에서 오래된 데이터를 다음과 같이 제거해야 합니다.</p>
<pre><code class="language-bash">curl -X PURGE https://www.python.org/ftp/python/3.12.0/Python-3.12.0.tar.xz
</code></pre>
<p>사람들이 릴리스 파일을 탐색하는 데 사용하므로 항상 디렉토리 목록의 캐시를 제거해야 합니다.</p>
<pre><code class="language-bash">curl -X PURGE https://www.python.org/ftp/python/3.12.0/
</code></pre>
</li>
<li>
<p><strong>극도로 신중하다면, 릴리스에 대한 완전히 깨끗한 테스트를 수행합니다.</strong> 여기에는 <code>www.python.org</code>에서 타볼을 다운로드하는 것이 포함됩니다. md5 체크섬이 일치하는지 확인합니다. 그런 다음 타볼의 압축을 해제하고 깨끗하게 <code>make test</code>를 수행합니다.</p>
<pre><code class="language-bash">make distclean
./configure
make test
</code></pre>
<p>회귀 테스트 스위트가 통과하는지 확인하기 위함입니다. 그렇지 않으면 어딘가에서 잘못된 것입니다!</p>
<p><strong>경고: 중단 및 확인!</strong></p>
<ul>
<li>WE로부터 녹색 신호를 받았습니까?</li>
<li>ME로부터 녹색 신호를 받았습니까?</li>
<li>DE로부터 녹색 신호를 받았습니까?
녹색 신호를 받았다면, 릴리스 엔지니어링 브랜치를 메인 저장소로 다시 병합할 시간입니다.</li>
</ul>
</li>
<li>
<p><strong>변경 사항을 GitHub에 푸시하려면, 일시적으로 관리자에 대한 브랜치 보호를 비활성화해야 합니다.</strong> <code>Settings | Branches</code> 페이지로 이동합니다. <code>https://github.com/python/cpython/settings/branches</code>
릴리스하는 브랜치의 설정을 "편집"합니다. 해당 브랜치의 설정 페이지가 로드됩니다. "Include administrators" 상자의 체크를 해제하고 하단의 "Save changes" 버튼을 누릅니다.</p>
</li>
<li>
<p><strong>릴리스 클론을 메인 개발 저장소로 병합합니다:</strong></p>
<pre><code class="language-bash"># upstream 저장소 브랜치의 깨끗한 복사본
git clone git@github.com:python/cpython.git merge
cd merge
# 올바른 브랜치를 체크아웃합니다:
# 1. 새로운 브랜치 릴리스를 포함하는 기능 사전 릴리스,
#    즉 알파 및 첫 번째 베타의 경우
git checkout main
# 2. 그 외 모든 다른 릴리스의 경우, 적절한 릴리스 브랜치를 체크아웃합니다.
git checkout 3.X
# 클론 저장소에서 새로 생성되고 서명된 태그를 가져옵니다.
git fetch --tags git@github.com:your-github-id/cpython.git v3.X.YaN
# 임시 릴리스 엔지니어링 브랜치를 다시 병합합니다.
git merge --no-squash v3.X.YaN
git commit -m 'Merge release engineering branch'
</code></pre>
<p>이것이 <code>new branch</code> 릴리스, 즉 첫 번째 베타인 경우, 이제 새로운 릴리스 브랜치를 생성합니다.</p>
<pre><code class="language-bash">git checkout -b 3.X
</code></pre>
<p>새로운 릴리스 브랜치를 설정하는 데 필요한 단계를 수행합니다. 여기에는 다음이 포함됩니다.</p>
<ul>
<li><code>README.rst</code>에서 <code>main</code>에 대한 모든 참조를 새 브랜치로 변경합니다 (특히 GitHub 저장소 URL).</li>
</ul>
</li>
<li>
<p><strong>모든 릴리스에 대해 릴리스 스크립트를 사용하여 안내된 릴리스 후 단계를 수행합니다:</strong></p>
<pre><code class="language-bash">.../release-tools/release.py --done 3.X.YaN
</code></pre>
<p>최종(final) 또는 릴리스 후보 2+ 릴리스의 경우, 병합 후 일부 정리가 필요할 수 있습니다. 최상위 <code>README.rst</code> 및 <code>include/patchlevel.h</code> 파일을 확인하여 현재 진행 중인 개발에 대한 원하는 릴리스 후 값을 반영하는지 확인합니다. <code>patchlevel</code>은 릴리스 태그에 <code>+</code>가 붙어야 합니다. 또한, 이 릴리스를 위해 표준 릴리스 브랜치에서 릴리스 엔지니어링 브랜치로 변경 사항을 <code>cherry-pick</code>한 경우, 이제 <code>Misc/NEWS.d/next</code> 디렉토리에서 <code>cherry-pick</code>된 각 <code>blurb</code> 항목을 수동으로 제거해야 합니다. 이는 해당 <code>blurb</code> 항목이 새 릴리스의 병합된 <code>x.y.z.rst</code> 파일에 이미 캡처되었기 때문입니다. 그렇지 않으면 <code>blurb</code> 항목이 <code>changelog.html</code> 파일에 두 번 나타납니다 (한 번은 Python <code>next</code> 아래, 다른 한 번은 <code>x.y.z</code> 아래).</p>
<p>이러한 변경 사항을 검토하고 커밋합니다.</p>
<pre><code class="language-bash">git commit -m 'Post release updates'
</code></pre>
<p>이것이 <code>new branch</code> 릴리스(예: 첫 번째 베타)인 경우, 다음 기능 릴리스 개발을 시작하기 위해 <code>main</code> 브랜치를 업데이트합니다. 완료되면 <code>main</code> 브랜치는 이제 Python <code>X.Y+1</code>을 빌드할 것입니다.</p>
<p>먼저, <code>main</code>을 다음 릴리스, 즉 <code>X.Y+1.a0</code>로 설정합니다.</p>
<pre><code class="language-bash">git checkout main
.../release-tools/release.py --bump 3.9.0a0
</code></pre>
<ul>

<pre><code class="language-bash"># 먼저 dry run을 수행합니다.
# **new branch** 릴리스 이전의 기능 사전 릴리스,
# 즉 기능 알파 릴리스의 경우:
git push --dry-run --tags git@github.com:python/cpython.git main
# 괜찮아 보인다면 실행합니다. 되돌릴 수 없습니다!
git push --tags git@github.com:python/cpython.git main

# **new branch** 릴리스, 즉 첫 번째 베타의 경우:
git push --dry-run --tags git@github.com:python/cpython.git 3.X
git push --dry-run --tags git@github.com:python/cpython.git main
# 괜찮아 보인다면 실행합니다. 되돌릴 수 없습니다!
git push --tags git@github.com:python/cpython.git 3.X
git push --tags git@github.com:python/cpython.git main

# 그 외 모든 릴리스의 경우:
git push --dry-run --tags git@github.com:python/cpython.git 3.X
# 괜찮아 보인다면 실행합니다. 되돌릴 수 없습니다!
git push --tags git@github.com:python/cpython.git 3.X
</code></pre>
<p>이것이 <code>new branch</code> 릴리스인 경우, 새로 생성된 브랜치(<code>3.X</code>)에 대한 브랜치 보호 규칙을 추가합니다. 이전 릴리스 브랜치(<code>3.X-1</code>)의 값을 템플릿으로 사용합니다. <code>https://github.com/python/cpython/settings/branches</code></p>
<p>또한 <code>3.x</code> 및 <code>needs backport to 3.X</code> 레이블을 GitHub 저장소에 추가합니다. <code>https://github.com/python/cpython/labels</code></p>

<p>릴리스에 새로운 유형의 파일이 추가되면, 누군가는 <code>add_to_pydotorg.py</code>를 업데이트하여 이 새로운 파일을 인식하도록 해야 합니다. (파일 유형이 제거될 때도 <code>add_to_pydotorg.py</code>를 업데이트하는 것이 가장 좋습니다.)</p>
<p>이 스크립트는 이 시점까지 Sigstore로 서명되지 않은 나머지 파일도 서명합니다. 다시 말하지만, 이 경우 <code>@python.org</code> 주소를 이 과정에 사용하십시오. 추가 정보: <code>https://www.python.org/download/sigstore/</code></p>

<h2>수명 종료 (End-of-life)로 이동 (Moving to End-of-life)</h2>
<p>현재 정책에 따라, 릴리스 브랜치는 일반적으로 초기 릴리스 후 5년이 지나면 수명 종료 상태에 도달합니다. 이 정책은 Python 개발자 가이드에서 더 자세히 논의됩니다. 수명 종료에 도달하면 릴리스 관리자로서 직접 수행하거나 다른 사람이 수행하도록 보장해야 하는 여러 작업이 있습니다. 그 작업 중 일부는 다음과 같습니다.</p>
<ul>
<li>선택적으로 최종 릴리스를 만들어 아직 릴리스되지 않은 변경 사항을 게시합니다.</li>
<li>현재 HEAD의 태그를 생성한 다음 CPython 저장소에서 브랜치를 삭제하여 릴리스 브랜치의 상태를 고정합니다. 현재 HEAD는 브랜치의 최종 보안 릴리스와 같거나 그 이후여야 합니다.
<pre><code class="language-bash">git fetch upstream
git tag --sign -m 'Final head of the former 3.3 branch' 3.3 upstream/3.3
git push upstream refs/tags/3.3
</code></pre>
모든 것이 잘 보인다면 브랜치를 삭제합니다. 이를 위해서는 저장소 관리자 권한이 있는 사람의 도움이 필요할 수 있습니다.
<pre><code class="language-bash">git push upstream --delete 3.3
# 또는 GitHub 설정 페이지에서 수행
</code></pre>
</li>
<li>다운로드 페이지의 "Active Python Releases" 목록에서 릴리스를 제거합니다. 이렇게 하려면 <code>python.org</code>의 관리 페이지에 로그인하여 <code>Boxes</code>로 이동하고 <code>downloads-active-releases</code> 항목을 편집합니다. 해당 릴리스의 관련 HTML 단락을 제거합니다. (변경 사항을 올바르게 적용했는지 확인하려면 <code>curl -X PURGE</code> 트릭을 사용하여 캐시를 제거해야 할 것입니다.)</li>
<li>은퇴한 브랜치에 대한 <code>python.org</code>의 각 릴리스 페이지에 은퇴 알림을 추가합니다. 예를 들어:
<ul>
<li><code>https://www.python.org/downloads/release/python-337/</code></li>
<li><code>https://www.python.org/downloads/release/python-336/</code></li>
</ul>
</li>
<li>개발자 가이드에서 브랜치 상태를 <code>end-of-life</code>로 설정하고, devguide의 다른 곳에 있는 브랜치 참조를 업데이트하거나 제거합니다.</li>
<li>이슈 트래커에서 릴리스를 은퇴시킵니다. 작업에는 다음이 포함됩니다.

<li>버전 목록에서 버전 레이블을 제거합니다.</li>

<h2>Windows 관련 참고 사항 (Windows Notes)</h2>
<p>Windows에는 MSI 설치 프로그램이 있으며, 다양한 Windows 버전에는 "특별한 제한 사항"이 있고, Windows 설치 프로그램은 사전 컴파일된 "외부" 바이너리(Tcl/Tk, expat 등)도 포함합니다.</p>
<p>설치 프로그램은 Azure Pipeline의 일부로 테스트됩니다. 과거에는 이러한 단계가 수동으로 수행되었습니다. 이는 기록을 위해 보관하고 있습니다.</p>
<p>설치 프로그램을 업로드하는 동시에, WE는 설치 프로그램을 사용하여 Python을 두 번 설치합니다. 한 번은 설치 프로그램이 제안하는 기본 디렉토리에, 다른 한 번은 이름에 공백이 포함된 디렉토리에 설치합니다. 각 설치에 대해 WE는 DOS 상자에서 전체 회귀 스위트를 실행하며, <code>-0</code> 옵션 사용 여부와 관계없이 실행합니다. 유지보수 릴리스의 경우, WE는 업그레이드 설치가 성공하는지 여부도 테스트합니다.</p>
<p>WE는 또한 시작 -> 메뉴 -> Python 그룹 아래에 생성된 모든 단축키를 시도합니다. 이 방식으로 IDLE을 시도할 때, 도움말 -> Python Documentation이 작동하는지 확인해야 합니다. 이 방식으로 pydoc을 시도할 때 ("Module Docs" 시작 메뉴 항목), "Start Browser" 버튼이 작동하는지 확인하고, 임의의 모듈(예: "random" )을 검색할 수 있는지, 그리고 "go to selected" 버튼이 작동하는지 확인해야 합니다.</p>
<p>여기서 얼마나 많은 것이 잘못될 수 있는지 놀랍습니다. 그리고 막판 체크인으로 이러한 것들 중 하나가 얼마나 자주 깨지는지 더욱 놀랍습니다. 당신이 "Windows 전문가"라면, 당신이 Windows에서 일상적으로 테스트하는 유일한 사람일 가능성이 높다는 것과 Windows가 단순히 엉망진창이라는 것을 명심하십시오.</p>
<p>각 대상 아키텍처에 대해 테스트를 반복합니다. 관리자 계정과 일반 사용자(Power User 아님) 계정 모두를 시도합니다.</p>
