<h1 id="pep-599-manylinux2014-플랫폼-태그"><a href="#pep-599-manylinux2014-플랫폼-태그">PEP 599 – manylinux2014 플랫폼 태그</a></h1>
<h2 id="개요-abstract"><a href="#개요-abstract">개요 (Abstract)</a></h2>
<p>이 PEP(Python Enhancement Proposal)는 기존 PEP 513에서 도입된 <code>manylinux2010</code> 태그를 계승하는 <code>manylinux2014</code> 플랫폼 태그의 생성을 제안합니다. 또한, PyPI(Python Package Index)와 <code>pip</code>가 호환 가능한 플랫폼에서 <code>manylinux2014</code> 배포판의 업로드, 다운로드, 설치를 지원하도록 업데이트할 것을 제안합니다.</p>
<h2 id="도입-배경-rationale"><a href="#도입-배경-rationale">도입 배경 (Rationale)</a></h2>
<p><code>manylinux</code> 표준은 Linux 환경에서 컴파일된 Python 휠(wheel) 패키지의 호환성을 보장하기 위해 도입되었습니다. <code>manylinux2010</code>의 기반이 되었던 CentOS 6는 2020년 11월 30일에 지원이 종료되었으며, 이 시점 이후에는 보안 패치를 포함한 추가 업데이트가 제공되지 않습니다. 이로 인해 <code>manylinux2010</code> 이미지로 빌드된 모든 휠은 구식 버전으로 남게 됩니다.</p>
<p>이러한 문제 해결을 위해, 기존 <code>manylinux</code> 표준을 이어가며 CentOS 7을 기반으로 하는 새로운 PEP 425 스타일의 플랫폼 태그인 <code>manylinux2014</code>를 제안합니다. <code>manylinux2014</code>는 CentOS 7의 라이브러리와 심볼 버전을 따르며, CentOS 7은 2024년 6월 30일에 지원이 종료될 예정입니다.</p>
<p><code>manylinuxYYYY</code> 패턴은 다음과 같은 장점 때문에 현재 방식을 유지하는 것이 좋습니다:</p>
<ul>
<li>명확하게 지정된 호환 라이브러리를 포함하는 잘 정의된 Docker 이미지를 사용할 수 있습니다.</li>
<li>여러 릴리스에 걸쳐 호환성 문제를 조사할 필요가 없습니다.</li>
<li>아키텍처별로 단일 빌드 이미지와 <code>auditwheel</code> 프로필을 제공합니다.</li>
</ul>
<p>단점으로는 다음이 있습니다:</p>
<ul>
<li>새로운 표준이 나올 때마다 새로운 PEP를 작성해야 합니다.</li>
<li>인스톨러(예: <code>pip</code>)에 새로운 플랫폼 태그를 추가해야 합니다.</li>
<li>인스톨러가 특정 릴리스 이전의 플랫폼 태그를 설치할 수 없습니다.</li>
</ul>
<p>어떤 제안에서든 Docker 이미지 및 해당 <code>auditwheel</code> 프로필을 정의, 준비 및 릴리스하는 데 필요한 시간과 노력과 같은 공통적인 과제들이 존재합니다. <code>manylinux2010</code>의 경우 PEP 승인부터 호환 가능한 빌드 환경이 게시되기까지 약 1년이 걸렸습니다. 그러나 이 PEP를 통해 이 과정이 잘 정의되어 있고 쉽게 반복 가능하여, 새로운 플랫폼 태그의 출시 일정을 단축할 수 있을 것으로 예상됩니다.</p>
<h2 id="manylinux2014-정책-the-manylinux2014-policy"><a href="#manylinux2014-정책-the-manylinux2014-policy">manylinux2014 정책 (The manylinux2014 policy)</a></h2>
<p>Linux 휠이 <code>manylinux2014</code> 태그를 부여받기 위한 기준은 다음과 같습니다:</p>
<ol>
<li>
<p><strong>지원 아키텍처:</strong> 휠은 CentOS 7 또는 CentOS 7 호환 기반 이미지(예: ubi7)에서 지원하는 다음 아키텍처 중 하나에 대해 컴파일된 바이너리 실행 파일 및 공유 객체만 포함해야 합니다:</p>
<ul>
<li><code>x86_64</code></li>
<li><code>i686</code></li>
<li><code>aarch64</code> (ARMv8)</li>
<li><code>armv7l</code> (ARMv7)</li>
<li><code>ppc64</code> (PowerPC)</li>
<li><code>ppc64le</code> (PowerPC Little-Endian)</li>
<li><code>s390x</code> (IBM Z)</li>
</ul>
<p>이 목록은 ARMv7, ARMv8, PowerPC 및 IBM Z 아키텍처에 대한 지원을 추가합니다.</p>
</li>
<li>
<p><strong>허용되는 외부 라이브러리:</strong> 휠의 바이너리 실행 파일 또는 공유 객체는 다음 목록에 있는 외부 라이브러리에 대해서만 링크되어야 합니다:</p>
<ul>
<li><code>libgcc_s.so.1</code></li>
<li><code>libstdc++.so.6</code></li>
<li><code>libm.so.6</code></li>
<li><code>libdl.so.2</code></li>
<li><code>librt.so.1</code></li>
<li><code>libc.so.6</code></li>
<li><code>libnsl.so.1</code></li>
<li><code>libutil.so.1</code></li>
<li><code>libpthread.so.0</code></li>
<li><code>libresolv.so.2</code></li>
<li><code>libX11.so.6</code></li>
<li><code>libXext.so.6</code></li>
<li><code>libXrender.so.1</code></li>
<li><code>libICE.so.6</code></li>
<li><code>libSM.so.6</code></li>
<li><code>libGL.so.1</code></li>
<li><code>libgobject-2.0.so.0</code></li>
<li><code>libgthread-2.0.so.0</code></li>
<li><code>libglib-2.0.so.0</code></li>
</ul>
<p>이 목록은 <code>manylinux2010</code>에서 허용되었던 외부 라이브러리와 동일하지만, Fedora 30에서 더 이상 사용되지 않는 <code>libcrypt.so.1</code>이 제거되었습니다. <code>libpythonX.Y</code>는 PEP 513에 명시된 동일한 이유로 포함될 수 없습니다.</p>
<p>Debian 기반 시스템에서는 이 라이브러리들이 <code>libc6</code>, <code>libgcc1</code>, <code>libgl1</code>, <code>libglib2.0-0</code>, <code>libice6</code>, <code>libsm6</code>, <code>libstdc++6</code>, <code>libx11-6</code>, <code>libxext6</code>, <code>libxrender1</code> 패키지에서 제공됩니다. RPM 기반 시스템에서는 <code>glib2</code>, <code>glibc</code>, <code>libICE</code>, <code>libX11</code>, <code>libXext</code>, <code>libXrender</code>, <code>libgcc</code>, <code>libstdc++</code>, <code>mesa</code> 패키지에서 제공됩니다.</p>
</li>
<li>
<p><strong>버전별 심볼 제한:</strong> 휠에 허용된 라이브러리에 링크된 바이너리 실행 파일 또는 공유 객체가 버전별 심볼을 내보내는 경우, 다음 최대 버전까지만 의존할 수 있습니다:</p>
<ul>
<li><code>GLIBC_2.17</code></li>
<li><code>CXXABI_1.3.7</code>, <code>CXXABI_TM_1</code>도 허용</li>
<li><code>GLIBCXX_3.4.19</code></li>
<li><code>GCC_4.8.0</code></li>
</ul>
<p>예를 들어, <code>manylinux2014</code> 휠은 <code>GLIBC_2.17</code>보다 이전 버전인 <code>GLIBC_2.12</code> 심볼을 요구하는 바이너리 아티팩트를 포함할 수 있습니다.</p>
</li>
<li>
<p><strong>CPython ABI 태그 및 PyFPE_jbuf:</strong></p>
<ul>
<li>CPython 2 또는 CPython 3.0부터 3.2까지의 버전을 위해 빌드된 휠은 Unicode ABI를 나타내는 CPython ABI 태그를 포함해야 합니다. 예를 들어, Python 2용 <code>manylinux2014</code> 휠은 UCS-4 ABI를 나타내는 <code>cpy27mu</code> 태그 또는 UCS-2 ABI를 나타내는 <code>cpy27m</code> 태그를 포함해야 합니다. (PEP 3149)</li>
<li>휠은 <code>PyFPE_jbuf</code> 심볼을 요구해서는 안 됩니다. 이는 <code>--with-fpectl</code> configure 플래그 없이 Python을 컴파일하여 달성할 수 있습니다.</li>
</ul>
</li>
</ol>
<h2 id="호환-휠-컴파일-compilation-of-compliant-wheels"><a href="#호환-휠-컴파일-compilation-of-compliant-wheels">호환 휠 컴파일 (Compilation of Compliant Wheels)</a></h2>
<p><code>manylinux1</code>과 마찬가지로, <code>auditwheel</code> 도구는 <code>manylinux2014</code> Docker 컨테이너에서 <code>pip wheel</code> 또는 <code>bdist_wheel</code>로 빌드된 Linux 휠에 <code>manylinux2014</code> 플랫폼 태그를 추가합니다.</p>
<h3 id="docker-이미지-docker-images"><a href="#docker-이미지-docker-images">Docker 이미지 (Docker Images)</a></h3>
<p><code>manylinux2014</code> 휠로 안정적으로 변환될 수 있는 바이너리 Linux 휠을 빌드하기 위해 CentOS 7 x86_64를 기반으로 하는 <code>manylinux2014</code> Docker 이미지가 제공되어야 합니다. 이 이미지에는 전체 컴파일러 스위트(gcc, g++, gfortran 4.8.5)와 최신 버전의 Python 및 pip가 설치됩니다.</p>
<h3 id="auditwheel"><a href="#auditwheel">Auditwheel</a></h3>
<p><code>auditwheel</code> 도구도 <code>manylinux2014</code> 휠을 생성하도록 업데이트될 것입니다. <code>auditwheel</code>의 동작과 목적은 PEP 513에서 정의된 바와 동일하게 유지됩니다.</p>
<h2 id="인스톨러의-플랫폼-감지-platform-detection-for-installers"><a href="#인스톨러의-플랫폼-감지-platform-detection-for-installers">인스톨러의 플랫폼 감지 (Platform Detection for Installers)</a></h2>
<p>플랫폼은 PEP 513에 설명된 <code>_manylinux</code> 모듈에 <code>manylinux2014_compatible</code>이라는 부울(boolean) 속성을 정의할 수 있습니다. 이 속성이 <code>False</code>인 경우, 해당 플랫폼은 <code>manylinux2014</code>와 호환되지 않는 것으로 간주됩니다.</p>
<p><code>_manylinux</code> 모듈을 찾을 수 없거나 <code>manylinux2014_compatible</code> 속성이 없는 경우, 도구는 <code>glibc</code> 버전 확인으로 대체될 수 있습니다. 플랫폼이 <code>glibc 2.17</code> 이상을 사용하는 경우, <code>_manylinux</code> 모듈이 다른 결과를 나타내지 않는 한 호환되는 것으로 간주됩니다.</p>
<p>제안된 알고리즘은 다음과 같습니다:</p>
<pre><code class="language-python">def is_manylinux2014_compatible():
    # Only Linux, and only supported architectures
    from distutils.util import get_platform
    if get_platform() not in [
        "linux-x86_64", "linux-i686", "linux-aarch64", "linux-armv7l",
        "linux-ppc64", "linux-ppc64le", "linux-s390x",
    ]:
        return False

    # Check for presence of _manylinux module
    try:
        import _manylinux
        return bool(_manylinux.manylinux2014_compatible)
    except (ImportError, AttributeError):
        # Fall through to heuristic check below
        pass

    # Check glibc version. CentOS 7 uses glibc 2.17.
    # PEP 513 contains an implementation of this function.
    return have_compatible_glibc(2, 17)
</code></pre>
<h2 id="manylinux2010-휠과의-하위-호환성-backwards-compatibility-with-manylinux2010-wheels"><a href="#manylinux2010-휠과의-하위-호환성-backwards-compatibility-with-manylinux2010-wheels">manylinux2010 휠과의 하위 호환성 (Backwards compatibility with manylinux2010 wheels)</a></h2>
<p>PEP 513에서 설명했듯이, <code>manylinux1</code>에서 허용된 라이브러리에 대해 지정된 심볼 버전은 상한선을 구성합니다. 이 PEP에서 <code>manylinux2014</code>에 정의된 심볼 버전도 마찬가지입니다. 결과적으로 <code>manylinux1</code> 및 <code>manylinux2010</code> 휠은 <code>manylinux2014</code> 휠로 간주됩니다. 따라서 <code>manylinux2014</code> 플랫폼 태그를 인식하는 <code>pip</code>는 <code>manylinux2014</code> 휠을 사용할 수 없을 때 <code>manylinux2010</code> 휠을 <code>manylinux2014</code> 플랫폼용으로 설치할 것입니다.</p>
<h2 id="pypi-지원-pypi-support"><a href="#pypi-지원-pypi-support">PyPI 지원 (PyPI Support)</a></h2>
<p>PyPI는 <code>manylinux2010</code>을 허용하는 것과 동일한 방식으로 <code>manylinux2014</code> 플랫폼 태그를 포함하는 휠의 업로드를 허용해야 합니다. 기술적으로 가능하다면, PyPI는 <code>manylinux2014</code> 휠의 호환성을 검증하려고 시도해야 하지만, 이 기능은 이 PEP의 채택을 위한 필수 요구 사항은 아닙니다.</p>
<p>패키지 작성자는 비호환 <code>manylinux2014</code> 휠을 PyPI에 업로드해서는 안 되며, PyPI가 비호환 휠의 업로드를 차단하기 시작할 수 있음을 인지해야 합니다.</p>
<h2 id="채택-acceptance"><a href="#채택-acceptance">채택 (Acceptance)</a></h2>
<p>PEP 599는 2019년 7월 31일 Paul Moore에 의해 승인되었습니다.</p>
