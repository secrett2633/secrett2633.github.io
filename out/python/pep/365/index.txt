<p><strong>상태:</strong> Rejected | <strong>유형:</strong> Standards Track | <strong>작성일:</strong> 30-Apr-2007</p>

<h1>PEP 365 – <code>pkg_resources</code> 모듈 추가</h1>
<ul>
<li><strong>작성자:</strong> Phillip J. Eby</li>
<li><strong>상태:</strong> Rejected (거절됨)</li>
<li><strong>유형:</strong> Standards Track</li>
<li><strong>주제:</strong> Packaging (패키징)</li>
<li><strong>생성일:</strong> 2007년 4월 30일</li>
</ul>
<h2>개요 (Abstract)</h2>
<p>이 PEP는 개선된 버전의 <code>pkg_resources</code> 모듈을 Python 표준 라이브러리에 추가할 것을 제안합니다.</p>
<p><code>pkg_resources</code>는 Python 패키지/버전 의존성을 찾고 관리하며, <code>.egg</code> 압축 파일 내부에 포함된 파일 및 리소스에 접근하는 데 사용되는 모듈입니다. 현재 <code>pkg_resources</code>는 전체 <code>setuptools</code> 배포판을 설치해야만 사용할 수 있지만, <code>setuptools</code>의 다른 부분에 의존하지 않습니다. 사실상 <code>Python Eggs</code>의 전체 런타임 지원 라이브러리 역할을 하며, 독립적으로 유용하게 사용될 수 있습니다.</p>
<p>또한, 한 가지 기능이 추가된다면 이 모듈은 <code>setuptools</code>, <code>workingenv</code>, <code>zc.buildout</code>를 포함한 여러 Python 패키지 관리 도구의 쉬운 부트스트랩(bootstrap) 설치를 지원할 수 있을 것입니다.</p>
<h2>제안 (Proposal)</h2>
<p>이 PEP는 <code>setuptools</code> 전체를 표준 라이브러리에 포함시키는 대신, <code>pkg_resources</code>만을 Python 2.6 및 3.0의 표준 라이브러리에 추가할 것을 제안합니다. <code>pkg_resources</code>는 지난 12개월 동안 사실상 새로운 기능이 추가되지 않아 <code>setuptools</code>의 다른 부분보다 훨씬 안정적입니다.</p>
<p>그러나 이 PEP는 <code>pkg_resources</code>가 표준 라이브러리에 추가되기 전에 새로운 기능이 추가될 것을 제안합니다. 구체적으로, 다음과 같은 방식으로 <code>PyPI</code>에서 <code>SomePackage</code> 버전 1.2를 다운로드하고 설치하도록 요청할 수 있어야 합니다:</p>
<pre><code class="language-bash">python -m pkg_resources SomePackage==1.2
</code></pre>
<p>이 기능은 <code>easy_install</code>을 대체하는 것이 아닙니다. 대신, <code>PyPI XML-RPC API</code>를 통해 다운로드 가능한 순수 Python <code>.egg</code> 파일을 <code>SomePackage</code>가 가지고 있어야 하며, 이 egg 파일들은 <code>$PYTHON_EGG_CACHE</code> 디렉터리에 배치되어 기본적으로 <code>import</code>될 수 없습니다. (또한 스크립트도 설치되지 않습니다.) 하지만, 다운로드된 egg에 설치 부트스트랩 코드가 포함되어 있다면 실행될 기회가 주어집니다.</p>
<p>이러한 제약 사항들은 코드를 매우 단순하게 유지하면서도 <code>setuptools</code>, <code>workingenv</code>, <code>zc.buildout</code>와 같은 패키지 관리 도구를 사용자들이 명령줄에 도구 이름을 제공하는 것만으로 다운로드할 수 있도록 충분히 강력하게 지원할 수 있게 합니다.</p>
<h2>도입 배경 (Rationale)</h2>
<p>많은 사용자들이 <code>setuptools</code>를 표준 라이브러리에 포함하여 번거로운 부트스트래핑 과정을 줄여달라고 요청했습니다. 그러나 대부분의 부트스트래핑 복잡성은 <code>setuptools</code>가 이미 설치되어 있지 않으면 <code>setuptools</code>로 설치된 코드가 <code>pkg_resources</code> 런타임 모듈을 사용할 수 없다는 사실에서 비롯됩니다. 따라서 <code>setuptools</code>를 설치하려면 (어떤 의미에서는) <code>setuptools</code>가 이미 설치되어 있어야 합니다.</p>
<p><code>workingenv</code> 및 <code>zc.buildout</code>와 같은 다른 Python 패키지 관리 도구들도 유사한 부트스트래핑 문제를 가지고 있습니다. 이들 모두 <code>setuptools</code>를 사용하지만, 사용자에게 "원스텝(one-step) 설치"에 가까운 것을 제공하고자 합니다. <code>pkg_resources</code>가 이미 존재하고 <code>PyPI</code>에서 사전 패키지된 egg를 다운로드할 수 있다면, 이러한 도구 및 향후 등장할 유사한 도구들을 위한 부트스트랩 유틸리티를 만드는 복잡성이 크게 줄어듭니다.</p>
<p>(이는 또한 <code>setuptools</code>를 사용하여 egg를 빌드하는 것과 달리, 단순히 egg를 사용하기 위해 <code>setuptools</code>를 설치할 필요가 없음을 의미할 것입니다.)</p>
<p>마지막으로, <code>setuptools</code> 또는 다른 패키징 도구를 통해 빌드된 egg에 대한 접근을 제공하는 것 외에도, Python 2.5부터 <code>distutils</code>가 <code>sys.path</code>에 이미 어떤 배포판이 있는지 식별하기 위해 <code>pkg_resources</code>가 읽을 수 있는 패키지 메타데이터(일명 PKG-INFO) 파일을 설치한다는 점에 주목해야 합니다. <code>RPM</code>과 같은 시스템 패키지 도구를 사용하여 Python 패키지가 설치되는 환경에서, <code>pkg_resources</code> 모듈은 패키지가 <code>setuptools</code> 대신 <code>distutils</code>를 통해 설치되었더라도 어떤 버전의 어떤 패키지가 설치되어 있는지 감지하는 API를 제공합니다.</p>
<h2>구현 및 문서 (Implementation and Documentation)</h2>
<p><code>pkg_resources</code> 구현은 Python SVN 저장소의 <code>/sandbox/trunk/setuptools/</code> 경로에서 관리됩니다. <code>pkg_resources.py</code> 및 <code>pkg_resources.txt</code>를 참조하세요. <code>pkg_resources</code>가 지원하는 egg 형식에 대한 문서는 <code>doc/formats.txt</code>에서 찾을 수 있습니다. 이러한 문서의 HTML 버전은 다음 링크에서 확인할 수 있습니다:</p>
<ul>
<li>http://peak.telecommunity.com/DevCenter/PkgResources</li>
<li>http://peak.telecommunity.com/DevCenter/EggFormats</li>
</ul>
<p>(이 HTML 버전은 <code>setuptools 0.6</code>용이며, Subversion trunk의 <code>.txt</code> 버전에서 발견되는 모든 변경 사항을 반영하지 않을 수 있습니다.)</p>
