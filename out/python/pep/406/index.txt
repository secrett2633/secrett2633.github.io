<p><strong>원문 링크:</strong> <a href="https://peps.python.org/pep-0406/">PEP 406 - Improved Encapsulation of Import State</a></p>
<p><strong>상태:</strong> Withdrawn | <strong>유형:</strong> Standards Track | <strong>작성일:</strong> 04-Jul-2011</p>

<h2>PEP 406 – Import 상태의 캡슐화 개선</h2>
<p><strong>상태:</strong> Withdrawn (철회됨)
<strong>유형:</strong> Standards Track
<strong>작성자:</strong> Alyssa Coghlan, Greg Slodkowicz
<strong>생성일:</strong> 2011년 7월 4일
<strong>Python 버전:</strong> 3.4
<strong>최종 수정일:</strong> 2025년 2월 1일</p>
<hr>
<h3>초록 (Abstract)</h3>
<p>이 PEP는 <code>importlib</code>의 일부로 새로운 <code>ImportEngine</code> 클래스를 도입하여 모듈 임포트(import)와 관련된 모든 상태를 단일 객체로 캡슐화할 것을 제안합니다. 이 객체의 새 인스턴스를 생성하는 것은 <code>__import__()</code> 함수를 오버라이드(override)함으로써 내장된 임포트 문의 구현을 완전히 대체하는 대안을 제공할 것입니다. 내장 임포트 기능 및 <code>ImportEngine</code> 객체를 통한 임포트와 함께 작동하도록, 이 PEP는 전역 임포트 상태를 임시로 대체하기 위한 컨텍스트 관리(context management) 기반 접근 방식을 제안합니다.</p>
<p>또한 이 PEP는 <code>GlobalImportEngine</code> 서브클래스(subclass)와 해당 클래스의 전역 접근 가능 인스턴스를 포함할 것을 제안하는데, 이는 프로세스 전역 상태에 "쓰기 통과(writes through)"합니다. 이는 제안된 캡슐화된 API와 레거시(legacy) 프로세스 전역 상태 간의 하위 호환성(backwards compatible) 브리지를 제공하며, 관련된 상태 업데이트(예: <code>sys.path</code>가 수정될 때 경로 캐시 항목을 선택적으로 무효화하는 것)를 직접적으로 지원합니다.</p>
<h3>PEP 철회 이유 (PEP Withdrawal Rationale)</h3>
<p>이 PEP가 처음 작성된 이후, Python 3.3의 PEP 420과 Python 3.4의 PEP 451의 일환으로 임포트 시스템에 상당한 변경 사항이 있었습니다.</p>
<p>임포트 상태의 캡슐화를 제공하는 것은 여전히 매우 바람직하지만, PEP 451을 기반으로 새로운 PEP에서 다루는 것이 더 적합하며, PEP 451 호환 <code>finder</code> 및 <code>loader</code>만 사용하도록 허용하는 것이 좋습니다 (이는 이전 <code>loader</code> API와 관련된 전역 상태의 직접 조작 문제를 많이 피할 수 있기 때문입니다).</p>
<h3>제안 배경 (Rationale)</h3>
<p>현재 임포트 시스템과 관련된 대부분의 상태는 <code>sys</code> 모듈의 모듈 수준 속성(module level attributes)으로 저장됩니다. 유일한 예외는 임포트 락(lock)인데, 이는 직접 접근할 수 없고 <code>imp</code> 모듈의 관련 함수를 통해서만 접근 가능합니다. 현재의 프로세스 전역 임포트 상태는 다음을 포함합니다.</p>
<ul>
<li><code>sys.modules</code></li>
<li><code>sys.path</code></li>
<li><code>sys.path_hooks</code></li>
<li><code>sys.meta_path</code></li>
<li><code>sys.path_importer_cache</code></li>
<li>임포트 락 (<code>imp.lock_held()</code>, <code>acquire_lock()</code>, <code>release_lock()</code>)</li>
</ul>
<p>이 상태를 격리하면 여러 임포트 상태를 프로세스 내에 편리하게 저장할 수 있습니다. 임포트 기능을 자체 포함된 객체에 배치하면 추가 기능(예: 모듈 임포트 알림 또는 어떤 모듈을 임포트할 수 있는지에 대한 세밀한 제어)을 추가하기 위해 서브클래싱(subclassing)도 가능해집니다. 이 엔진은 기존 프로세스 전역 상태와 상호 작용하기 위해 임포트 엔진 API를 사용할 수 있도록 서브클래스화될 수도 있습니다.</p>
<p>네임스페이스(namespace) PEPs (특히 PEP 402)는 <code>sys.path</code>가 수정될 때 패키지 경로를 올바르게 업데이트하기 위해 추가적인 프로세스 전역 상태의 잠재적인 필요성을 제기합니다.</p>
<p>마지막으로, 이 모든 상태에 대해 일관된 객체를 제공하면 임포트 상태를 임시로 대체할 수 있는 컨텍스트 관리 기능을 제공하는 것도 가능해집니다.</p>
<h3>제안 (Proposal)</h3>
<p>저희는 임포트 기능을 캡슐화하기 위해 <code>ImportEngine</code> 클래스를 도입할 것을 제안합니다. 여기에는 필요할 때 내장된 <code>__import__()</code>의 대안으로 사용될 수 있는 <code>__import__()</code> 메서드와 <code>importlib.import_module()</code>과 동등한 <code>import_module()</code> 메서드가 포함됩니다.</p>
<p>가정되고 유지되어야 할 전역 임포트 상태 불변식(invariants)이 존재하기 때문에, 우리는 <code>ImportEngine</code>과 동일한 인터페이스를 가지지만 현재 전역 임포트 상태에 직접 접근하는 <code>GlobalImportState</code> 클래스를 도입합니다. 이는 클래스 속성(class properties)을 사용하여 쉽게 구현될 수 있습니다.</p>
<h3>세부 사항 (Specification)</h3>
<h4><code>ImportEngine</code> API</h4>
<p>제안된 확장은 다음 객체들로 구성됩니다.</p>
<ul>
<li>
<p><strong><code>importlib.engine.ImportEngine</code></strong></p>
<ul>
<li><strong><code>from_engine(self, other)</code></strong>: 다른 <code>ImportEngine</code> 인스턴스로부터 새로운 임포트 객체를 생성합니다. 새 객체는 <code>other</code>의 상태 복사본으로 초기화됩니다. <code>importlib.engine.sysengine</code>에서 호출될 때, <code>from_engine()</code>은 전역 임포트 상태의 복사본으로 <code>ImportEngine</code> 객체를 생성하는 데 사용될 수 있습니다.</li>
<li><strong><code>__import__(self, name, globals={}, locals={}, fromlist=[], level=0)</code></strong>: 내장 <code>__import__()</code> 함수의 재구현입니다. 모듈 임포트는 전역 임포트 상태 대신 <code>ImportEngine</code> 인스턴스에 저장된 상태를 사용하여 진행됩니다. <code>__import__</code> 기능의 전체 문서는를 참조하십시오. <code>ImportEngine</code> 및 그 서브클래스의 <code>__import__()</code>는 <code>__builtin__.__import__</code>를 <code>ImportEngine().__import__</code>로 교체하여 임포트 문의 동작을 사용자 정의하는 데 사용될 수 있습니다.</li>
<li><strong><code>import_module(name, package=None)</code></strong>: <code>ImportEngine</code> 인스턴스에 저장된 임포트 상태를 사용하는 <code>importlib.import_module()</code>의 재구현입니다. 전체 참조는을 참조하십시오.</li>
<li><strong><code>modules</code>, <code>path</code>, <code>path_hooks</code>, <code>meta_path</code>, <code>path_importer_cache</code></strong>: 프로세스 전역 <code>sys</code>의 해당 속성들의 인스턴스별 버전입니다.</li>
</ul>
</li>
<li>
<p><strong><code>importlib.engine.GlobalImportEngine(ImportEngine)</code></strong></p>
<ul>
<li>전역 상태에 엔진과 유사한 접근을 제공하는 편의 클래스입니다. <code>ImportEngine</code>과 마찬가지로 <code>__import__()</code>, <code>import_module()</code>, <code>from_engine()</code> 메서드를 제공하지만, <code>sys</code>의 전역 상태에 쓰기 통과합니다.</li>
<li>다양한 네임스페이스 패키지 메커니즘을 지원하기 위해 <code>sys.path</code>가 변경될 때 <code>pkgutil.extend_path</code>와 같은 도구는 임포트 상태의 다른 부분(이 경우 패키지 <code>__path__</code> 속성)도 수정해야 합니다. 경로 임포터 캐시도 다양한 변경이 있을 때 무효화되어야 합니다.</li>
<li><code>ImportEngine</code> API는 단일 작업의 일부로 관련된 임포트 상태 업데이트를 자동으로 수행하는 편의 메서드를 제공할 것입니다.</li>
</ul>
</li>
</ul>
<h4>전역 변수 (Global variables)</h4>
<ul>
<li><strong><code>importlib.engine.sysengine</code></strong>: <code>GlobalImportEngine</code>의 미리 생성된 인스턴스입니다. 선택적 엔진 매개변수를 받아들이도록 업데이트된 임포터(importer) 및 로더(loader)와 <code>ImportEngine.from_engine(sysengine)</code>를 사용하여 프로세스 전역 임포트 상태의 복사본으로 시작하는 데 사용될 예정입니다.</li>
</ul>
<h4><code>finder</code>/<code>loader</code> 인터페이스의 변경 없음 (No changes to finder/loader interfaces)</h4>
<p>PEP 302 API를 추가 상태를 받아들이도록 업데이트하려는 시도 대신, 이 PEP는 <code>ImportEngine</code>이 컨텍스트 관리 프로토콜(decimal 모듈의 컨텍스트 대체 메커니즘과 유사)을 지원할 것을 제안합니다.</p>
<p><code>ImportEngine</code>의 컨텍스트 관리 메커니즘은 다음과 같습니다.</p>
<ul>
<li><strong>진입 시 (On entry):</strong>
<ul>
<li>임포트 락 획득</li>
<li>전역 임포트 상태를 임포트 엔진 자체의 상태로 대체</li>
</ul>
</li>
<li><strong>종료 시 (On exit):</strong>
<ul>
<li>이전 전역 임포트 상태 복원</li>
<li>임포트 락 해제</li>
</ul>
</li>
</ul>
<p>이에 대한 정확한 API는 미정입니다 (그러나 <code>decimal.localcontext</code>에 의해 생성된 것과 유사하게 별도의 컨텍스트 관리 객체를 사용할 가능성이 높습니다).</p>
<h3>미해결 문제 (Open Issues)</h3>
<ul>
<li><strong>전역 임포트 상태로 폴백(fall back)하기 위한 API 설계</strong>: 현재 제안은 <code>from_engine()</code> API에 의존하여 전역 임포트 상태로 폴백합니다. 대신 동적으로 전역 임포트 상태로 폴백하는 변형을 제공하는 것이 바람직할 수 있습니다. 그러나 "가능한 한 고립된" 디자인으로 시작하는 큰 장점 중 하나는 엔진 인스턴스 상태와 프로세스 전역 상태 간의 경계를 다양한 방식으로 흐리게 하는 서브클래스를 실험할 수 있다는 것입니다.</li>
<li><strong>내장(Builtin) 및 확장(Extension) 모듈은 프로세스 전역이어야 함</strong>: 플랫폼 제한으로 인해 각 내장 및 확장 모듈의 복사본은 각 프로세스에 하나만 존재할 수 있습니다. 따라서 각 <code>ImportEngine</code> 인스턴스가 이러한 모듈을 독립적으로 로드하는 것은 불가능합니다. 가장 간단한 해결책은 <code>ImportEngine</code>이 이러한 모듈 로드를 거부하고 <code>ImportError</code>를 발생하는 것입니다. <code>GlobalImportEngine</code>은 이러한 모듈을 정상적으로 로드할 수 있습니다. <code>ImportEngine</code>은 미리 채워진 모듈 캐시에서 이러한 모듈을 계속 반환할 것입니다. 직접 로드하는 경우에만 문제가 발생합니다.</li>
<li><strong>대체의 범위 (Scope of substitution)</strong>: 이전 미해결 문제와 관련하여 컨텍스트 관리 API를 사용할 때 어떤 상태를 대체할 것인가 하는 질문이 있습니다. <code>sys.modules</code>를 교체하는 것은 캐시된 참조 때문에 신뢰할 수 없을 수 있으며, 일부 모듈의 독립적인 복사본을 가지는 것이 플랫폼 제한으로 인해 단순히 불가능하다는 근본적인 사실이 존재합니다. 이 PEP의 일부로서 다음을 명시적으로 문서화해야 할 필요가 있습니다.
<ul>
<li>전역 임포트 상태의 어떤 부분을 대체할 수 있는지 (그리고 해당 상태에 대한 참조를 캐시하면서 대체 사례를 처리하지 않는 코드를 버그로 선언).</li>
<li>어떤 부분을 제자리에서 수정해야 하는지 (따라서 <code>ImportEngine</code> 컨텍스트 관리 API에 의해 대체되지 않거나 <code>ImportEngine</code> 인스턴스에 범위가 지정되지 않음).</li>
</ul>
</li>
</ul>
<h3>참조 구현 (Reference Implementation)</h3>
<p>Greg Slodkowicz는 2011년 Google Summer of Code의 일환으로 Brett Cannon의 <code>importlib</code>을 기반으로 한 이 PEP의 초기 초안에 대한 참조 구현을 개발했습니다. 현재 구현은 기존 코드를 수정하는 것을 피하므로 불필요하게 많은 것을 복제합니다. 실제 구현은 영향을 받는 모든 코드를 제자리에서 수정할 것입니다.</p>
<p>해당 PEP의 초기 초안은 선택적 엔진 인스턴스를 전달하는 것을 지원하기 위해 PEP 302 API를 변경할 것을 제안했습니다. 이는 임포트된 모듈로부터의 추가 임포트에 제대로 영향을 미치지 않는 (심각한) 단점이 있었기 때문에, 전역 상태를 대체하기 위한 컨텍스트 관리 기반 제안으로 변경되었습니다.</p>
