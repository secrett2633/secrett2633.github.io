<p>주어진 URL의 PEP 554 문서를 바탕으로 한국어 번역 및 요약을 제공합니다. 이 문서는 Python 3.12에서 [PEP 684]를 통해 per-interpreter GIL이 도입된 이후 특히 중요해진 <code>stdlib</code>에 다중 인터프리터를 추가하는 제안입니다.</p>
<hr>
<h1>PEP 554 – <code>stdlib</code>의 다중 인터프리터 (Multiple Interpreters in the Stdlib)</h1>
<ul>
<li><strong>작성자:</strong> Eric Snow</li>
<li><strong>상태:</strong> 대체됨 (Superseded) - 이 PEP의 내용은 [PEP 734]로 이어집니다.</li>
<li><strong>유형:</strong> 표준 트랙 (Standards Track)</li>
<li><strong>생성일:</strong> 2017년 9월 5일</li>
<li><strong>Python 버전:</strong> 3.13</li>
<li><strong>대체된 PEP:</strong> [PEP 734]</li>
</ul>
<h2>개요 (Abstract)</h2>
<p>CPython은 1.5 버전(1997년)부터 동일 프로세스 내에서 여러 개의 인터프리터("서브 인터프리터"라고도 함)를 지원해왔습니다. 이 기능은 C-API를 통해 사용할 수 있었으며, 서로 간의 상대적인 격리를 통해 새로운 형태의 동시성(concurrency) 접근 방식을 가능하게 합니다.</p>
<p>이 제안은 <code>stdlib</code>에 <code>interpreters</code> 모듈을 도입합니다. 이 모듈은 C-API에서 이미 제공하는 다중 인터프리터의 기본 기능과 인터프리터 간 통신을 위한 기본 지원을 노출합니다. 특히 [PEP 684]가 Python 3.12에 per-interpreter GIL을 도입한 이후 이 모듈은 더욱 중요해졌습니다.</p>
<h2>제안 (Proposal)</h2>
<p>핵심 요약은 다음과 같습니다.</p>
<ol>
<li>새로운 <code>stdlib</code> 모듈인 "<code>interpreters</code>"를 추가합니다.</li>
<li><code>concurrent.futures.InterpreterPoolExecutor</code>를 추가합니다.</li>
<li>확장 모듈(Extension Module) 관리자를 위한 도움을 제공합니다.</li>
</ol>
<h3><code>interpreters</code> 모듈 (The “interpreters” Module)</h3>
<p><code>interpreters</code> 모듈은 다중 인터프리터 기능에 대한 고수준 인터페이스를 제공하며, 새로운 저수준 <code>_interpreters</code> 모듈(threading 모듈과 유사)을 래핑합니다. 이 모듈은 CPython에 이미 존재하는 다중 인터프리터 지원을 노출하는 것 외에도, 인터프리터 간에 데이터를 전달하기 위한 기본적인 메커니즘을 지원합니다. 이는 대상 서브 인터프리터의 <code>__main__</code> 모듈에 "공유 가능한(shareable)" 객체를 설정하는 것을 포함합니다. <code>os.pipe()</code>와 같은 일부 객체는 추가 통신에 사용될 수 있습니다. 이 모듈은 또한 교차 인터프리터 통신의 시연으로 "채널(channels)"의 최소한의 구현을 제공할 것입니다.</p>
<p>객체는 생성된 인터프리터에 묶여 있기 때문에 인터프리터 간에 공유되지 않습니다. 대신, 객체의 데이터가 인터프리터 간에 전달됩니다.</p>
<h3><code>interpreters</code> 모듈 API 요약</h3>
<p>다음은 <code>interpreters</code> 모듈의 API 요약입니다.</p>
<p><strong>인터프리터 생성 및 사용:</strong></p>
<ul>
<li><code>list_all() -> [Interpreter]</code>: 존재하는 모든 인터프리터를 가져옵니다.</li>
<li><code>get_current() -> Interpreter</code>: 현재 실행 중인 인터프리터를 가져옵니다.</li>
<li><code>get_main() -> Interpreter</code>: 메인 인터프리터를 가져옵니다.</li>
<li><code>create() -> Interpreter</code>: 새로운 (유휴) Python 인터프리터를 초기화합니다.</li>
</ul>
<p><strong><code>class Interpreter</code>:</strong></p>
<ul>
<li><code>.id</code>: 인터프리터의 ID (읽기 전용).</li>
<li><code>.is_running() -> bool</code>: 인터프리터가 현재 코드를 실행 중인지 여부.</li>
<li><code>.close()</code>: 인터프리터를 종료하고 파괴합니다.</li>
<li><code>.set_main_attrs(**kwargs)</code>: <code>__main__</code>에 "공유 가능한" 객체를 바인딩합니다.</li>
<li><code>.get_main_attr(name)</code>: <code>__main__</code>에서 "공유 가능한" 객체를 가져옵니다.</li>
<li><code>.exec(src_str, /)</code>: 주어진 소스 코드를 (현재 스레드에서) 인터프리터에서 실행합니다.</li>
</ul>
<p><strong>인터프리터 간 통신:</strong></p>
<ul>
<li><code>is_shareable(obj) -> Bool</code>: 객체의 데이터를 인터프리터 간에 전달할 수 있는지 여부.</li>
<li><code>create_channel() -> (RecvChannel, SendChannel)</code>: 인터프리터 간 데이터 전달을 위한 새로운 채널을 생성합니다.</li>
</ul>
<h3><code>concurrent.futures.InterpreterPoolExecutor</code></h3>
<p><code>ThreadPoolExecutor</code>를 확장하여 각 스레드 작업을 서브 인터프리터에서 실행하는 실행자(executor)가 추가될 것입니다. 초기에는 <code>Interpreter.exec()</code>가 받아들이는 작업(예: 문자열 스크립트)만 지원되지만, 나중에는 함수 지원 및 작업과 인수를 pickle로 직렬화하는 별도의 메서드도 지원될 수 있습니다.</p>
<h3>확장 모듈 관리자를 위한 도움 (Help for Extension Module Maintainers)</h3>
<p>다단계 초기화([PEP 489])를 구현하는 확장 모듈은 격리되어 다중 인터프리터와 호환되는 것으로 간주됩니다. 그렇지 않은 경우 "호환되지 않는" 것으로 간주됩니다.</p>
<p>많은 확장 모듈이 여전히 호환되지 않습니다. 호환성 문제를 완화하고 호환성을 가속화하기 위해 다음과 같은 조치를 취할 것입니다.</p>
<ul>
<li>확장 모듈이 다중 인터프리터 사용을 지원할 필요가 없음을 명확히 합니다.</li>
<li>호환되지 않는 모듈이 서브 인터프리터에서 임포트될 때 <code>ImportError</code>를 발생시킵니다.</li>
<li>관리자가 호환성을 달성하는 데 도움이 되는 자료(예: 문서)를 제공합니다.</li>
<li>Cython 및 PyPI에서 가장 많이 사용되는 확장 모듈의 관리자에게 연락하여 피드백을 받고 지원을 제공합니다.</li>
</ul>

<h2>도입 배경 (Rationale)</h2>
<p>동일 프로세스 내에서 여러 인터프리터에서 코드를 실행하는 것은 유용한 수준의 격리를 제공합니다. 이는 여러 방식으로 활용될 수 있으며, 서브 인터프리터는 이러한 격리를 확장할 수 있는 잘 정의된 프레임워크를 제공합니다 ([PEP 684] 참조).</p>
<p>Alyssa (Nick) Coghlan은 멀티 프로세싱과의 비교를 통해 몇 가지 이점을 설명했습니다. 서브 인터프리터 간 통신은 공유 메모리를 통한 서브 프로세스 통신과 유사하게 작동할 것으로 예상됩니다. 외부 관점에서 단일 프로세스로 보이며 OS에 추가적인 부담을 주지 않는다는 장점이 있습니다. 또한, OS가 기본적으로 제공하는 프로세스 격리에 구멍을 뚫을 필요가 없으므로 보안 위험 프로필도 다릅니다.</p>
<p>CPython은 1.5 버전부터 다중 인터프리터를 지원했지만, C-API를 통해서만 접근 가능했기 때문에 널리 사용되지 못했습니다. <code>stdlib</code>에 기존 기능을 노출하면 이러한 상황을 역전시키는 데 도움이 될 것입니다.</p>
<p>이 제안은 동일한 Python 프로세스 내에서 서로 격리된 다중 인터프리터의 기본 기능을 활성화하는 데 중점을 둡니다.</p>
<h3>우려 사항 (Concerns)</h3>
<ul>
<li><strong>"서브 인터프리터는 그만한 가치가 없다"</strong> : 일부는 서브 인터프리터가 Python의 공식적인 부분이 될 만큼 충분한 이점을 제공하지 않는다고 주장합니다. 그러나 다중 인터프리터는 격리된 실행 스레드에 초점을 맞춘 새로운 동시성 모델을 제공하며, [PEP 684]를 통해 GIL로 인해 불가능했던 다중 CPU 코어 동시 사용 가능성을 제공합니다.</li>
<li><strong>"stdlib에 다중 인터프리터 지원을 추가하는 것은 C 확장 모듈 작성자에게 추가적인 부담을 준다"</strong> : C 전역 변수를 사용하여 내부 상태를 저장하는 확장 모듈은 다중 인터프리터 환경에서 문제가 될 수 있습니다. [PEP 3121] 및 [PEP 489]는 이 문제에 대한 해결책을 제시했습니다. 이 PEP는 실제 추가 유지 관리 부담이 적고, 서브 인터프리터의 전반적인 이점이 이러한 비용을 상쇄할 만큼 충분히 크다고 주장합니다.</li>
<li><strong>"새로운 동시성 API를 만드는 것은 더 많은 고민과 실험이 필요하다"</strong> : 이 PEP는 새로운 동시성 API를 제안하는 것이 아닙니다. 대신, Python에는 비교적 새로운 동시성 모델과 관련된 패턴을 따르는 코드를 작성하는 데 사용될 수 있는 최소한의 도구(예: 서브 인터프리터, 채널)를 노출합니다.</li>
<li><strong>"GIL을 여전히 공유한다면 서브 인터프리터를 노출하는 것은 의미가 없다"</strong> : 이는 흔한 오해입니다. 이 PEP는 인터프리터가 더 이상 GIL을 공유하지 않는다는 약속을 포함하지 않습니다. 가치는 기존 기능의 노출을 늘리고, (대부분) 격리된 인터프리터 실행을 노출하며, per-interpreter GIL을 준비하고, 실험을 장려하는 데 있습니다.</li>
<li><strong>"데이터 공유는 멀티 코어 시나리오에서 캐시 성능에 부정적인 영향을 미칠 수 있다"</strong> : 현재 인터프리터 간에 데이터를 실제로 공유할 계획이 없으며, 대신 복사하는 데 중점을 두므로 문제가 되지 않을 것입니다.</li>
</ul>
<h3>서브 인터프리터에 대하여 (About Subinterpreters)</h3>
<ul>
<li><strong>동시성 (Concurrency)</strong> : CPython의 인터프리터에 내재된 의도적인 격리는 Communicating Sequential Processes (CSP)와 같은 메시지 전달 방식을 통한 격리된 실행 스레드에 초점을 맞춘 동시성 모델에 잘 맞습니다.</li>
<li><strong>공유 데이터 (Shared Data)</strong> : CPython의 인터프리터는 스레드와 달리 본질적으로 격리되어 있습니다. 따라서 동일한 공유 메모리 통신 접근 방식은 작동하지 않습니다. 객체는 생성된 인터프리터에 묶여 있으므로 인터프리터 간에 객체를 안전하게 공유하는 데 제약이 있습니다. 이 제안은 <code>Interpreter.set_main_attrs()</code>를 사용하여 <code>__main__</code> 모듈에 "공유 가능한" 객체를 설정하는 최소한의 솔루션을 제시합니다.</li>
<li><strong>인터프리터 격리 (Interpreter Isolation)</strong> : CPython의 인터프리터는 서로 엄격하게 격리되도록 의도되었습니다. 각 인터프리터는 모든 모듈, 클래스, 함수 및 변수의 자체 사본을 가집니다. 그러나 파일 디스크립터, 저수준 환경 변수, 프로세스 메모리, 내장 타입(예: <code>dict</code>, <code>bytes</code>), 싱글톤(예: <code>None</code>), 내장/확장/frozen 모듈의 기초 정적 모듈 데이터와 같은 일부 프로세스 전역 상태는 공유됩니다. 또한, C 전역 변수에 의존하는 확장 모듈과 같은 버그나 잘못된 구현으로 인해 일부 격리가 불완전합니다.</li>
<li><strong>기존 사용 (Existing Usage)</strong> : 다중 인터프리터 지원은 널리 사용되는 기능은 아니었습니다. <code>mod_wsgi</code>, OpenStack Ceph, JEP 등 몇 가지 사례가 있었지만, 기능의 유용성을 판단할 수 있는 충분한 샘플은 아닙니다.</li>
</ul>
<h3><code>interpreters</code> 모듈 API</h3>
<p>모듈은 다음 함수들을 제공합니다.</p>
<ul>
<li><code>list_all() -> [Interpreter]</code>: 존재하는 모든 인터프리터 목록을 반환합니다.</li>
<li><code>get_current() -> Interpreter</code>: 현재 실행 중인 인터프리터를 반환합니다.</li>
<li><code>get_main() -> Interpreter</code>: 메인 인터프리터를 반환합니다.</li>
<li><code>create() -> Interpreter</code>: 새 Python 인터프리터를 초기화하고 반환합니다.</li>
<li><code>is_shareable(obj) -> bool</code>: 객체가 인터프리터 간에 "공유"될 수 있는지 여부를 반환합니다.</li>
</ul>
<p><code>Interpreter</code> 클래스의 상세 메서드는 "API summary for interpreters module" 섹션을 참조하십시오.</p>
<h3>잡히지 않은 예외 (Uncaught Exceptions)</h3>
<p><code>Interpreter.exec()</code>에서 잡히지 않은 예외는 <code>interp.exec()</code>가 호출된 코드에 "효과적으로" 전파됩니다. 예외가 인터프리터 간에 유출되는 것을 방지하기 위해, 예외와 트레이스백의 대리자(surrogate)를 생성하여 (<code>traceback.TracebackException</code> 참조) 새로운 <code>interpreters.RunFailedError</code>의 <code>__cause__</code>로 설정하고 이 <code>RunFailedError</code>를 발생시킵니다.</p>
<h3>인터프리터 제한 사항 (Interpreter Restrictions)</h3>
<p><code>interpreters.create()</code>로 생성된 모든 새 인터프리터는 실행하는 코드에 특정 제한 사항이 있습니다.</p>
<ul>
<li>다단계 초기화를 구현하지 않는 확장 모듈 임포트 실패.</li>
<li>데몬 스레드(daemon threads) 생성 불가.</li>
<li><code>os.fork()</code> 허용 안됨 (따라서 멀티프로세싱 불가).</li>
<li><code>os.exec*()</code> 허용 안됨 (하지만 서브프로세스처럼 "fork+exec"는 허용).</li>
</ul>
<p>기존 C-API로 생성된 인터프리터와 "메인" 인터프리터는 이러한 제한 사항이 없습니다.</p>
<h3>통신 API (API For Communication)</h3>
<p>객체 간 데이터 공유(통신) 메커니즘이 없으면 다중 인터프리터 지원은 덜 유용합니다. 이 제안에서는 실제 Python 객체를 인터프리터 간에 공유하는 것을 피하고, <code>Interpreter.set_main_attrs()</code> 메서드를 통해 <code>__main__</code> 모듈의 전역 변수를 설정하는 기본 메커니즘을 제공합니다. 값은 "공유 가능(shareable)"해야 합니다.</p>
<h4>공유 가능한 타입 (Shareable Types)</h4>
<p>객체는 해당 타입이 공유 가능한 인스턴스를 지원하는 경우 "공유 가능"합니다. 타입은 객체를 인터프리터 독립적인 데이터로 변환하고 다른 쪽에서 다시 객체로 변환하는 데 사용되는 새로운 내부 프로토콜을 구현해야 합니다.</p>
<p>초기에는 다음과 같은 최소한의 간단하고 불변하는 내장 타입이 지원될 것입니다.</p>
<ul>
<li><code>None</code></li>
<li><code>bool</code></li>
<li><code>bytes</code></li>
<li><code>str</code></li>
<li><code>int</code></li>
<li><code>float</code></li>
</ul>
<p>또한, 초기에는 <code>memoryview</code> (PEP 3118 버퍼 공유 허용) 및 채널과 같은 소수의 복잡한 타입도 지원됩니다.</p>
<h4>OS 파이프를 통한 통신 (Communicating Through OS Pipes)</h4>
<p>전용 통신 객체가 없더라도 <code>os.pipe()</code>와 같은 기존 도구를 사용할 수 있습니다.</p>
<h4>채널 (Channels)</h4>
<p><code>interpreters</code> 모듈에는 인터프리터 간에 객체 데이터를 전달하기 위한 전용 솔루션인 <strong>채널</strong> 이 포함될 것입니다. 채널은 파이프, 큐, CSP의 채널에서 영감을 받은 단방향 FIFO(First-In, First-Out) 메커니즘입니다. 채널은 <code>send</code>와 <code>receive</code> 두 가지 작업을 가집니다. 핵심은 채널이 Python 객체 자체보다는 객체에서 파생된 데이터를 전송한다는 것입니다.</p>
<p><code>interpreters</code> 모듈은 채널과 관련된 다음 함수를 제공합니다.</p>
<ul>
<li><code>create_channel() -> (RecvChannel, SendChannel)</code>: 새 채널을 생성하고 채널 끝에 해당하는 <code>RecvChannel</code>과 <code>SendChannel</code>을 반환합니다.</li>
</ul>
<p>모듈은 또한 다음 채널 관련 클래스를 제공합니다.</p>
<ul>
<li><code>class RecvChannel(id)</code>: 채널의 수신 끝.
<ul>
<li><code>recv(*, timeout=None)</code>: 채널에서 다음 객체를 반환합니다.</li>
<li><code>recv_nowait(default=None)</code>: 채널에서 다음 객체를 반환합니다.</li>
</ul>
</li>
<li><code>class SendChannel(id)</code>: 채널의 송신 끝.
<ul>
<li><code>send(obj, *, timeout=None)</code>: 객체(즉, 해당 데이터)를 채널의 "recv" 끝으로 보냅니다.</li>
<li><code>send_nowait(obj)</code>: 객체를 채널의 "recv" 끝으로 보냅니다.</li>
</ul>
</li>
</ul>
<h4>공유 객체에 대한 주의사항 (Caveats For Shared Objects)</h4>
<p>다시 말하지만, Python 객체는 인터프리터 간에 공유되지 않습니다. 그러나 어떤 경우에는 해당 객체가 래핑하는 데이터가 실제로 공유될 수도 있습니다(예: [PEP 3118] 버퍼). 이러한 경우 원본 인터프리터의 객체는 다른 인터프리터에서 공유 데이터가 더 이상 사용되지 않을 때까지 유지됩니다.</p>
<h3>문서화 (Documentation)</h3>
<p>새로운 <code>stdlib</code> 문서 페이지에는 다음 내용이 포함될 것입니다.</p>
<ul>
<li>확장 모듈이 다중 인터프리터 지원을 요구하지 않는다는 명확한 주석.</li>
<li>서브 인터프리터에 대한 설명.</li>
<li>다중 인터프리터 사용 방법(및 통신 방법)에 대한 간략한 예시.</li>
<li>다중 인터프리터 사용의 한계 요약.</li>
<li>다중 인터프리터 호환성을 보장하기 위한 자료 링크.</li>
<li>이 PEP의 API 정보 대부분.</li>
</ul>
<h2>대안 솔루션 (Alternative Solutions)</h2>
<p><code>concurrent.futures</code> 또는 <code>multiprocessing</code> 모듈에 인터프리터 지원을 추가하는 것은 적합하지 않습니다. 명확한 지원은 <code>interpreters</code> 모듈에 있어야 합니다.</p>

<h2>연기된 기능 (Deferred Functionality)</h2>
<p>이 제안을 최소한으로 유지하기 위해 다음 기능들은 향후 고려를 위해 제외되었습니다.</p>
<ul>
<li>편의성 API 추가 (예: <code>Interpreter.run()</code> 또는 <code>Interpreter.call()</code>).</li>
<li>현재 스레드에서 실행되는 인터프리터에 대한 혼란 방지.</li>
<li>"<code>running</code>"과 "<code>has threads</code>" 명확화.</li>
<li>공유를 위한 Dunder 메서드 (<code>__xid__</code>).</li>
<li><code>Interpreter.call()</code>.</li>
<li><code>Interpreter.run_in_thread()</code>.</li>
<li>동기화 프리미티브(Synchronization Primitives).</li>
<li>CSP 라이브러리.</li>
<li>문법적 지원.</li>
<li><code>multiprocessing</code> 통합.</li>
<li>C 확장 모듈 <code>opt-in/opt-out</code>.</li>
<li>채널 오염(Poisoning channels).</li>
<li><code>__main__</code> 재설정.</li>
<li>인터프리터 상태 재설정 (<code>Interpreter.reset()</code>).</li>
<li>기존 인터프리터 상태 복사 (<code>Interpreter.copy()</code>).</li>
<li>공유 가능한 파일 디스크립터 및 소켓.</li>
<li><code>async</code> 통합.</li>
<li>이터레이션(iteration) 지원.</li>
<li>채널 컨텍스트 관리자.</li>
<li>파이프 및 큐.</li>
<li><code>send()</code>에서 락 반환.</li>
<li>채널의 우선순위 지원.</li>
<li>설정 상속 지원.</li>
<li>예외 공유 가능하게 만들기.</li>
<li>직렬화를 통해 모든 것을 공유 가능하게 만들기.</li>
<li><code>RunFailedError.__cause__</code>를 lazy하게 만들기.</li>
<li><code>interp.exec()</code>에서 값 반환.</li>
<li>공유 가능한 동기화 프리미티브 추가.</li>
<li><code>SystemExit</code> 및 <code>KeyboardInterrupt</code> 다르게 전파.</li>
<li>채널 끝 클래스에 명시적 <code>release()</code> 및 <code>close()</code> 추가.</li>
<li><code>SendChannel.send_buffer()</code> 추가.</li>
<li>스레드에서 자동 실행.</li>
</ul>
<h2>거부된 아이디어 (Rejected Ideas)</h2>
<ul>
<li><strong>명시적 채널 연관 (Explicit channel association)</strong> : <code>recv()</code> 및 <code>send()</code> 호출 시 인터프리터가 채널과 암시적으로 연관되는 방식이 유지됩니다. 명시적인 <code>add_channel()</code>/<code>remove_channel()</code> 메서드는 불필요한 복잡성만 추가한다고 판단되었습니다.</li>
<li><strong>파이프 기반 API 추가</strong> : 두 인터프리터 간의 단방향 FIFO 파이프는 대부분의 사용 사례에 충분하지만, 채널을 통한 다대다 단방향 FIFO를 지원하는 것이 더 큰 이점을 제공합니다.</li>
<li><strong>큐 기반 API 추가</strong> : 큐와 버퍼링된 채널은 거의 동일하지만, <code>stdlib</code>의 <code>queue.Queue</code>와의 혼동을 피하기 위해 "Channel"이라는 이름이 사용되었습니다.</li>
<li><strong>"enumerate"</strong> : <code>list_all()</code> 함수는 모든 인터프리터 목록을 제공합니다. <code>threading</code> 모듈의 <code>enumerate()</code>와 달리, "list_all"이라는 이름은 Python 사용자에게 더 명확하다고 판단되었습니다.</li>
<li><strong>인터프리터 간 예외 유출 방지를 위한 대안 솔루션</strong> : <code>RunFailedError</code>를 발생시키고 <code>__cause__</code>에 원본 예외의 안전한 프록시를 래핑하는 현재 방식이 채택되었습니다. 이는 예외 객체 및 트레이스백이 인터프리터 경계를 넘어 유출되는 것을 방지합니다.</li>
<li><strong>각 새 인터프리터를 항상 자체 스레드와 연관시키기</strong> : 인터프리터는 본질적으로 스레드에 묶여 있지 않으며, <code>Interpreter.exec()</code> 호출만 스레드에 묶입니다. 각 인터프리터를 자체 스레드와 연관시키는 것은 C-API와의 차이, 암시적인 스레드 생성의 마법, 비효율성, 유효한 사용 사례 방해 등의 문제로 거부되었습니다.</li>
<li><strong><code>Interpreter.exec()</code>에 여러 동시 호출 허용</strong> : 각 인터프리터는 하나의 <code>__main__</code> 모듈만 가지므로, 동시 호출을 허용하면 <code>__main__</code> 공유를 처리하거나 새로운 메커니즘을 발명해야 하는 복잡성이 발생합니다.</li>
<li><strong><code>RunFailedError</code>에 "reraise" 메서드 추가</strong> : <code>__cause__</code>를 사용하는 것만으로도 충분히 유용하다고 판단되어 <code>reraise</code> 메서드는 불필요하다고 판단되었습니다.</li>
</ul>
<h2>구현 (Implementation)</h2>
<p>PEP의 구현은 4가지 부분으로 나뉩니다.</p>
<ol>
<li>이 PEP에서 설명하는 고수준 모듈 (대부분 저수준 C 확장 모듈의 가벼운 래퍼)</li>
<li>저수준 C 확장 모듈.</li>
<li>저수준 모듈에 필요한 내부 C-API에 대한 추가 기능.</li>
<li>저수준 모듈을 촉진하는 CPython 런타임의 보조 수정/변경.</li>
</ol>
<p>대부분의 구현은 이미 완료되었거나 거의 완료되었습니다. 특히 <code>_xxsubinterpreters</code> 모듈로 병합된 저수준 모듈은 테스트 목적으로 사용되고 있습니다.
