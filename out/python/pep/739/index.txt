<h2>PEP 739 – <code>build-details.json</code> 1.0 — Python 빌드 세부 정보를 위한 정적 설명 파일</h2>
<h3>개요 (Abstract)</h3>
<p>이 PEP는 Python 설치의 빌드 세부 정보를 포함하는 정적 설명 파일인 <code>build-details.json</code>을 소개합니다. 이 문서는 파일 형식 버전 1.0의 정의와 이 파일의 표준 위치를 지정합니다.</p>
<h3>제안 배경 (Rationale)</h3>
<p>Python 설치를 조사(introspect)할 때, 코드를 실행하는 것은 종종 바람직하지 않거나 불가능할 수 있습니다. 정적 설명 파일을 사용하면 인터프리터를 실행할 필요 없이 Python 설치의 다양한 빌드 세부 정보를 확인할 수 있습니다. 이는 교차 컴파일(cross-compilation), Python 런처(launchers) 등과 같은 사용 사례에 유용합니다.</p>
<h3>범위 (Scope)</h3>
<p><code>build-details.json</code>은 설치 전체에 걸쳐 적용되는 파일입니다. 즉, Python 설치의 모든 환경에서 일정하게 유지되는 정보만 포함해야 합니다. <code>site-packages</code> 경로와 같이 Python 환경에 특정한 정보는 이 파일의 범위에 포함되지 않습니다. PEP 작성자들은 향후 PEP를 통해 정적 환경 설명 파일이 도입될 것으로 예상하고 있습니다.</p>
<h3>명세 (Specification)</h3>
<p>Python 3.14부터 이 PEP 또는 향후 버전에서 지정된 형식을 따르는 <code>build-details.json</code>이라는 파일이 플랫폼 독립적인 표준 라이브러리 디렉터리(예: <code>/usr/lib/python3.14/build-details.json</code>)에 설치되어야 합니다. 단, 기술적인 제약으로 인해 불가능한 경우는 예외입니다.</p>
<p><strong>주의사항 (Attention)</strong>
이 PEP에서 지정한 표준 위치 외에도, <code>build-details.json</code> 파일은 추가적인 위치에 다른 이름으로 설치될 수 있습니다. 그럼에도 불구하고, 파일은 표준 위치에서 계속 사용할 수 있어야 합니다. 실제로 PEP 작성자들은 향후 PEP에서 더 나은 검색 가능성을 가진 추가 설치 위치를 정의할 것으로 예상합니다.</p>
<h3>형식 (Format)</h3>
<p>형식 명세는 아래 제공된 JSON Schema 정의에 의해 정의되며, 여기서는 사람이 읽을 수 있는 형식으로 렌더링됩니다.</p>

<h5><code>libpython.dynamic_stableabi</code></h5>
<ul>
<li><code>Type</code>: <code>string</code></li>
<li><code>Description</code>: 안정적인 ABI(Stable ABI)를 위한 동적 <code>libpython</code> 라이브러리 경로. 절대 경로이거나 <code>base_prefix</code>에 대한 상대 경로입니다. Python 설치가 Stable ABI를 대상으로 하는 동적 <code>libpython</code> 라이브러리를 제공하는 경우 이 필드가 있어야 하며, 그렇지 않으면 이 항목은 누락됩니다. 이 키가 존재하는 경우 <code>dynamic</code>도 설정되어야 합니다.</li>
<li><code>Examples</code>: <code>/usr/lib/libpython3.so</code>, <code>lib/libpython3.so</code> 등.</li>
<li><code>Required</code>: <code>False</code></li>
</ul>
<h5><code>libpython.static</code></h5>
<ul>
<li><code>Type</code>: <code>string</code></li>
<li><code>Description</code>: 정적 <code>libpython</code> 라이브러리 경로. 절대 경로이거나 <code>base_prefix</code>에 대한 상대 경로입니다. Python 설치가 정적 <code>libpython</code> 라이브러리를 제공하는 경우 이 필드가 있어야 하며, 그렇지 않으면 이 항목은 누락됩니다.</li>

<li><code>Required</code>: <code>False</code></li>
</ul>
<h5><code>libpython.link_extensions</code></h5>
<ul>
<li><code>Type</code>: <code>boolean</code></li>
<li><code>Description</code>: 동적 <code>libpython</code>에 대해 빌드된 확장자가 이에 연결되어야 하는지 여부. Python 설치가 동적 <code>libpython</code> 라이브러리를 제공하는 경우 이 필드가 있어야 하며, 그렇지 않으면 이 항목은 누락됩니다.</li>
<li><code>Required</code>: <code>False</code></li>

<h3>기각된 아이디어 (Rejected Ideas)</h3>
<h4>환경별 데이터 포함 (Including environment-specific data)</h4>
<p>이 PEP 논의에서 주요 요청 중 하나는 <code>site-packages</code> 경로와 같은 다른 종류의 정보 포함이었습니다. PEP 작성자들은 Python 환경에 관한 정보는 별도의 파일로 제공되어야 한다고 판단했습니다.</p>
<p>환경별 데이터를 구성 파일에 포함한다는 것은 해당 파일이 환경에 종속적이라는 의미이며, 가상 환경(virtual environments)은 자체 구성 파일이 필요하게 됩니다. 이는 가상 환경이 기본 Python 설치 업데이트 후에도 유지되므로, 정적 구성 파일이 오래되어 데이터가 신뢰할 수 없게 될 가능성을 만들고, 그 목적을 저해한다는 문제가 있습니다.</p>
<p>제안된 해결책은 이 PEP에서 부분적으로 구현되었으며, 기본 Python 설치와 관련된 <code>build-details.json</code> 파일과 특정 환경과 관련된 <code>environment.json</code> 파일을 갖는 것입니다.</p>
<p><code>build-details.json</code>이 Python 배포의 일부가 됨에 따라, 기본 Python 설치가 업데이트될 때 <code>build-details.json</code>도 함께 업데이트되어 정적 설명 파일이 절대 오래되지 않도록 보장합니다.</p>
