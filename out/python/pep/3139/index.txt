<p><strong>원문 링크:</strong> <a href="https://peps.python.org/pep-3139/">PEP 3139 - Cleaning out sys and the “interpreter” module</a></p>
<p><strong>상태:</strong> Rejected | <strong>유형:</strong> Standards Track | <strong>작성일:</strong> 04-Apr-2008</p>
</blockquote>
<h1>PEP 3139 – <code>sys</code> 모듈 및 "interpreter" 모듈 정리</h1>
<ul>
<li><strong>작성자:</strong> Benjamin Peterson</li>
<li><strong>상태:</strong> Rejected (거부됨)</li>
<li><strong>유형:</strong> Standards Track</li>
<li><strong>작성일:</strong> 2008년 4월 4일</li>
<li><strong>Python 버전:</strong> 3.0</li>
</ul>
<h2>거부 통지 (Rejection Notice)</h2>
<p>Guido van Rossum의 '-0.5' 투표로 이 PEP는 거부되었습니다. 자세한 내용은 python-3000 메일링 리스트 아카이브를 참조하십시오.</p>
<h2>개요 (Abstract)</h2>
<p>이 PEP는 CPython 구현에 특화된 인터프리터(interpreter) 함수들을 위한 새로운 저수준 모듈을 제안합니다. 이는 <code>sys</code> 모듈을 정리하고, 일반적인 Python 기능과 구현 세부 사항(implementation details)을 분리하는 것을 목표로 합니다.</p>
<h2>배경 (Rationale)</h2>
<p>현재 <code>sys</code> 모듈은 크게 두 가지 유형의 함수와 데이터를 포함하고 있습니다:</p>
<ol>
<li>
<p><strong>모든 Python 구현에서 사용 가능하며, Python 가상 머신(Virtual Machine)의 일반적인 실행과 관련된 데이터 및 함수.</strong></p>
<ul>
<li><code>argv</code>, <code>byteorder</code>, <code>path</code>, <code>path_hooks</code>, <code>meta_path</code>, <code>path_importer_cache</code>, <code>modules</code></li>
<li><code>copyright</code>, <code>hexversion</code>, <code>version</code>, <code>version_info</code></li>
<li><code>displayhook</code>, <code>__displayhook__</code></li>
<li><code>excepthook</code>, <code>__excepthook__</code>, <code>exc_info</code>, <code>exc_clear</code></li>
<li><code>exec_prefix</code>, <code>prefix</code>, <code>executable</code></li>
<li><code>exit</code></li>
<li><code>flags</code>, <code>py3kwarning</code>, <code>dont_write_bytecode</code>, <code>warn_options</code></li>
<li><code>getfilesystemencoding</code></li>
<li><code>get/setprofile</code></li>
<li><code>get/settrace</code>, <code>call_tracing</code></li>
<li><code>getwindowsversion</code></li>
<li><code>maxint</code>, <code>maxunicode</code></li>
<li><code>platform</code></li>
<li><code>ps1</code>, <code>ps2</code></li>
<li><code>stdin</code>, <code>stderr</code>, <code>stdout</code>, <code>__stdin__</code>, <code>__stderr__</code>, <code>__stdout__</code></li>
<li><code>tracebacklimit</code></li>
</ul>
</li>
<li>
<p><strong>CPython 인터프리터에 영향을 미치는 데이터 및 함수.</strong></p>
<ul>
<li><code>get/setrecursionlimit</code></li>
<li><code>get/setcheckinterval</code></li>
<li><code>_getframe</code>, <code>_current_frame</code></li>
<li><code>getrefcount</code></li>
<li><code>get/setdlopenflags</code></li>
<li><code>settscdumps</code></li>
<li><code>api_version</code></li>
<li><code>winver</code>, <code>dllhandle</code></li>
<li><code>float_info</code></li>
<li><code>_compact_freelists</code></li>
<li><code>_clear_type_cache</code></li>
<li><code>subversion</code></li>
<li><code>builtin_module_names</code></li>
<li><code>callstats</code></li>
<li><code>intern</code></li>
</ul>
</li>
</ol>
<p>두 번째 그룹에 속하는 항목들은 수년 동안 꾸준히 증가하여 <code>sys</code> 모듈을 복잡하게 만들었습니다. 심지어 Guido van Rossum도 일부 항목들을 인지하지 못한다고 언급했습니다.</p>
<p>이러한 항목들을 다른 모듈로 이동시키면, 다른 Python 구현체들에게 어떤 함수가 구현되어야 하고 어떤 함수가 그렇지 않은지에 대한 명확한 메시지를 전달할 수 있을 것입니다. 또한, <code>types</code> 모듈의 내용이 표준 라이브러리 전반에 분산되어야 한다는 제안도 있었는데, 이 <code>interpreter</code> 모듈은 프레임(frames) 및 코드 객체(code objects)와 같은 내부 유형(internal types)을 위한 훌륭한 저장소가 될 수 있을 것입니다.</p>
<h2>상세 사양 (Specification)</h2>
<p>새로운 내장(builtin) 모듈인 "interpreter"가 추가될 예정이었습니다 (이름에 대해서는 "Naming" 섹션 참조). 위에 언급된 두 번째 그룹의 항목들은 다음과 같이 표준 라이브러리로 분할될 예정이었습니다:</p>
<ul>
<li>
<p><strong><code>interpreter</code> 모듈로 이동할 항목:</strong></p>
<ul>
<li><code>get/setrecursionlimit</code></li>
<li><code>get/setcheckinterval</code></li>
<li><code>_getframe</code>, <code>_current_frame</code></li>
<li><code>get/setdlopenflags</code></li>
<li><code>settscdumps</code></li>
<li><code>api_version</code></li>
<li><code>winver</code>, <code>dllhandle</code></li>
<li><code>float_info</code></li>
<li><code>_clear_type_cache</code></li>
<li><code>subversion</code></li>
<li><code>builtin_module_names</code></li>
<li><code>callstats</code></li>
<li><code>intern</code></li>
</ul>
</li>
<li>
<p><strong><code>gc</code> (가비지 컬렉터) 모듈로 이동할 항목:</strong></p>
<ul>
<li><code>getrefcount</code></li>
<li><code>_compact_freelists</code></li>
</ul>
</li>
</ul>
<h2>전환 계획 (Transition Plan)</h2>
<p>Python 3.x에 구현되면 <code>interpreter</code> 모듈은 2.6 버전으로 백포팅(back-ported)될 예정이었습니다. 또한, 대체되는 <code>sys</code> 함수에는 Py3k 경고가 추가될 예정이었습니다.</p>
<h2>미해결 문제 (Open Issues)</h2>
<h3>어떤 항목을 이동해야 하는가?</h3>
<ul>
<li><strong><code>dont_write_bytecode</code>:</strong> 일부는 바이트코드(bytecode) 작성 기능을 구현 세부 사항으로 보고 이동해야 한다고 주장했습니다. 반대 의견으로는 현재의 모든 완전한 Python 구현이 어떤 형태로든 바이트코드를 작성하므로 이 기능을 비활성화할 수 있는 것이 가치 있다는 것이었습니다. 또한, 만약 이동한다면 <code>imp</code> 모듈에 두기를 바라는 의견도 있었습니다.</li>
</ul>
<h3><code>imp</code> 모듈로 이동하는 항목이 있는가?</h3>
<p><code>dont_write_bytecode</code> 또는 <code>builtin_module_names</code>가 <code>imp</code> 모듈에 잘 어울릴 수 있다는 의견이 있었습니다.</p>
<h3>명명 (Naming)</h3>
<p>작성자는 새로운 모듈의 이름으로 "interpreter"를 제안했습니다. "pyvm"도 제안되었고, "cpython"이라는 이름은 많은 지지를 받았습니다.</p>
