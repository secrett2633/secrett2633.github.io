<p><strong>원문 링크:</strong> <a href="https://peps.python.org/pep-0446/">PEP 446 - Make newly created file descriptors non-inheritable</a></p>
<p><strong>상태:</strong> Final | <strong>유형:</strong> Standards Track | <strong>작성일:</strong> 05-Aug-2013</p>

<h2>PEP 446: 새로 생성되는 파일 디스크립터를 기본적으로 비상속으로 만들기</h2>
<h3>요약 (Abstract)</h3>
<p>자식 프로세스로 파일 디스크립터가 누출(leaking)되는 것은 다양한 문제를 야기하며, 주요 보안 취약점으로 알려져 있습니다. <code>subprocess</code> 모듈에서 <code>close_fds=True</code> 매개변수를 사용하는 것이 모든 경우에 가능한 것은 아닙니다.</p>
<p>이 PEP는 이러한 문제의 위험을 줄이기 위해 Python에 의해 새로 생성되는 모든 파일 디스크립터를 기본적으로 비상속(non-inheritable)으로 만들 것을 제안합니다. 또한, 이 PEP는 비상속 파일 디스크립터를 생성하기 위한 원자적(atomic) 플래그를 지원하는 운영체제에서 다중 스레드 애플리케이션의 경쟁 조건(race condition)을 수정합니다.</p>
<p>이 변경 사항이 코드 호환성에 영향을 미칠 수 있음을 인지하고 있지만, 인류 전체의 이익을 위해 이를 진행합니다.</p>
<h3>제안 배경 (Rationale)</h3>
<h4>파일 디스크립터 상속</h4>
<p>각 운영체제는 파일 디스크립터의 상속을 다르게 처리합니다. Windows는 기본적으로 비상속 핸들을 생성하는 반면, UNIX 및 Windows의 POSIX API는 기본적으로 상속 가능한(inheritable) 파일 디스크립터를 생성합니다. Python은 단일 코드 베이스를 유지하고 파일 디스크립터에 동일한 타입을 사용하기 위해 네이티브 Windows API보다 POSIX API를 선호하므로, 상속 가능한 파일 디스크립터를 생성해왔습니다.</p>
<p><code>os.pipe()</code>는 Windows에서는 비상속 파이프를 생성하고, UNIX에서는 상속 가능한 파이프를 생성하는 예외가 있습니다. 이는 <code>os.pipe()</code>가 Windows에서는 네이티브 API인 <code>CreatePipe()</code>를 호출하고 UNIX에서는 POSIX API인 <code>pipe()</code>를 호출하는 구현상의 특징 때문입니다.</p>
<h4>상속 가능한 파일 디스크립터의 문제점</h4>
<p>대부분의 경우, 자식 프로세스로 "누출"된 상속 가능한 파일 디스크립터는 큰 버그를 일으키지 않기 때문에 눈에 띄지 않을 수 있지만, 이는 문제를 해결하지 않아도 된다는 의미는 아닙니다.</p>
<p>상속된 파일 디스크립터와 관련된 두 가지 일반적인 문제점:</p>
<ul>
<li><strong>Windows에서 디렉터리 제거 문제:</strong> 디렉터리 내에 열린 모든 파일 핸들이 닫히기 전에는 디렉터리를 제거할 수 없습니다. 이는 파일 자체에도 해당될 수 있습니다.</li>
<li><strong>리스닝 소켓 재사용 불가:</strong> 리스닝 소켓이 자식 프로세스로 누출되면, 부모 및 자식 프로세스가 모두 종료되기 전에는 소켓 주소를 재사용할 수 없습니다. 예를 들어, 웹 서버가 프로세스를 처리하기 위해 새 프로그램을 실행하고, 해당 프로그램이 완료되지 않은 상태에서 서버가 재시작되면, TCP 포트가 여전히 사용 중이므로 서버가 시작할 수 없습니다.</li>
</ul>
<h4>보안 취약점</h4>
<p>민감한 파일 핸들 및 파일 디스크립터의 누출은 보안 취약점으로 이어질 수 있습니다. 신뢰할 수 없는 자식 프로세스가 암호와 같은 민감한 데이터를 읽거나, 누출된 파일 디스크립터를 통해 부모 프로세스를 제어할 수 있습니다. 리스닝 소켓이 누출되면 자식 프로세스는 새로운 연결을 수락하여 민감한 데이터를 읽을 수 있습니다.</p>
<h4><code>subprocess</code> 모듈에서 수정된 문제</h4>
<p>상속된 파일 디스크립터는 <code>subprocess</code> 모듈에서 4가지 문제를 일으켰습니다:</p>
<ul>
<li>Issue #2320: <code>subprocess</code>에서 <code>stdin</code> 사용 시 경쟁 조건</li>
<li>Issue #3006: <code>subprocess.Popen</code>이 <code>close()</code> 후에도 소켓이 열린 상태로 유지되도록 함</li>
<li>Issue #7213: <code>subprocess</code>가 <code>Popen</code> 인스턴스 간에 열린 파일 디스크립터를 누출하여 중단 발생</li>
<li>Issue #12786: <code>stdin</code>이 닫혔을 때 <code>subprocess.wait()</code>가 멈춤</li>
</ul>
<p>이러한 문제들은 Python 3.2에서 <code>subprocess</code> 모듈의 4가지 변경 사항을 통해 해결되었습니다.</p>
<h4>비상속 파일 디스크립터의 원자적 생성</h4>
<p>다중 스레드 애플리케이션에서는 새 프로그램이 생성되기 직전에 상속 가능한 파일 디스크립터가 생성될 수 있으며, 이 파일 디스크립터가 비상속으로 설정되기 전에 자식 프로세스로 누출될 수 있습니다. 이러한 경쟁 조건은 파일 디스크립터가 생성될 때부터 비상속으로 직접 생성되면 피할 수 있습니다.</p>
<p>FreeBSD, Linux, Mac OS X, Windows 및 기타 여러 운영체제는 파일 디스크립터 생성 시 상속 플래그를 원자적으로 지워 비상속 파일 디스크립터를 생성하는 것을 지원합니다.</p>
<h3>제안 (Proposal)</h3>
<h4>비상속 파일 디스크립터</h4>
<p>다음 함수들은 새로 생성되는 파일 디스크립터를 기본적으로 비상속으로 만들도록 수정됩니다.</p>
<ul>
<li><code>asyncore.dispatcher.create_socket()</code></li>
<li><code>io.FileIO</code></li>
<li><code>io.open()</code></li>
<li><code>open()</code></li>
<li><code>os.dup()</code></li>
<li><code>os.fdopen()</code></li>
<li><code>os.open()</code></li>
<li><code>os.openpty()</code></li>
<li><code>os.pipe()</code></li>
<li><code>select.devpoll()</code></li>
<li><code>select.epoll()</code></li>
<li><code>select.kqueue()</code></li>
<li><code>socket.socket()</code></li>
<li><code>socket.socket.accept()</code></li>
<li><code>socket.socket.dup()</code></li>
<li><code>socket.socket.fromfd()</code></li>
<li><code>socket.socketpair()</code></li>
</ul>
<p><code>os.dup2()</code>는 여전히 기본적으로 상속 가능하게 생성됩니다. 이는 <code>os.dup2()</code>의 가장 일반적인 사용 사례가 표준 스트림(stdin, stdout, stderr)의 파일 디스크립터를 대체하는 것이고, 표준 스트림은 자식 프로세스에 의해 상속될 것으로 예상되기 때문입니다.</p>
<p>원자적 플래그를 사용할 수 있는 경우, 파일 디스크립터를 비상속으로 만들기 위해 사용됩니다. 그러나 원자적 플래그를 사용할 수 없는 경우 대체(fallback)가 필요하므로 원자성은 항상 보장되지 않습니다.</p>
<h4>새로운 함수 및 메서드</h4>
<p>모든 플랫폼에서 사용 가능한 새로운 함수:</p>
<ul>
<li><code>os.get_inheritable(fd: int)</code>: 파일 디스크립터가 자식 프로세스에 의해 상속될 수 있으면 <code>True</code>, 그렇지 않으면 <code>False</code>를 반환합니다.</li>
<li><code>os.set_inheritable(fd: int, inheritable: bool)</code>: 지정된 파일 디스크립터의 상속 플래그를 설정합니다.</li>
</ul>
<p>Windows에서만 사용 가능한 새로운 함수:</p>
<ul>
<li><code>os.get_handle_inheritable(handle: int)</code>: 핸들이 자식 프로세스에 의해 상속될 수 있으면 <code>True</code>, 그렇지 않으면 <code>False</code>를 반환합니다.</li>
<li><code>os.set_handle_inheritable(handle: int, inheritable: bool)</code>: 지정된 핸들의 상속 플래그를 설정합니다.</li>
</ul>
<p>새로운 메서드:</p>
<ul>
<li><code>socket.socket.get_inheritable()</code>: 소켓이 자식 프로세스에 의해 상속될 수 있으면 <code>True</code>, 그렇지 않으면 <code>False</code>를 반환합니다.</li>
<li><code>socket.socket.set_inheritable(inheritable: bool)</code>: 지정된 소켓의 상속 플래그를 설정합니다.</li>
</ul>
<h4>기타 변경 사항</h4>
<ul>
<li>UNIX에서 <code>subprocess</code>는 <code>pass_fds</code> 매개변수의 파일 디스크립터를 상속 가능하게 만듭니다. 이 파일 디스크립터는 <code>fork()</code> 후 <code>execv()</code> 전에 자식 프로세스에서 상속 가능하게 되므로, 부모 프로세스의 파일 디스크립터 상속 플래그는 변경되지 않습니다.</li>
<li><code>os.dup2()</code>에 새로운 선택적 <code>inheritable</code> 매개변수가 추가되었습니다: <code>os.dup2(fd, fd2, inheritable=True)</code>. <code>fd2</code>는 기본적으로 상속 가능하게 생성되지만, <code>inheritable</code>이 <code>False</code>인 경우 비상속으로 생성됩니다.</li>
</ul>
<h3>하위 호환성 (Backward Compatibility)</h3>
<p>이 PEP는 파일 디스크립터 상속에 의존하는 애플리케이션의 동작을 변경합니다. 개발자들은 파일 디스크립터 상속을 이식성 있게 처리하는 고수준 Python 모듈인 <code>subprocess</code>를 재사용할 것을 권장합니다.</p>
<p><code>subprocess</code> 모듈을 <code>pass_fds</code> 매개변수와 함께 사용하거나, 표준 스트림을 리디렉션하기 위해 <code>os.dup2()</code>만 사용하는 애플리케이션은 영향을 받지 않을 것입니다.</p>
<p>파일 디스크립터가 기본적으로 비상속으로 만들어지므로, Python은 더 이상 POSIX를 따르지 않습니다. 하지만 Python은 POSIX를 준수하도록 설계된 것이 아니라 이식 가능한 애플리케이션을 개발하도록 설계되었습니다.</p>
<h3>관련 작업 (Related Work)</h3>
<p>Go, Perl, Ruby와 같은 프로그래밍 언어는 새로 생성되는 파일 디스크립터를 기본적으로 비상속으로 만듭니다.</p>
<ul>
<li>Go 1.0 (2009년) 이후</li>
<li>Perl 1.0 (1987년) 이후</li>
<li>Ruby 2.0 (2013년) 이후</li>
</ul>
<p>Python으로 작성된 SCons 프로젝트는 Windows에서 파일을 비상속으로 만들기 위해 내장 함수 <code>file()</code> 및 <code>open()</code>을 재정의합니다.</p>
<h3>거부된 대안 (Rejected Alternatives)</h3>
<h4>새로운 <code>open_noinherit()</code> 함수 추가</h4>
<p>2007년 6월, Henning von Bargen은 자식 프로세스의 상속된 파일 디스크립터 문제를 해결하기 위해 새로운 <code>open_noinherit()</code> 함수를 추가할 것을 제안했습니다. 당시 <code>subprocess</code> 모듈의 <code>close_fds</code> 매개변수 기본값은 <code>False</code>였습니다.</p>
