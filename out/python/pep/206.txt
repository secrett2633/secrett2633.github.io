<p><strong>원문 링크:</strong> <a href="https://peps.python.org/pep-0206/" target="_blank" rel="noopener noreferrer">PEP 206 - Python Advanced Library</a></p>
<p><strong>상태:</strong> Withdrawn | <strong>유형:</strong> Informational | <strong>작성일:</strong> 14-Jul-2000</p>

<p>PEP 206 – Python Advanced Library는 Python의 "Batteries Included" 철학을 확장하여 고품질의 자주 사용되는 서드 파티 확장 모듈 컬렉션을 제안했던 문서입니다. 이 PEP는 현재 철회(Withdrawn)되었습니다.</p>
<hr>
<h2 id="pep-206-python-advanced-library-python-고급-라이브러리"><a href="#pep-206-python-advanced-library-python-고급-라이브러리">PEP 206 – Python Advanced Library (Python 고급 라이브러리)</a></h2>
<ul>
<li><strong>작성자:</strong> A.M. Kuchling</li>
<li><strong>상태:</strong> 철회됨 (Withdrawn)</li>
<li><strong>유형:</strong> 정보 제공 (Informational)</li>
<li><strong>생성일:</strong> 2000년 7월 14일</li>
</ul>
<h3 id="1-서론-introduction"><a href="#1-서론-introduction">1. 서론 (Introduction)</a></h3>
<p>이 PEP는 "Python Advanced Library"를 설명합니다. Python Advanced Library는 고품질의 자주 사용되는 서드 파티 확장 모듈들의 컬렉션을 의미합니다.</p>
<h3 id="2-batteries-included-철학-batteries-included-philosophy"><a href="#2-batteries-included-철학-batteries-included-philosophy">2. "Batteries Included" 철학 (Batteries Included Philosophy)</a></h3>
<p>Python 배포판은 오랫동안 "batteries included" (배터리 포함)이라는 철학을 유지해왔습니다. 이는 사용자가 별도의 패키지를 다운로드할 필요 없이 즉시 사용할 수 있는 풍부하고 다재다능한 표준 라이브러리를 제공하는 것을 의미합니다. 이러한 접근 방식은 많은 프로젝트에서 Python 언어가 빠르게 시작할 수 있도록 돕습니다.</p>
<p>그러나 표준 라이브러리 모듈이 항상 최적의 선택은 아니었습니다. 일부 라이브러리 모듈은 급하게 만들어졌거나(예: <code>calendar</code>, <code>commands</code>), 설계가 미흡하여 수정이 거의 불가능한 경우(예: <code>cgi</code>), 또는 다른 더 완전한 모듈에 의해 구식화된 경우(예: <code>binascii</code>는 <code>binhex</code>, <code>uu</code>, <code>base64</code> 모듈과 동일한 기능을 제공)가 있었습니다.</p>
<p>이 PEP는 Python을 다양한 애플리케이션 도메인에서 더욱 경쟁력 있게 만들 수 있는 서드 파티 모듈 목록을 제시하며, 이를 "Python Advanced Library"로 구성하고자 했습니다.</p>
<h3 id="3-목표-및-구현-deliverables"><a href="#3-목표-및-구현-deliverables">3. 목표 및 구현 (Deliverables)</a></h3>
<p>제안된 결과물은 특정 애플리케이션 도메인에 맞는 패키지를 검색, 빌드 및 설치하는 스크립트 세트였습니다. 당시 Python Package Index (PyPI)가 패키지를 자동으로 찾아 다운로드할 수 있는 충분한 정보를 담고 있었기 때문에, 이러한 구현이 적절하다고 보았습니다.</p>
<p>이 문서는 서드 파티 모듈에 의해 대체된 표준 라이브러리 모듈을 제거할 것을 제안하지 않았습니다. 이는 많은 하위 호환성 문제(backward-compatibility problems)를 수반하기 때문에 당장은 고려할 가치가 없다고 판단했습니다.</p>
<h3 id="4-제안된-애플리케이션-도메인-및-모듈-suggested-application-domains-and-modules"><a href="#4-제안된-애플리케이션-도메인-및-모듈-suggested-application-domains-and-modules">4. 제안된 애플리케이션 도메인 및 모듈 (Suggested Application Domains and Modules)</a></h3>
<p>PEP 206은 다양한 애플리케이션 도메인에 대해 유용할 수 있는 서드 파티 모듈들을 예시로 들었습니다.</p>
<ul>
<li>
<p><strong>웹 작업 (Web tasks):</strong></p>
<ul>
<li>XML 파싱: <code>ElementTree</code> + <code>SAX</code></li>
<li>URL 검색: <code>libcurl</code> 또는 다른 대안</li>
<li>HTML 파싱: <code>mxTidy</code>, <code>HTMLParser</code></li>
<li>비동기 네트워크 I/O: <code>Twisted</code></li>
<li>웹 프레임워크: WSGI 게이트웨이, <code>Paste</code></li>
</ul>
</li>
<li>
<p><strong>과학 프로그래밍 (Scientific Programming):</strong></p>
<ul>
<li>수치 계산: <code>Numeric</code>, <code>SciPy</code></li>
<li>그래픽: <code>PIL</code> (Pillow), <code>Chaco</code></li>
</ul>
</li>
<li>
<p><strong>애플리케이션 개발 (Application Development):</strong></p>
<ul>
<li>그래픽: PDF 생성을 위한 <code>Reportlab</code></li>
</ul>
</li>
<li>
<p><strong>교육 (Education):</strong></p>
<ul>
<li>그래픽: <code>PyGame</code></li>
</ul>
</li>
</ul>
<h3 id="5-gnu-general-public-license-적용-소프트웨어-software-covered-by-the-gnu-general-public-license"><a href="#5-gnu-general-public-license-적용-소프트웨어-software-covered-by-the-gnu-general-public-license">5. GNU General Public License 적용 소프트웨어 (Software covered by the GNU General Public License)</a></h3>
<p>일부 서드 파티 모듈은 GNU General Public License (GPL) 및 GNU Lesser General Public License (LGPL)에 따라 제공될 수 있습니다. 이러한 패키지를 다운로드하고 설치하는 스크립트를 제공하거나, 모든 패키지를 단일 타르볼(tarball) 또는 CD-ROM으로 묶는 것은 라이선스의 "단순 집합(mere aggregation)" 조항에 따라 GPL에 어려움을 일으키지 않을 것이라고 언급되었습니다.</p>
<h3 id="6-미해결-문제-open-issues"><a href="#6-미해결-문제-open-issues">6. 미해결 문제 (Open Issues)</a></h3>
<p>이 PEP에서는 몇 가지 미해결 문제를 제기했습니다.</p>
<ul>
<li>어떤 다른 애플리케이션 도메인이 중요한가?</li>
<li>이 제안이 Ubuntu 또는 Debian 패키지 세트에 불과해야 하는가? <code>PyGame</code>과 같은 것을 컴파일하는 것은 매우 복잡할 수 있으며 자동화하기 어려울 수 있다.</li>
</ul>
<h3 id="7-결론-및-영향"><a href="#7-결론-및-영향">7. 결론 및 영향</a></h3>
<p>PEP 206은 Python의 표준 라이브러리가 충분하지 않거나 구식인 특정 영역에서 서드 파티 모듈을 공식적으로 추천하고 통합하려는 초기 시도였습니다. 이 PEP는 "Withdrawn" (철회됨) 상태로, 제안된 내용이 공식적으로 채택되지는 않았습니다.</p>
<p>하지만 이 PEP의 논의는 Python 생태계가 점차 발전하면서 PyPI와 같은 패키지 관리 시스템의 중요성을 강조하고, 서드 파티 라이브러리가 Python 개발의 핵심 요소로 자리 잡는 데 기여했습니다. 오늘날 <code>pip</code>와 PyPI는 Python 개발자들이 필요한 라이브러리를 쉽게 찾고 설치할 수 있도록 하는 사실상의 표준이 되었으며, 이는 PEP 206이 해결하고자 했던 문제의 현대적인 해답이라고 볼 수 있습니다. 현재는 특정 "Advanced Library"를 공식적으로 묶기보다는, PyPI를 통해 방대한 서드 파티 라이브러리 생태계가 자율적으로 성장하고 있습니다.</p>
