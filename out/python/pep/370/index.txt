<p><strong>상태:</strong> Final | <strong>유형:</strong> Standards Track | <strong>작성일:</strong> 11-Jan-2008</p>

<h1>PEP 370 – 사용자별 site-packages 디렉터리</h1>
<p><strong>작성자:</strong> Christian Heimes
<strong>상태:</strong> Final
<strong>유형:</strong> Standards Track
<strong>생성일:</strong> 2008년 1월 11일
<strong>Python 버전:</strong> 2.6, 3.0</p>
<h2>개요 (Abstract)</h2>
<p>이 PEP는 사용자들이 자신의 홈 디렉터리에 Python 패키지를 로컬로 설치할 수 있도록, <strong>사용자별 <code>site-packages</code> 디렉터리</strong> 를 새로 제안합니다.</p>
<h2>도입 배경 (Rationale)</h2>
<p>현재 Python 버전에는 사용자 홈 디렉터리에 패키지를 설치하는 통일된 방법이 없습니다 (Mac Framework 빌드 제외). 사용자들은 시스템 관리자에게 패키지 설치 또는 업데이트를 요청하거나, Virtual Python, Working Env, Virtual Env와 같은 다양한 임시 해결책(workaround) 중 하나를 사용해야 했습니다.</p>
<p>이 PEP의 목표는 기존 도구들을 대체하거나 Python의 격리된 설치를 구현하는 것이 아닙니다. 단지 각 사용자를 위한 추가적인 <code>site-packages</code> 디렉터리라는 가장 일반적인 사용 사례를 구현하는 것입니다.</p>
<p>이 기능은 <code>PYTHONPATH</code> 환경 변수를 사용하여 구현할 수 없습니다. 환경 변수는 단순히 <code>sys.path</code>의 시작 부분에 새 디렉터리를 추가하지만, 해당 디렉터리 내의 <code>.pth</code> 파일을 파싱(parse)하지 않습니다. 여러 애플리케이션과 Python <code>egg</code> 파일에는 완전히 기능을 갖춘 <code>site-packages</code> 경로가 필요합니다.</p>
<h2>사양 (Specification)</h2>
<h3><code>site</code> 디렉터리 (<code>site-packages</code>)</h3>
<p><code>sys.path</code>에 있는 디렉터리입니다. 일반 디렉터리와 달리, 이 디렉터리 내의 <code>.pth</code> 파일도 처리됩니다.</p>
<h3>사용자 <code>site</code> 디렉터리 (user site directory)</h3>
<p>사용자의 홈 디렉터리 내에 있는 <code>site</code> 디렉터리입니다. 사용자 <code>site</code> 디렉터리는 특정 Python 버전에 종속됩니다. 경로는 주 버전(major)과 부 버전(minor)만을 포함하는 버전 번호를 포함합니다.</p>
<ul>
<li><strong>Unix (Mac OS X 포함):</strong> <code>~/.local/lib/python2.6/site-packages</code></li>
<li><strong>Windows:</strong> <code>%APPDATA%/Python/Python26/site-packages</code></li>
</ul>
<h3>사용자 데이터 디렉터리 (user data directory)</h3>
<p>일반적으로 사용자 <code>site</code> 디렉터리의 상위 디렉터리입니다. <code>config</code> 파일, 문서, 이미지 및 번역과 같이 Python 버전에 특화된 데이터를 위한 공간입니다.</p>
<ul>
<li><strong>Unix (Mac OS X 포함):</strong> <code>~/.local/lib/python2.6</code></li>
<li><strong>Windows:</strong> <code>%APPDATA%/Python/Python26</code></li>
</ul>
<h3>사용자 기본 디렉터리 (user base directory)</h3>
<p>사용자의 홈 디렉터리 내에 위치합니다. 사용자 <code>site</code> 디렉터리와 사용자 <code>config</code> 디렉터리가 이 기본 디렉터리 안에 있습니다. 일부 시스템에서는 이 디렉터리가 타사 앱과 공유될 수 있습니다.</p>
<ul>
<li><strong>Unix (Mac OS X 포함):</strong> <code>~/.local</code></li>
<li><strong>Windows:</strong> <code>%APPDATA%/Python</code></li>
</ul>
<h3>사용자 스크립트 디렉터리 (user script directory)</h3>
<p>바이너리 및 스크립트 파일을 위한 디렉터리입니다. 여러 Python 버전에서 공유되며, 스크립트의 대상 디렉터리입니다.</p>
<ul>
<li><strong>Unix (Mac OS X 포함):</strong> <code>~/.local/bin</code></li>
<li><strong>Windows:</strong> <code>%APPDATA%/Python/Scripts</code></li>
</ul>
<h3>Windows 관련 사항</h3>
<p>Windows에서는 Application Data 디렉터리 (일명 <code>APPDATA</code>)가 애플리케이션 데이터를 위한 가장 적절한 장소로 선택되었습니다. Microsoft는 소프트웨어가 <code>USERPROFILE</code>에 쓰지 않도록 권장하며, "내 문서" 역시 애플리케이션 데이터에 적합하지 않습니다. 코드는 Win32 API를 쿼리하는 대신 <code>%APPDATA%</code> 환경 변수를 사용합니다.</p>
<p>애플리케이션 데이터 디렉터리는 로밍 프로필(roaming profile)의 일부입니다. 도메인 로그인이 있는 네트워크에서는 애플리케이션 데이터가 중앙 서버에서 복사되거나 복사될 수 있습니다. 이는 로그인 및 로그오프 속도를 저하시킬 수 있습니다. 사용자들은 <code>PYTHONUSERBASE</code>를 “<code>%HOMEDRIVE%%HOMEPATH%Applicata Data</code>”와 같은 값으로 설정하여 데이터를 서버에 유지할 수 있습니다. 더 자세한 정보는 로컬 관리자에게 문의해야 합니다.</p>
<h3>Unix 관련 사항</h3>
<p>Unix에서는 <code>~/.python</code>보다 <code>~/.local</code>이 선택되었는데, 이는 이 디렉터리가 <code>/usr/local</code>과 유사하게 이미 다른 여러 프로그램에서 사용되고 있기 때문입니다.</p>
<h3>Mac OS X 관련 사항</h3>
<p>Mac OS X에서도 Python은 <code>~/.local</code> 디렉터리를 사용합니다. Python의 Framework 빌드는 <code>~/Library/Python/2.6/site-packages</code>를 추가 검색 경로로 포함합니다.</p>
<h2>구현 (Implementation)</h2>
<p><code>site</code> 모듈에 새로운 메서드 <code>adduserpackage()</code>가 추가되어 적절한 디렉터리를 검색 경로에 추가합니다. Python이 시작될 때 디렉터리가 존재하지 않으면 추가되지 않습니다. 그러나 사용자 <code>site</code> 디렉터리와 사용자 <code>base</code> 디렉터리의 위치는 <code>distutils</code>를 위한 내부 변수에 저장됩니다.</p>
<p>사용자 <code>site</code> 디렉터리는 시스템 <code>site</code> 디렉터리보다 먼저 추가되지만, Python의 검색 경로와 <code>PYTHONPATH</code> 다음입니다. 이 설정은 사용자가 시스템 관리자가 설치한 패키지보다 다른 버전의 패키지를 설치할 수 있도록 허용하지만, 사용자가 실수로 표준 라이브러리(stdlib) 모듈을 덮어쓰는 것을 방지합니다. <code>PYTHONPATH</code>를 사용하면 stdlib 모듈을 여전히 덮어쓸 수 있습니다.</p>
<p>보안상의 이유로, 유효 사용자 ID 또는 그룹 ID가 프로세스 UID/GID와 같지 않은 경우 사용자 <code>site</code> 디렉터리는 <code>sys.path</code>에 추가되지 않습니다. 이는 <code>suid</code> 애플리케이션에 대한 코드 주입에 대한 추가적인 장벽입니다. 그러나 Python <code>suid</code> 스크립트는 항상 <code>-E</code> 및 <code>-s</code> 옵션을 사용해야 하며, 그렇지 않으면 사용자가 자신의 코드를 몰래 삽입할 수 있습니다.</p>
<p>사용자 <code>site</code> 디렉터리는 새로운 옵션 <code>-s</code> 또는 환경 변수 <code>PYTHONNOUSERSITE</code>를 통해 비활성화할 수 있습니다. 이 기능은 <code>site.ENABLE_USER_SITE</code>를 <code>False</code>로 설정하여 전역적으로 비활성화할 수 있습니다. 이는 <code>site.py</code>를 편집하여 설정해야 하며, <code>sitecustomize.py</code> 또는 그 이후에는 변경할 수 없습니다.</p>
<p>사용자 <code>base</code> 디렉터리의 경로는 환경 변수 <code>PYTHONUSERBASE</code>로 덮어쓸 수 있습니다. <code>PYTHONUSERBASE</code>가 설정되지 않았거나 비어 있는 경우 기본 위치가 사용됩니다.</p>
<p><code>distutils.command.install</code> (<code>setup.py install</code>)은 사용자 <code>site</code> 디렉터리에 패키지를 설치하기 위한 새로운 인자 <code>--user</code>를 얻습니다. 필요한 디렉터리는 필요할 때 생성됩니다.</p>
<p><code>distutils.command.build_ext</code> (<code>setup.py build_ext</code>)는 새로운 인자 <code>--user</code>를 얻으며, 이는 사용자 <code>base</code> 디렉터리의 <code>include/</code> 및 <code>lib/</code> 디렉터리를 헤더 파일 및 라이브러리 검색 경로에 추가합니다. 또한 <code>lib/</code> 디렉터리를 <code>rpath</code>에 추가합니다.</p>
<p><code>site</code> 모듈은 사용자 <code>base</code> 또는 사용자 <code>site</code> 디렉터리의 경로를 표준 출력으로 인쇄하기 위한 두 가지 인자 <code>--user-base</code>와 <code>--user-site</code>를 얻습니다. 이 기능은 스크립팅을 위한 것으로, 예를 들어 <code>./configure --prefix $(python2.5 -m site --user-base)</code>와 같이 사용될 수 있습니다.</p>
<p><code>distutils.sysconfig</code>는 <code>site</code>의 private 변수에 접근하는 메서드를 얻게 될 것입니다. (아직 구현되지 않음)</p>
<p>Windows 업데이터도 업데이트해야 합니다. 이는 새 Explorer 창에서 사용자 <code>site</code> 디렉터리를 여는 메뉴 항목을 생성해야 합니다.</p>
