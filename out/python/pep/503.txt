<h2 id="pep-503-simple-repository-api"><a href="#pep-503-simple-repository-api">PEP 503 – Simple Repository API</a></h2>
<ul>
<li><strong>작성자:</strong> Donald Stufft</li>
<li><strong>BDFL-Delegate:</strong> Donald Stufft</li>
<li><strong>토론:</strong> Distutils-SIG list</li>
<li><strong>상태:</strong> Final</li>
<li><strong>유형:</strong> Standards Track</li>
<li><strong>주제:</strong> Packaging</li>
<li><strong>생성일:</strong> 2015년 9월 4일</li>
<li><strong>최종 수정일:</strong> 2025년 8월 20일 (GMT)</li>
</ul>
<h3 id="초록-abstract"><a href="#초록-abstract">초록 (Abstract)</a></h3>
<p>Python 패키지 저장소에는 다양한 구현체가 존재하며, 이를 사용하는 도구 또한 많습니다. 이 중 "simple" 저장소 API의 모습을 정의하는 표준 구현체는 PyPI에 사용되는 구현체입니다. 이 문서는 simple 저장소 API의 올바른 동작을 문서화하고 API를 명확하게 명시합니다.</p>
<h3 id="사양-specification"><a href="#사양-specification">사양 (Specification)</a></h3>
<p>simple API를 구현하는 저장소는 <strong>베이스 URL (base URL)</strong> 에 의해 정의됩니다. 이는 모든 추가 URL의 최상위 URL입니다. PyPI의 베이스 URL이 <code>https://pypi.org/simple/</code>이므로 이 API를 "simple" 저장소라고 부릅니다.</p>
<p><strong>참고:</strong> 이 문서의 모든 후속 URL은 베이스 URL에 대한 상대 경로입니다. 예를 들어, PyPI의 URL을 기준으로 <code>/foo/</code>는 <code>https://pypi.org/simple/foo/</code>가 됩니다.</p>
<p>저장소 내에서, 루트 URL (<code>/</code>, 이 PEP에서는 베이스 URL을 나타냄)은 저장소 내 각 프로젝트에 대한 단일 <code>&#x3C;a></code> (앵커) 요소를 포함하는 유효한 HTML5 페이지여야 합니다.</p>
<ul>
<li>앵커 태그의 텍스트는 <strong>프로젝트의 이름</strong> 이어야 합니다.</li>
<li><code>href</code> 속성은 해당 특정 프로젝트의 URL로 <strong>링크</strong> 되어야 합니다.</li>
</ul>
<p><strong>예시:</strong></p>
<pre><code class="language-html">&#x3C;!DOCTYPE html>
&#x3C;html>
&#x3C;body>
    &#x3C;a href="/frob/">frob&#x3C;/a>
    &#x3C;a href="/spamspamspam/">spamspamspam&#x3C;/a>
&#x3C;/body>
&#x3C;/html>
</code></pre>
<p>루트 URL 아래에는 저장소에 포함된 각 개별 프로젝트에 대한 다른 URL이 있습니다. 이 URL의 형식은 <code>/&#x3C;project>/</code>이며, 여기서 <code>&#x3C;project></code>는 해당 프로젝트의 <strong>정규화된 이름 (normalized name)</strong> 으로 대체됩니다. 따라서 "HolyGrail"이라는 프로젝트는 <code>/holygrail/</code>과 같은 URL을 가집니다.
이 URL은 프로젝트의 파일별로 단일 앵커 요소를 포함하는 유효한 HTML5 페이지로 응답해야 합니다.</p>
<ul>
<li><code>href</code> 속성은 다운로드할 파일의 위치로 연결되는 URL이어야 합니다.</li>
<li>앵커 태그의 텍스트는 URL의 최종 경로 구성 요소(파일 이름)와 <strong>일치</strong> 해야 합니다.</li>
<li>URL은 다음과 같은 구문으로 URL Fragment 형태의 해시를 포함해야 합니다: <code>#&#x3C;hashname>=&#x3C;hashvalue></code>.
<ul>
<li><code>&#x3C;hashname></code>은 해시 함수의 소문자 이름(예: <code>sha256</code>)입니다.</li>
<li><code>&#x3C;hashvalue></code>는 16진수로 인코딩된 다이제스트(digest)입니다.</li>
</ul>
</li>
</ul>
<p><strong>API에 대한 추가 제약 조건:</strong></p>
<ul>
<li>HTML5 페이지로 응답하는 모든 URL은 <code>/</code>로 끝나야 하며, 저장소는 <code>/</code>가 없는 URL을 <code>/</code>를 추가하도록 리다이렉션해야 합니다.</li>
<li>URL은 올바른 위치를 가리키는 한 절대 또는 상대 경로일 수 있습니다.</li>
<li>파일이 저장소에 대해 상대적으로 어디에 호스팅되어야 하는지에 대한 제약은 없습니다.</li>
<li>필수 앵커 요소가 존재하는 한, API 페이지에는 다른 HTML 요소가 있을 수 있습니다.</li>
<li>저장소는 정규화되지 않은 URL을 정규화된 표준 URL로 리다이렉션할 수 있습니다 (예: <code>/Foobar/</code>를 <code>/foobar/</code>로 리다이렉션). 그러나 클라이언트는 이 리다이렉션에 의존해서는 안 되며, <strong>정규화된 URL을 요청</strong> 해야 합니다.</li>
<li>저장소는 Python 표준 라이브러리의 <code>hashlib</code> 모듈을 통해 사용 가능하다고 보장되는 해시 함수 중 하나를 선택해야 합니다 (현재 <code>md5</code>, <code>sha1</code>, <code>sha224</code>, <code>sha256</code>, <code>sha384</code>, <code>sha512</code>). 현재 권장 사항은 <code>sha256</code>을 사용하는 것입니다.</li>
<li>특정 배포 파일에 GPG 서명이 있는 경우, 해당 파일과 같은 이름에 <code>.asc</code>가 추가된 형태로 함께 존재해야 합니다. 예를 들어, <code>/packages/HolyGrail-1.0.tar.gz</code> 파일이 있고 관련 서명이 있다면, 서명은 <code>/packages/HolyGrail-1.0.tar.gz.asc</code>에 위치해야 합니다.</li>
<li>저장소는 파일 링크에 <code>data-gpg-sig</code> 속성을 <code>true</code> 또는 <code>false</code> 값으로 포함하여 GPG 서명 존재 여부를 나타낼 수 있습니다. 이를 사용하는 저장소는 모든 링크에 이 속성을 포함해야 합니다.</li>
<li>저장소는 파일 링크에 <code>data-requires-python</code> 속성을 포함할 수 있습니다. 이는 해당 릴리스에 대한 PEP 345에 명시된 <code>Requires-Python</code> 메타데이터 필드를 노출합니다. 이 속성이 존재할 경우, 설치 도구는 요구 사항을 충족하지 않는 Python 버전으로 설치할 때 다운로드를 무시해야 합니다.</li>
</ul>
<p><strong>예시:</strong></p>
<pre><code class="language-html">&#x3C;a href="..." data-requires-python="&#x26;gt;=3">...&#x3C;/a>
</code></pre>
<p>속성 값에서 <code>&#x3C;</code>와 <code>></code>는 각각 HTML 인코딩된 <code>&#x26;lt;</code>와 <code>&#x26;gt;</code>여야 합니다.</p>
<h3 id="정규화된-이름-normalized-names"><a href="#정규화된-이름-normalized-names">정규화된 이름 (Normalized Names)</a></h3>
<p>이 PEP는 "정규화된" 프로젝트 이름의 개념을 참조합니다. PEP 426에 따르면, 이름에 유효한 문자는 ASCII 알파벳, ASCII 숫자, <code>.</code>, <code>-</code>, <code>_</code> 뿐입니다. 이름은 소문자여야 하며, <code>.</code>, <code>-</code>, <code>_</code> 문자열의 모든 연속은 단일 <code>-</code> 문자로 대체되어야 합니다. 이는 Python에서 <code>re</code> 모듈을 사용하여 구현할 수 있습니다.</p>
<pre><code class="language-python">import re

def normalize(name):
    return re.sub(r"[-_.]+", "-", name).lower()
</code></pre>
