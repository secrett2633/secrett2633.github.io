<h1>PEP 792 – simple index에 프로젝트 상태 마커 추가 (Project status markers in the simple index)</h1>
<h2>초록 (Abstract)</h2>
<p>이 PEP는 simple index에서 제공되는 표준화된 프로젝트 상태 마커(project status markers) 세트와, 이러한 마커를 HTML 및 JSON simple index에서 통신하는 메커니즘을 제안합니다.</p>
<h2>배경 및 동기 (Rationale and Motivation)</h2>
<p>프로젝트의 "상태"는 Python 패키징 생태계의 규모와 복잡성이 증가함에 따라 더욱 중요해지는 중요한 메타데이터입니다. 프로젝트가 유지보수되고 있는지 또는 사용하기에 적합한지 판단하는 데 프로젝트 상태(또는 최근 활동과 같은 대리 지표)를 아는 것이 유용합니다.</p>
<p>현재 Python 패키징에는 프로젝트의 "상태"를 전달하는 최소 세 가지 메커니즘이 있습니다.</p>
<ol>
<li><strong>Trove 분류자 (Trove classifiers):</strong> PEP 301에 명시된 대로 배포 패키지 메타데이터에 포함될 수 있습니다. 유연하지만, 상태를 업데이트할 때마다 새로운 배포를 푸시해야 하며, 이전 배포의 분류자는 업데이트될 수 없다는 한계가 있습니다.</li>
<li><strong>Yanking (회수):</strong> PEP 592에 명시된 대로 배포 및 릴리스를 "yanked"로 표시할 수 있습니다. 이는 종속성 해결에 적합하지 않은 것으로 간주되며, 특정 릴리스에 대한 단일 목적의 상태 표시입니다.</li>
<li><strong>PyPI 프로젝트 상태 (PyPI project statuses):</strong> 전체 프로젝트(모든 릴리스 및 배포)에 적용됩니다. PyPI 관리자는 프로젝트를 "quarantine"할 수 있고, 프로젝트 소유자는 "archive"할 수 있습니다. 그러나 이 상태는 현재 PyPI의 API를 통해 노출되지 않고 사용자 대면 웹페이지에만 렌더링됩니다.</li>
</ol>
<p>현재 프로젝트 상태 지표 중 기계 판독 가능(machine-readable), 일반적(general), 인덱스 독립적(index-agnostic), 그리고 릴리스별 또는 배포별이 아닌 프로젝트 전체에 적용되는 네 가지 특성을 모두 만족하는 것은 없습니다.</p>
<p>| 메커니즘             | 기계 판독 가능 | 일반적 | 인덱스 독립적 | 프로젝트 전체 |
| :------------------- | :------------- | :----- | :------------ | :------------ |
| Trove 분류자         | ✅             | ✅     | ✅            | ❌            |
| Yanking              | ✅             | ❌     | ✅            | ✅            |
| PyPI 프로젝트 상태 | ✅             | ✅     | ❌            | ✅            |</p>
<p>이 PEP는 위 네 가지 조건을 모두 만족하는 인덱스 독립적인 메커니즘으로 PyPI의 프로젝트 상태를 채택할 것을 제안합니다.</p>
<h2>명세 (Specification)</h2>
<p>이 PEP는 프로젝트 상태 마커 세트와 표준 HTML 및 JSON 인덱스에서의 표현이라는 두 가지 측면을 명시합니다.</p>
<h3>프로젝트 상태 마커 (Project status markers)</h3>
<p>프로젝트는 항상 정확히 하나의 상태를 가집니다. 명시적으로 상태가 명시되지 않으면 프로젝트는 <code>active</code> 상태로 간주됩니다. 인덱스는 필요에 따라 이 PEP에 지정된 상태 마커의 하위 집합을 구현할 수 있습니다.</p>
<ul>
<li>
<p><strong><code>active</code></strong>: 프로젝트가 활성화된 상태입니다. 이것이 프로젝트의 기본 상태입니다.</p>
<ul>
<li><strong>인덱스 의미론:</strong> 새 배포 업로드를 허용하고 기존 배포 다운로드를 제공해야 합니다.</li>
<li><strong>인스톨러 의미론:</strong> 없음.</li>
</ul>
</li>
<li>
<p><strong><code>archived</code></strong>: 프로젝트가 향후 업데이트될 예정이 없습니다.</p>
<ul>
<li><strong>인덱스 의미론:</strong> 새 배포 업로드를 허용하지 않지만, 기존 배포 다운로드를 제공해야 합니다.</li>
<li><strong>인스톨러 의미론:</strong> 프로젝트 아카이브에 대한 경고를 생성할 수 있습니다.</li>
</ul>
</li>
<li>
<p><strong><code>quarantined</code></strong>: 프로젝트가 일반적으로 사용하기에 안전하지 않은 것으로 간주됩니다(예: 멀웨어).</p>
<ul>
<li><strong>인덱스 의미론:</strong> 새 배포 업로드를 허용하지 않으며, 프로젝트의 어떤 배포도 다운로드 제공을 해서는 안 됩니다.</li>
<li><strong>인스톨러 의미론:</strong> 프로젝트 격리에 대한 경고를 생성할 수 있습니다.</li>
</ul>
</li>
<li>
<p><strong><code>deprecated</code></strong>: 프로젝트가 더 이상 사용되지 않으며, 다른 프로젝트로 대체되었을 수 있습니다.</p>
<ul>
<li><strong>인덱스 의미론:</strong> <code>active</code>와 동일한 의미론을 가집니다.</li>
<li><strong>인스톨러 의미론:</strong> 프로젝트의 비추천(deprecation)에 대한 경고를 생성할 수 있습니다.</li>
</ul>
</li>
</ul>
<h3>인덱스 API의 상태 마커 (Status markers in the index APIs)</h3>
<p>이 PEP는 인덱스 API의 버전 1.4를 정의합니다. 모든 변경 사항은 루트 인덱스 응답이 아닌 각 프로젝트의 인덱스 응답 내에서 프로젝트 수준에서 발생합니다.</p>
<ul>
<li>
<p><strong>HTML index</strong>:</p>
<ul>
<li><code>pypi:repository-version</code>을 <code>1.4</code>로 정의해야 합니다.</li>
<li>프로젝트의 상태 마커를 <code>content</code>로 갖는 <code>pypi:project-status</code> 메타 태그를 추가해야 합니다. (프로젝트가 <code>active</code>인 경우 생략 가능)</li>
<li>프로젝트 상태의 맥락을 설명하는 자유 형식 텍스트를 <code>content</code>로 갖는 <code>pypi:project-status-reason</code> 메타 태그를 포함할 수 있습니다. (<code>active</code>이거나 이유가 없는 경우 생략 가능)</li>
</ul>
</li>
<li>
<p><strong>JSON index</strong>:</p>
<ul>
<li><code>meta.api-version</code>을 <code>1.4</code>로 정의해야 합니다.</li>
<li>프로젝트의 상태 마커를 값으로 갖는 <code>project-status.state</code> 키를 JSON 응답에 포함해야 합니다. (<code>active</code>인 경우 생략 가능)</li>
<li>프로젝트 상태의 맥락을 설명하는 자유 형식 텍스트를 값으로 갖는 <code>project-status.reason</code> 키를 JSON 응답에 포함할 수 있습니다. (<code>active</code>이거나 이유가 없는 경우 생략 가능)</li>
</ul>
</li>
</ul>
<h2>향후 고려 사항 (Future Considerations)</h2>
<p>이 PEP는 <code>active</code>, <code>archived</code>, <code>quarantined</code>, <code>deprecated</code>의 네 가지 프로젝트 상태 마커만 정의합니다. 향후 PEP 또는 PyPA 표준 프로세스에서 필요에 따라 추가 상태 마커를 정의할 수 있습니다. 또한, 사용자 제어 메타데이터(예: 프로젝트 아카이브에 대한 설명)를 포함하도록 프로젝트 상태 메커니즘을 확장하는 것도 고려될 수 있습니다.</p>
<h2>보안 영향 (Security Implications)</h2>
<p>이 PEP는 프로젝트 상태 마커 추가와 관련된 긍정적 또는 부정적인 보안 영향을 식별하지 않습니다.</p>
<h2>교육 방법 (How to Teach This)</h2>
<p>Python 커뮤니티에 이 PEP를 교육하는 데는 두 가지 측면이 있습니다.</p>
<ol>
<li><strong>일반 패키지 유지보수자:</strong> 프로젝트를 아카이브하거나 비추천하는 등의 프로젝트 상태 마커를 설정하는 능력에 대해 알려야 합니다. PyPI와 협력하여 유지보수자 지향 문서 및 커뮤니케이션(블로그 게시물, PyPI 사용자 문서 업데이트 포함)을 제공할 예정입니다.</li>
<li><strong>인스톨러 및 인덱스 유지보수자:</strong> 새로운 프로젝트 상태 마커와 이를 해석하는 방법에 대해 알려야 합니다. PEP 저자는 PyPI에 참조 구현을 수행하고, pip과 같은 인기 있는 인스톨러 유지보수자와 협력하여 프로젝트 상태를 어느 정도 노출할지 결정하도록 도울 것입니다.</li>
</ol>
<h2>거부된 아이디어 (Rejected Ideas)</h2>
<ul>
<li><strong>"예약된" 키 사용 (Using “reserved” keys):</strong> 표준을 직접 표준화하는 대신 기존 메커니즘 내에서 비표준 방식으로 통신하는 방법이 고려되었지만, 표준화된 의미론 부족 및 HTML simple index에 동등한 메커니즘이 없다는 단점으로 인해 거부되었습니다.</li>
<li><strong>PyPI의 비표준 JSON API에 프로젝트 마커 추가 (Project markers in PyPI's non-standard JSON API):</strong> PyPI의 비표준 JSON API에만 프로젝트 상태 마커를 노출하는 방안도 고려되었으나, 표준 및 비표준 API 간의 차이를 심화시킨다는 단점으로 인해 거부되었습니다.</li>
<li><strong>동시에 여러 프로젝트 상태 마커 사용 (Multiple project status markers at once):</strong> 초기 버전에서는 프로젝트가 <code>archived</code>와 <code>quarantined</code>와 같이 동시에 여러 마커를 가질 수 있도록 지원하는 것을 고려했지만, 의미론 병합 시 충돌 해결 메커니즘 및 마커 간의 배타성 상태 머신을 지정해야 하는 복잡성 때문에 거부되었습니다.</li>
</ul>
