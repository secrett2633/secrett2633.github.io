<h1 id="pep-718-subscriptable-functions-서브스크립트-가능한-함수"><a href="#pep-718-subscriptable-functions-서브스크립트-가능한-함수">PEP 718 – Subscriptable functions (서브스크립트 가능한 함수)</a></h1>
<h2 id="개요-abstract"><a href="#개요-abstract">개요 (Abstract)</a></h2>
<p>이 PEP는 타이핑(typing) 목적으로 함수 객체(function objects)를 서브스크립트 가능하게 만드는 것을 제안합니다. 이를 통해 개발자는 타입 체커(type checker)가 생성하는 타입에 대해 명시적인 제어를 할 수 있게 됩니다. 이는 익명 함수의 파라미터 타입을 추론할 수 있는 양방향 추론(bi-directional inference)이나 특수화(specialisation)와 같은 기존 방식이 불충분한 경우에 특히 유용합니다. 또한, 이 제안은 함수가 일반 클래스와 마찬가지로 서브스크립트(subscript)될 수 있도록 하여 일관성을 제공합니다.</p>
<h2 id="동기-motivation"><a href="#동기-motivation">동기 (Motivation)</a></h2>
<p>현재 파이썬의 타입 시스템에는 제네릭(generic) 함수의 타입 파라미터를 추론하기 어렵거나 불가능한 몇 가지 상황이 있습니다. 이 PEP는 이러한 문제점들을 해결하고자 합니다.</p>
<h3 id="알-수-없는-타입-unknown-types"><a href="#알-수-없는-타입-unknown-types">알 수 없는 타입 (Unknown Types)</a></h3>
<p>특정 상황에서 제네릭 함수의 타입 파라미터를 추론하는 것이 불가능합니다.</p>
<p>예를 들어:</p>
<pre><code class="language-python">def make_list[T](*args: T) -> list[T]: ...
reveal_type(make_list()) # 타입 체커가 T에 대한 의미 있는 타입을 추론할 수 없음
</code></pre>
<p><code>FunctionType</code> 인스턴스를 서브스크립트 가능하게 만들면, 이 생성자에 타입을 명시할 수 있습니다.</p>
<pre><code class="language-python">reveal_type(make_list[int]()) # 타입은 list[int]
</code></pre>
<p>현재로서는 정확한 타입을 제공하기 위해 할당을 사용해야 합니다.</p>
<pre><code class="language-python">x: list[int] = make_list()
reveal_type(x) # 타입은 list[int]
</code></pre>
<p>그러나 이 코드는 간단한 함수 호출을 위해 여러 줄을 차지하여 불필요하게 장황합니다.</p>
<p>유사하게, 다음 예시에서 <code>T</code>는 현재 의미 있게 추론될 수 없으므로, 추가적인 할당 없이는 <code>x</code>가 <code>untyped</code> 상태가 됩니다.</p>
<pre><code class="language-python">def factory[T](func: Callable[[T], Any]) -> Foo[T]: ...
reveal_type(factory(lambda x: "Hello World" * x))
</code></pre>
<p>함수 객체가 서브스크립트 가능하다면, 더 구체적인 타입을 지정할 수 있습니다.</p>
<pre><code class="language-python">reveal_type(factory[int](lambda x: "Hello World" * x)) # 타입은 Foo[int]
</code></pre>
<h3 id="결정-불가능한-추론-undecidable-inference"><a href="#결정-불가능한-추론-undecidable-inference">결정 불가능한 추론 (Undecidable Inference)</a></h3>
<p>일부 경우에는 서브클래스(subclass) 관계 때문에 타입 추론이 불가능할 수 있습니다. 그러나 함수를 특수화(specialise)할 수 있다면, 타입 체커는 의미 있는 타입을 추론할 수 있습니다.</p>
<pre><code class="language-python">def foo[T](x: Sequence[T] | T) -> list[T]: ...
reveal_type(foo[bytes](b"hello"))
</code></pre>
<p>현재 타입 체커는 이러한 경우에 일관적으로 타입을 합성하지 못합니다.</p>
<h3 id="해결-불가능한-타입-파라미터-unsolvable-type-parameters"><a href="#해결-불가능한-타입-파라미터-unsolvable-type-parameters">해결 불가능한 타입 파라미터 (Unsolvable Type Parameters)</a></h3>
<p>현재 특수화되지 않은 리터럴(unspecialised literals)의 경우, 다음과 같은 상황에서 타입을 결정하는 것이 불가능합니다.</p>
<pre><code class="language-python">def foo[T](x: list[T]) -> T: ...
reveal_type(foo([])) # 타입 체커가 T를 추론할 수 없음 (다시 한번)
``````python
reveal_type(foo[int]([])) # 타입은 int
</code></pre>
<p>또한, 특정 타입이 함수에 미리 전달되어야 하는 경우에 이를 지정할 수 있다는 점도 유용합니다.</p>
<pre><code class="language-python">words = ["hello", "world"]
foo[int](words) # Invalid: list[str]은 list[int]와 호환되지 않음
</code></pre>
<p>서브스크립션을 허용함으로써 함수와 메서드는 제네릭 클래스와 일관성을 갖게 됩니다. 제안된 모든 변경사항은 호출 가능한 제네릭 클래스를 사용하여 구현될 수 있지만, 문법적 설탕(syntactic sugar)은 매우 환영받을 것입니다.</p>
<p>이로 인해 함수를 특수화하고 새로운 팩토리(factory)로 사용하는 것이 가능합니다.</p>
<pre><code class="language-python">make_int_list = make_list[int]
reveal_type(make_int_list()) # 타입은 list[int]
</code></pre>
<h3 id="모노모피즘-및-재구현-monomorphisation-and-reification"><a href="#모노모피즘-및-재구현-monomorphisation-and-reification">모노모피즘 및 재구현 (Monomorphisation and Reification)</a></h3>
<p>이 제안은 또한 모노모피즘(monomorphisation)과 재구현된 타입(reified types)의 가능성을 열어줍니다.
이는 여러 번 요청되었던 기능의 문을 열어줄 수 있습니다.
<strong>참고:</strong> 이 기능 자체는 이 PEP에서 제안되는 것은 아니지만, 미래에 구현될 수 있습니다.</p>
<p>이러한 기능의 문법은 다음과 같을 수 있습니다.</p>
<pre><code class="language-python">def foo[T](): return T.__value__
assert foo[int]() is int
</code></pre>
<h2 id="근거-rationale"><a href="#근거-rationale">근거 (Rationale)</a></h2>
<p>이 PEP에서 "함수 객체(Function objects)"는 <code>FunctionType</code>, <code>MethodType</code>, <code>BuiltinFunctionType</code>, <code>BuiltinMethodType</code>, <code>MethodWrapperType</code>을 의미합니다.</p>
<p><code>MethodType</code>의 경우 다음과 같이 작성할 수 있어야 합니다.</p>
<pre><code class="language-python">class Foo:
    def make_list[T](self, *args: T) -> list[T]: ...
Foo().make_list[int]()
</code></pre>
<p>그리고 이는 <code>FunctionType</code>과 유사하게 작동해야 합니다.</p>
<p><code>BuiltinFunctionType</code>의 경우, 내장 제네릭 함수(예: <code>max</code>와 <code>min</code>)가 파이썬에서 정의된 함수처럼 작동해야 합니다. 내장 함수는 가능한 한 파이썬으로 구현된 함수처럼 동작해야 합니다.</p>
<p><code>BuiltinMethodType</code>은 <code>BuiltinFunctionType</code>과 동일한 타입입니다.</p>
<p><code>MethodWrapperType</code> (예: <code>object().__str__</code>의 타입)은 제네릭 매직 메서드(generic magic methods)에 유용합니다.</p>
<h2 id="명세-specification"><a href="#명세-specification">명세 (Specification)</a></h2>
<p>함수 객체는 런타임에 서브스크립션을 허용하기 위해 <code>__getitem__</code>을 구현해야 합니다. 이는 <code>__origin__</code>이 호출 가능한 객체(callable)로 설정되고 <code>__args__</code>가 전달된 타입으로 설정된 <code>types.GenericAlias</code> 인스턴스를 반환해야 합니다.</p>
<p>타입 체커는 함수 서브스크립션을 지원해야 하며, 함수 서브스크립션에 전달되는 파라미터가 제네릭 호출 가능 클래스(generic callable class)와 동일한 규칙을 따라야 한다는 것을 이해해야 합니다.</p>
<h3 id="__orig_class__-설정"><a href="#__orig_class__-설정"><code>__orig_class__</code> 설정</a></h3>
<p>현재 <code>__orig_class__</code>는 <code>GenericAlias.__call__</code>에서 호출된 클래스를 생성한 <code>GenericAlias</code>의 인스턴스로 설정되는 속성입니다. 예를 들어:</p>
<pre><code class="language-python">class Foo[T]: ...
assert Foo[int]().__orig_class__ == Foo[int]
</code></pre>
<p>현재 <code>__orig_class__</code>는 무조건적으로(unconditionally) 설정됩니다. 그러나 생성된 인스턴스에서 잠재적인 타입 소거(type erasure)를 방지하기 위해, <code>__origin__</code>이 어떤 함수 객체의 인스턴스인 경우에는 이 속성을 설정해서는 안 됩니다.</p>
<p>다음 코드 스니펫은 이 변경 없이는 런타임에 실패할 것입니다. 왜냐하면 <code>__orig_class__</code>가 <code>Foo[int]</code>가 아니라 <code>bar[str]</code>이 될 것이기 때문입니다.</p>
<pre><code class="language-python">def bar[U](): return Foo[int]()
assert bar[str]().__orig_class__ == Foo[int]
</code></pre>
<h3 id="typingoverload와의-상호작용"><a href="#typingoverload와의-상호작용"><code>@typing.overload</code>와의 상호작용</a></h3>
<p>오버로드(overload)된 함수는 런타임 타입에 영향을 미치지 않으므로 기존과 거의 동일하게 작동해야 합니다. 유일한 변경 사항은 더 많은 상황에서 타입이 결정 가능해지고, 오버로드의 순서나 유니온(unions)에 맡기는 대신 개발자가 동작/오버로드를 명시적으로 지정할 수 있게 된다는 것입니다.</p>
<h2 id="하위-호환성-backwards-compatibility"><a href="#하위-호환성-backwards-compatibility">하위 호환성 (Backwards Compatibility)</a></h2>
<p>현재 이러한 클래스들은 서브클래싱(subclassing)이 불가능하므로, 이미 <code>__getitem__</code>을 구현하고 있는 클래스와 관련하여 하위 호환성 문제는 없습니다.</p>
