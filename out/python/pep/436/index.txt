<p><strong>원문 링크:</strong> <a href="https://peps.python.org/pep-0436/">PEP 436 - The Argument Clinic DSL</a></p>
<p><strong>상태:</strong> Final | <strong>유형:</strong> Standards Track | <strong>작성일:</strong> 22-Feb-2013</p>
</blockquote>
<p>다음은 PEP 436 – The Argument Clinic DSL 문서의 번역 및 요약입니다. 이 문서는 CPython 구현에서 내장 함수(built-in function)의 인자 처리를 용이하게 하기 위한 DSL(Domain Specific Language)인 "Argument Clinic"을 제안합니다.</p>
<hr>
<h1>PEP 436 – The Argument Clinic DSL</h1>
<h2>개요 (Abstract)</h2>
<p>이 문서는 CPython 구현 내에서 내장 함수(built-in functions)의 인자 처리를 용이하게 하는 DSL인 "Argument Clinic"을 제안합니다.</p>
<h2>배경 및 목표 (Rationale and Goals)</h2>
<p>Python의 주요 구현체인 CPython은 Python과 C 언어의 혼합으로 작성됩니다. CPython의 구현 세부 사항 중 하나는 Python 프로그램에서 사용할 수 있지만 C로 작성된 "내장(built-in)" 함수입니다. Python 프로그램이 내장 함수를 호출하고 인자를 전달할 때, 이 인자들은 Python 값에서 C 값으로 변환되어야 합니다. 이 과정을 "인자 파싱(parsing arguments)"이라고 합니다.</p>
<p>CPython 3.3을 기준으로, 내장 함수는 거의 항상 <code>PyArg_ParseTuple()</code> 또는 <code>PyArg_ParseTupleAndKeywords()</code> 두 함수 중 하나를 사용하여 인자를 파싱합니다. 전자는 위치 인자(positional parameters)만 처리하고, 후자는 키워드 인자(keyword and keyword-only parameters)도 수용하며 새로운 코드에 권장됩니다.</p>
<p>두 함수 모두 호출자는 "포맷 문자열(format string)"을 사용하여 인자 파싱을 위한 변환을 지정합니다. 각 인자는 "포맷 유닛(format unit)"에 해당하며, 이는 파싱 함수에게 허용할 Python 타입과 해당 인자에 대한 적절한 C 값으로 변환하는 방법을 알려주는 짧은 문자 시퀀스입니다.</p>
<p><code>PyArg_ParseTuple()</code>은 처음 고안되었을 때는 합리적이었습니다. 그러나 시간이 지나면서 <code>PyArg_Parse</code> 인터페이스는 여러 면에서 확장되었고, 현재의 API는 사용하기 불편할 정도로 복잡해졌습니다.</p>
<p><strong>기존 <code>PyArg_Parse</code> API의 문제점:</strong></p>
<ul>
<li><strong>복잡한 포맷 유닛:</strong> 현재 40가지의 다른 포맷 유닛이 있으며, 일부는 3글자 길이입니다. 이는 문서를 계속 참조하지 않고는 포맷 문자열을 이해하거나 파싱하기 어렵게 만듭니다.</li>
<li><strong>메타 포맷 유닛:</strong> 포맷 문자열 내에 숨겨질 수 있는 6개의 메타 포맷 유닛("()|$:;")이 있습니다.</li>
<li><strong>모호한 포맷 유닛:</strong> 포맷 유닛이 많아질수록 구현자가 사용하기 쉬운 니모닉을 선택하기 어렵고, 일부 포맷 유닛은 미묘한 차이만 있어 의미를 이해하기 어렵습니다.</li>
<li><strong>불편한 Docstring:</strong> Docstring은 정적 C 문자열로 지정되어 C 문자열 인용 규칙을 따라야 하므로 읽고 편집하기가 다소 불편합니다.</li>
<li><strong>반복적인 코드 수정:</strong> <code>PyArg_ParseTupleAndKeywords()</code>를 사용하여 함수에 새 인자를 추가할 때 코드의 6가지 다른 부분을 수정해야 합니다.
<ul>
<li>인자를 저장할 변수 선언.</li>
<li><code>PyArg_ParseTupleAndKeywords()</code>의 올바른 위치에 해당 변수의 포인터와 모든 "길이" 또는 "변환기" 인자를 올바른 순서로 전달.</li>
<li><code>PyArg_ParseTupleAndKeywords()</code>에 전달되는 "키워드" 배열의 올바른 위치에 인자 이름 추가.</li>
<li>포맷 문자열의 올바른 위치에 포맷 유닛 추가.</li>
<li>docstring의 프로토타입에 인자 추가.</li>
<li>docstring에 인자 문서화.</li>
</ul>
</li>
<li><strong>시그니처 정보 부족:</strong> 현재 내장 함수가 "시그니처" 정보( <code>inspect.getfullargspec</code> 및 <code>inspect.Signature</code> 참조)를 제공하는 메커니즘이 없습니다.</li>
</ul>
<p><strong>Argument Clinic의 목표:</strong>
Argument Clinic의 목표는 이러한 단점들을 해결하고, 다음을 통해 기존 API를 대체하는 것입니다.</p>
<ul>
<li>각 인자를 한 번만 지정하면 됩니다.</li>
<li>인자에 대한 모든 정보가 한 곳에 함께 유지됩니다.</li>
<li>각 인자에 대해 변환 함수(conversion function)를 지정하면, Argument Clinic이 Python 값을 C 값으로 변환하는 것을 처리합니다.</li>
<li>Argument Clinic은 매개변수화된 변환 함수(parameterized conversion functions)를 통해 인자 처리 동작을 미세 조정할 수 있도록 합니다.</li>
<li>Docstring은 일반 텍스트로 작성됩니다. 함수 docstring은 필수이며, 인자별 docstring은 권장됩니다.</li>
<li>이를 통해 Argument Clinic은 CPython이 내부적으로 필요로 하는 모든 일상적이고 반복적인 코드와 데이터 구조를 생성합니다.</li>
<li>인터페이스를 지정한 후, 다음 단계는 단순히 네이티브 C 타입을 사용하여 구현을 작성하는 것입니다.</li>
<li>인자 파싱의 모든 세부 사항은 자동으로 처리됩니다.</li>
</ul>
<p>Argument Clinic은 <a href="http://nedbatchelder.com/code/cog/">Cog</a>에서 영감을 받아 전처리기(preprocessor)로 구현되었습니다. Clinic을 사용하려면 C 소스 코드에 특별한 텍스트 문자열로 시작하고 끝나는 블록 주석을 추가한 다음, 해당 파일에 Clinic을 실행합니다. Clinic은 블록 주석을 찾아 내용을 처리하고, 그 출력을 주석 바로 뒤에 있는 C 소스 파일에 다시 작성합니다. Clinic의 출력은 소스 코드의 일부가 되어 개정 관리(revision control)에 체크인되고 소스 패키지와 함께 배포됩니다.</p>
<p>Argument Clinic의 향후 목표는 다음과 같습니다.</p>
<ul>
<li>내장 함수에 대한 시그니처 정보 제공.</li>
<li>Python의 대체 구현체가 자동화된 라이브러리 호환성 테스트를 생성할 수 있도록 지원.</li>
<li>생성된 코드의 개선을 통해 인자 파싱 속도 향상.</li>
</ul>
<h2>DSL 구문 요약 (DSL Syntax Summary)</h2>
<p>Argument Clinic DSL은 다음과 같이 C 파일에 내장된 주석으로 지정됩니다. DSL 구문은 Python의 <code>def</code> 문을 모방하여 Python 코어 개발자들에게 익숙함을 제공합니다.</p>

Argument Clinic is a preprocessor, meaning it processes your code before the actual C compiler sees it. It reads your special "Clinic DSL" comments, generates C code from them, and writes the generated code back into your C file.</p>
<h3>Argument Clinic DSL의 일반적인 동작 (General Behavior Of the Argument Clinic DSL)</h3>
<ul>
<li>Docstring을 제외한 모든 줄에서 <code>#</code>은 한 줄 주석 구분자로 사용됩니다.</li>
<li>빈 줄은 항상 무시됩니다.</li>
<li>Python과 마찬가지로, Argument Clinic DSL에서는 선행 공백(leading whitespace)이 중요합니다.
<ul>
<li>"함수(function)" 섹션의 첫 번째 줄은 함수 선언입니다.</li>
<li>함수 선언 아래에 들여쓰기된 줄은 인자를 선언하며, 한 줄에 하나씩입니다.</li>
<li>그 아래에 더 들여쓰기된 줄은 인자별 docstring입니다.</li>
<li>마지막으로, 0번째 열로 다시 들여쓰기된 첫 번째 줄은 인자 선언을 끝내고 함수 docstring을 시작합니다.</li>
</ul>
</li>
<li>인자 docstring은 선택 사항이지만, 함수 docstring은 필수입니다. 인자를 지정하지 않는 함수는 함수 선언 다음에 docstring만 지정할 수 있습니다.</li>
</ul>
<h3>모듈 및 클래스 선언 (Module and Class Declarations)</h3>
<p>C 파일이 모듈 또는 클래스를 구현하는 경우 Clinic에 이를 선언해야 합니다. 구문은 간단합니다.</p>
<pre><code>module module_name
</code></pre>
<p>또는</p>
<pre><code>class module_name.class_name
</code></pre>
<p>모듈 이름 또는 클래스 이름은 항상 최상위 모듈부터의 전체 점선 경로(dotted path)여야 합니다. 중첩된 모듈 및 클래스가 지원됩니다. (이들은 실제 특수 구문이 아니라 <code>Directives</code>로 구현됩니다.)</p>
<h3>함수 선언 (Function Declaration)</h3>
<p>함수 선언의 전체 형식은 다음과 같습니다.</p>
<pre><code>dotted.name [ as legal_c_id ] [ -> return_annotation ]
</code></pre>
<ul>
<li><code>dotted.name</code>은 최상위 패키지부터 시작하는 함수의 전체 이름이어야 합니다 (예: "os.stat" 또는 "curses.window.addch").</li>
<li><code>as legal_c_id</code> 구문은 선택 사항입니다. Argument Clinic은 함수의 이름을 사용하여 생성된 C 함수의 이름을 만듭니다. 경우에 따라 생성된 이름이 C 프로그램의 네임스페이스에 있는 다른 전역 이름과 충돌할 수 있습니다. <code>as legal_c_id</code> 구문을 사용하면 생성된 이름을 원하는 이름으로 재정의할 수 있습니다. <code>legal_c_id</code>는 유효한 C 식별자여야 합니다. 이 키워드가 생략되면 <code>as</code> 키워드도 생략되어야 합니다.</li>
<li><code>return_annotation</code>도 선택 사항입니다. 생략하면 화살표(<code>-></code>)도 생략되어야 합니다. 지정된 경우 반환 주석 값은 <code>ast.literal_eval</code>과 호환되어야 하며, 반환 변환기(return converter)로 해석됩니다.</li>
</ul>
<h3>인자 선언 (Parameter Declaration)</h3>
<p>인자 선언 줄의 전체 형식은 다음과 같습니다.</p>
<pre><code>name: converter [ (parameter=value [, parameter2=value2]) ] [ = default]
</code></pre>
<ul>
<li><code>name</code>은 유효한 C 식별자여야 합니다. 이름과 콜론 사이에 공백이 허용되지만(권장 스타일은 아님), 콜론과 변환기 사이에는 공백이 허용되고 권장됩니다.</li>
<li><code>converter</code>는 Argument Clinic에 등록된 "변환기 함수(converter functions)" 중 하나의 이름입니다. Clinic은 여러 내장 변환기와 함께 제공되며, 새로운 변환기도 동적으로 추가할 수 있습니다. 변환기를 선택하면 입력에서 허용되는 Python 타입을 자동으로 제한하고 출력 변수의 타입을 지정하게 됩니다.</li>
<li>변환기 뒤에 괄호가 오면, 이 괄호는 변환 함수에 대한 인자를 묶습니다. 구문은 Python 함수 호출에 인자를 제공하는 것과 유사합니다. 인자는 항상 "키워드 전용 인자(keyword-only parameters)"처럼 이름이 지정되어야 하며, 인자에 제공되는 값은 Python 리터럴 값과 유사하게 됩니다. 이러한 인자는 항상 선택 사항이므로, 모든 변환 함수를 어떤 인자 없이도 호출할 수 있습니다. 이 경우 괄호를 완전히 생략할 수도 있습니다. 이는 항상 빈 괄호를 지정하는 것과 동일합니다. 이러한 인자에 제공되는 값은 <code>ast.literal_eval</code>과 호환되어야 합니다.</li>
<li><code>default</code>는 Python 리터럴 값입니다. 기본값은 선택 사항이며, 지정하지 않으면 등호도 생략해야 합니다. 기본값이 없는 인자는 암시적으로 필수(required)입니다. 기본값은 생성된 C 코드에서 동적으로 할당되며, Python 값으로 지정되지만 생성된 C 코드에서는 네이티브 C 값으로 변환됩니다. 이 수동 변환 단계로 인해 허용되는 기본값은 많지 않습니다.</li>
</ul>
<h3>레거시 변환기 (Legacy Converters)</h3>
<p>기존 코드를 Argument Clinic으로 변환하는 편의를 위해, Clinic은 <code>PyArg_ParseTuple</code> 포맷 유닛과 일치하는 레거시 변환기 세트를 제공합니다. 이는 포맷 유닛을 포함하는 C 문자열로 지정됩니다. 예를 들어, 인자 "foo"가 Python <code>int</code>를 받아 C <code>int</code>를 내보내도록 지정하려면 다음과 같이 지정할 수 있습니다.</p>
<pre><code class="language-c">foo : "i"
</code></pre>
<p>(C 문자열과 더 유사하게 하기 위해 항상 이중 따옴표를 사용해야 합니다.)</p>
<p>이들은 <code>PyArg_ParseTuple</code> 포맷 유닛과 유사하지만, 구현이 파싱을 위해 <code>PyArg_Parse</code> 함수를 호출한다는 보장은 없습니다.</p>
<p>이 구문은 인자를 지원하지 않습니다. 따라서 입력 인자가 필요한 포맷 유닛("O!", "O&#x26;", "es", "es#", "et", "et#")은 지원하지 않습니다. 이러한 변환 중 하나를 필요로 하는 인자는 레거시 구문을 사용할 수 없습니다. (그러나 기본값은 제공할 수 있습니다.)</p>
<h3>인자 Docstrings (Parameter Docstrings)</h3>
<p>인자 선언 아래에 나타나고 해당 인자 선언보다 더 들여쓰기된 모든 줄은 해당 인자의 docstring입니다. 이러한 모든 줄은 첫 번째 줄이 왼쪽으로 정렬될 때까지 "들여쓰기 제거(dedented)"됩니다.</p>
<h3>인자 줄의 특별 구문 (Special Syntax For Parameter Lines)</h3>
<p>인자 섹션에서는 네 가지 특별한 기호가 사용될 수 있습니다. 각각은 인자 선언과 동일한 수준으로 들여쓰기된 줄에 단독으로 나타나야 합니다.</p>
<ul>
<li><code>*</code>: 모든 후속 인자가 키워드 전용(keyword-only)임을 설정합니다.</li>
<li><code>[</code>: 선택적 인자 "그룹"의 시작을 설정합니다. "그룹"은 다른 "그룹" 안에 중첩될 수 있습니다. (<code>/</code> 아래의 "Functions With Positional-Only Parameters" 참조). 현재 <code>[</code>은 모든 인자가 위치 전용(positional-only)으로 표시된 함수에서만 사용할 수 있습니다.</li>
<li><code>]</code>: 선택적 인자 "그룹"을 끝냅니다.</li>
<li><code>/</code>: 모든 선행 인자가 위치 전용(positional-only)임을 설정합니다. 현재 Argument Clinic은 위치 전용 인자와 비위치 전용 인자를 모두 포함하는 함수를 지원하지 않습니다. 따라서 <code>/</code>가 함수에 지정된 경우, 현재 항상 마지막 인자 뒤에 와야 합니다. 또한 Argument Clinic은 현재 위치 전용 인자에 대한 기본값을 지원하지 않습니다. ( <code>/</code>의 의미는 Guido가 한때 제안했던 Python의 위치 전용 인자 구문을 따릅니다).</li>
</ul>
<h3>함수 Docstring (Function Docstring)</h3>
<p>함수 선언 후 선행 공백이 없는 첫 번째 줄은 함수 docstring의 첫 번째 줄입니다. Clinic 블록의 모든 후속 줄은 docstring의 일부로 간주되며, 선행 공백은 유지됩니다.</p>
<p>함수 docstring 내에 <code>{parameters}</code> 문자열이 단독으로 한 줄에 나타나면, Argument Clinic은 docstring이 있는 모든 인자의 목록을 삽입하며, 각 인자 뒤에 해당 docstring이 따라옵니다. 인자 이름은 한 줄에 단독으로 표시되고, docstring은 다음 줄에 시작하며, docstring의 모든 줄은 두 칸 들여쓰기됩니다. (인자별 docstring이 없는 인자는 생략됩니다.) 전체 목록은 <code>{parameters}</code> 토큰 앞에 나타난 선행 공백에 의해 들여쓰기됩니다.</p>
<p>docstring에 <code>{parameters}</code> 문자열이 나타나지 않으면 Argument Clinic은 docstring이 빈 줄로 끝나지 않는 경우 빈 줄을 삽입하고, 인자 목록을 0번째 열에 삽입하여 docstring 끝에 이를 추가합니다.</p>
<h3>변환기 (Converters)</h3>
<p>Argument Clinic에는 미리 초기화된 변환기 함수 레지스트리가 포함되어 있습니다.</p>
<p><strong>예시 변환기 함수:</strong></p>
<ul>
<li><code>int</code>: <code>__int__</code>를 구현하는 Python 객체를 허용하며 C <code>int</code>를 내보냅니다.</li>
<li><code>byte</code>: Python <code>int</code>를 허용하며 <code>unsigned char</code>를 내보냅니다. 정수는 <code>[0, 256)</code> 범위에 있어야 합니다.</li>
<li><code>str</code>: Python <code>str</code> 객체를 허용하며 C <code>char *</code>를 내보냅니다. <code>ascii</code> 코덱을 사용하여 문자열을 자동으로 인코딩합니다.</li>
<li><code>PyObject</code>: 모든 객체를 허용하며 변환 없이 C <code>PyObject *</code>를 내보냅니다.</li>
</ul>
<p><strong>모든 변환기가 허용하는 매개변수:</strong></p>
<ul>
<li><code>doc_default</code>: Python 컨텍스트에서 인자의 실제 기본값 대신 사용할 Python 값.</li>
<li><code>required</code>: 일반적으로 기본값이 있는 모든 인자는 자동으로 선택 사항(optional)입니다. <code>required</code>가 설정된 인자는 기본값이 있더라도 필수(non-optional)로 간주됩니다. 생성된 문서에도 기본값이 표시되지 않습니다.</li>
</ul>
<p><strong>개별적으로 허용할 수 있는 선택적 매개변수:</strong></p>
<ul>
<li><code>annotation</code>: 이 인자에 대한 인자별 annotation을 명시적으로 지정합니다.</li>
<li><code>bitwise</code>: 부호 없는 정수를 허용하는 변환기용.</li>
<li><code>encoding</code>: <code>str</code>을 허용하는 변환기용. 유니코드 문자열을 <code>char *</code>로 인코딩할 때 사용할 인코딩.</li>
<li><code>immutable</code>: 변경 불가능한(immutable) 값만 허용합니다.</li>
<li><code>length</code>: 이터러블 타입(iterable types)을 허용하는 변환기용.</li>
<li><code>nullable</code>: 이 변환기는 일반적으로 <code>None</code>을 허용하지 않지만, 이 경우에는 허용해야 합니다.</li>
<li><code>types</code>: 이 객체에 허용되는 Python 타입을 나타내는 문자열 목록. "buffer", "mapping", "number", "sequence"와 같은 Python 프로토콜을 나타내는 네 가지 문자열도 있습니다.</li>
<li><code>zeroes</code>: 문자열 타입을 허용하는 변환기용. 변환된 값은 내장된 널 문자(embedded zeroes)를 가질 수 있어야 합니다.</li>
</ul>
<h3>반환 변환기 (Return Converters)</h3>
<p>반환 변환기는 개념적으로 변환기의 역 작업을 수행합니다. 즉, 네이티브 C 값을 해당 Python 값으로 변환합니다.</p>
<h3>지시어 (Directives)</h3>
<p>Argument Clinic은 Clinic 코드 블록 내에서 "지시어(directives)"도 허용합니다. 지시어는 C의 프라그마(pragma)와 유사하며, Argument Clinic의 동작을 수정하는 문장입니다.</p>
<p>지시어의 형식은 다음과 같습니다.</p>
<pre><code>directive_name [argument [second_argument [ ... ]]]
</code></pre>
<p>지시어는 위치 인자만 받습니다.</p>
<p>Clinic 코드 블록에는 하나 이상의 지시어 또는 함수 선언이 포함되어야 합니다. 둘 다 포함할 수 있으며, 이 경우 모든 지시어는 함수 선언 앞에 와야 합니다.</p>
<h3>Python 코드 (Python Code)</h3>
<p>Argument Clinic은 C 파일 내에 Python 코드를 포함하는 것도 허용하며, Argument Clinic이 파일을 처리할 때 제자리에서 실행됩니다. 임베디드 코드는 다음과 같습니다.</p>
<pre><code class="language-c">/*[python]
# this is python code!
print("/" + "* Hello world! *" + "/")
[python]*/
</code></pre>
<p>Argument Clinic은 <code>builtins.print</code>를 사용하여 출력된 모든 것을 출력합니다. "clinic" 섹션의 경우 출력은 유효한 C 코드입니다.</p>
<h3>출력 (Output)</h3>
<p>Argument Clinic은 Clinic 코드 섹션 바로 뒤에 C 파일 내에 인라인으로 출력을 작성합니다. "python" 섹션의 경우 출력은 <code>builtins.print</code>를 사용하여 인쇄된 모든 것입니다. "clinic" 섹션의 경우 출력은 유효한 C 코드로, 다음을 포함합니다.</p>
<ul>
<li>함수에 대한 올바른 <code>methoddef</code> 구조를 제공하는 <code>#define</code>.</li>
<li>"impl" 함수에 대한 프로토타입 – 이 함수를 구현하기 위해 작성해야 하는 부분입니다.</li>
<li>모든 인자 처리를 처리하고 "impl" 함수를 호출하는 함수.</li>
<li>"impl" 함수의 정의 줄 및 출력 끝을 나타내는 주석.</li>
</ul>
<p>개발자는 출력 바로 뒤에 <code>impl</code> 함수의 본문을 작성합니다 (즉, 출력 끝 주석 바로 뒤에 왼쪽 중괄호를 작성하고 그 안에 내장 함수를 구현합니다).</p>
<p>Argument Clinic은 <code>impl</code> 함수의 인자들을 자동으로 정의합니다. 이 함수는 원래 전달된 <code>self</code> 인자, 정의한 모든 인자, 그리고 추가로 생성된 인자("length" 인자; "group" 인자)를 받습니다.</p>
<p>Argument Clinic은 출력 섹션에 대한 체크섬(checksum)도 작성합니다. 이는 중요한 안전 기능입니다. 만약 개발자가 출력을 수동으로 수정하면, Clinic은 체크섬이 일치하지 않는 것을 감지하고 파일을 덮어쓰기를 거부합니다.</p>
<h3>위치 전용 인자를 사용하는 함수 (Functions With Positional-Only Parameters)</h3>
<p>C로 구현된 상당수의 Python 내장 함수는 인자 처리를 위해 오래된 위치 전용 API(<code>PyArg_ParseTuple()</code>)를 사용합니다. 경우에 따라 이러한 내장 함수는 전달된 인자의 수에 따라 인자를 다르게 파싱합니다. 이는 혼란스러운 유연성을 제공할 수 있습니다. 즉, 선택적 인자 그룹이 있을 수 있으며, 이들은 모두 지정되거나 전혀 지정되지 않아야 합니다. 그리고 때로는 이러한 그룹이 왼쪽에 올 수도 있습니다! (대표적인 예: <code>curses.window.addch()</code>).</p>
<p>Argument Clinic은 이러한 레거시 사용 사례를 지원하기 위해 인자를 그룹으로 지정할 수 있도록 합니다. 각 선택적 인자 그룹은 대괄호(<code>[]</code>)로 표시됩니다. 이러한 그룹은 필수 인자의 오른쪽이나 왼쪽 모두에 올 수 있습니다.</p>
<p>Clinic이 생성하는 <code>impl</code> 함수는 그룹당 추가 인자인 <code>int group_{left|right}_&#x3C;x></code>를 추가합니다. 여기서 <code>x</code>는 필수 인자로부터 멀어지면서 각 그룹에 할당되는 단조 증가하는 숫자입니다. 이 인자는 해당 호출에서 그룹이 지정되면 0이 아닌 값이고, 지정되지 않으면 0입니다.</p>
<p>이 모드에서는 기본 인자를 지정할 수 없습니다.</p>
<p>또한, 함수에 여러 그룹을 지정하여 인자 수에서 유효한 그룹 세트로 여러 유효한 매핑이 존재할 수 있습니다. 이 경우 Clinic은 오류 메시지와 함께 중단됩니다. 위치 전용 작업은 레거시 사용 사례에만 해당하며, 이러한 특이한 동작을 사용하는 모든 레거시 함수는 모호하지 않은 매핑을 가지므로 이는 문제가 되지 않을 것입니다.</p>
