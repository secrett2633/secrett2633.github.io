<p><strong>원문 링크:</strong> <a href="https://peps.python.org/pep-0319/">PEP 319 - Python Synchronize/Asynchronize Block</a></p>
<p><strong>상태:</strong> Rejected | <strong>유형:</strong> Standards Track | <strong>작성일:</strong> 24-Feb-2003</p>
</blockquote>
<p>PEP 319 – Python Synchronize/Asynchronize 블록</p>
<p><strong>상태:</strong> 거부됨 (Rejected)
<strong>작성자:</strong> Michel Pelletier
<strong>생성일:</strong> 2003년 2월 24일
<strong>Python 버전:</strong> 2.4
<strong>참고:</strong> 이 PEP는 PEP 343에 찬성하여 거부되었습니다.</p>
<hr>
<h2>초록 (Abstract)</h2>
<p>이 PEP는 Python에 'synchronize'와 'asynchronize'라는 두 가지 새로운 키워드를 추가할 것을 제안합니다. 이 키워드들은 멀티스레드 환경에서 공유 데이터에 접근할 때 발생하는 동시성 문제를 보다 간결하고 안전하게 처리하기 위한 목적으로 제시되었습니다.</p>
<h2>선언 (Pronouncement)</h2>
<p>이 PEP는 PEP 343에 찬성하여 거부되었습니다. PEP 343은 'with' 문을 도입하여 자원 관리 및 동기화와 같은 패턴을 보다 일반적이고 유연하게 처리할 수 있도록 했습니다.</p>
<h2>'synchronize' 키워드</h2>
<p>Python에서 코드 동기화의 개념은 저수준(low-level)에 속합니다. 현재 프로그래머는 다음과 같은 의사 코드 패턴의 세부 사항을 인지하고 있어야 동기화 코드를 작성할 수 있습니다.</p>
<p><strong>현재의 동기화 패턴:</strong></p>
<pre><code class="language-python">initialize_lock()
...
acquire_lock()
try:
    change_shared_data()
finally:
    release_lock()
</code></pre>
<p>이 <code>try/finally</code> 블록 내에서 락(lock)을 획득하고 해제하는 코드는 오류가 발생하기 쉬워 악명 높은 동시성 스레드 락킹 문제를 야기할 수 있습니다.</p>
<p>PEP 319는 위 코드를 다음과 같이 대체할 것을 제안합니다.</p>
<p><strong>제안된 'synchronize' 패턴:</strong></p>
<pre><code class="language-python">synchronize:
    change_shared_data()
</code></pre>
<p>이 방식의 장점은 문법이 더 간단하고 사용자 오류의 가능성이 줄어든다는 것입니다.</p>
<h2>'asynchronize' 키워드</h2>
<p>'synchronize' 블록 내에서 코드를 실행하는 동안, 프로그래머는 잠시 비동기적으로 실행하여 블로킹 I/O 루틴이나 불확실한 시간을 필요로 하고 동기화를 요구하지 않는 다른 작업을 수행하기를 원할 수 있습니다. 이 코드는 일반적으로 다음 패턴을 따릅니다.</p>
<p><strong>현재의 비동기화 패턴 (동기화 블록 내):</strong></p>
<pre><code class="language-python">initialize_lock()
...
acquire_lock()
try:
    change_shared_data()
    release_lock() # 비동기 상태로 전환
    do_blocking_io()
    acquire_lock() # 다시 동기화
    change_shared_data2()
finally:
    release_lock()
</code></pre>
<p>코드의 비동기 섹션은 시각적으로 명확하지 않아 주석으로 표시해야 합니다. 제안된 'asynchronize' 키워드를 사용하면 이 코드는 훨씬 깔끔해지고 이해하기 쉬워지며 오류 발생 가능성이 줄어듭니다.</p>
<p><strong>제안된 'asynchronize' 패턴:</strong></p>
<pre><code class="language-python">synchronize:
    change_shared_data()
    asynchronize:
        do_blocking_io()
    change_shared_data2()
</code></pre>
<p>비동기 블록이 아닌 곳에서 'asynchronize' 키워드를 만나면 오류가 발생하거나 경고가 발생할 수 있습니다 (모든 코드 블록은 암묵적으로 비동기적이기 때문입니다). 이 키워드는 특히 메인 루프 내에서 블로킹 I/O를 위해 락을 해제하는 패턴에 유용하며, CPython 인터프리터 자체에서도 이 패턴이 광범위하게 사용됩니다.</p>
<h2>동기화 대상 (Synchronization Targets)</h2>
<p>제안된 'synchronize' 및 'asynchronize' 키워드는 코드 블록을 동기화합니다. 그러나 프로그래머는 스레드가 동기화할 대상 객체를 지정하기를 원할 수 있습니다. 모든 객체는 동기화 대상이 될 수 있습니다.</p>
<p>예를 들어, 두 개의 큐를 개별적으로 동기화하는 <code>TwoWayQueue</code> 클래스에서 <code>synchronize queue:</code>와 같이 특정 객체에 대한 동기화를 지정할 수 있습니다. 현재 Python에서는 이 기능을 위해 큐와 락을 연결하는 별도의 클래스를 정의해야 합니다.</p>
<p><strong>제안된 <code>synchronize</code> 키워드를 사용한 <code>TwoWayQueue</code> 예시:</strong></p>
<pre><code class="language-python">class TwoWayQueue:
    def __init__(self):
        self.front = []
        self.rear = []

    # ... (putFront, getFront, putRear, getRear 메서드 생략)

    def put(self, item, queue):
        synchronize queue: # 'queue' 객체를 동기화 대상으로 지정
            queue.append(item)

    def get(self, queue):
        synchronize queue: # 'queue' 객체를 동기화 대상으로 지정
            item = queue[0]
            del queue[0]
            return item
</code></pre>
<p><strong>현재 Python에서의 동일한 기능 (락 객체를 직접 관리):</strong></p>
<pre><code class="language-python">import thread

class LockableQueue:
    def __init__(self):
        self.queue = []
        self.lock = thread.allocate_lock()

class TwoWayQueue:
    def __init__(self):
        self.front = LockableQueue()
        self.rear = LockableQueue()

    # ... (putFront, getFront, putRear, getRear 메서드 생략)

    def put(self, item, queue):
        queue.lock.acquire()
        try:
            queue.queue.append(item)
        finally:
            queue.lock.release()

    def get(self, queue):
        queue.lock.acquire()
        try:
            item = queue.queue[0]
            del queue.queue[0]
            return item
        finally:
            queue.lock.release()
</code></pre>
<p>이전 예제는 락을 큐와 연결하기 위해 추가 클래스를 정의해야 했지만, 첫 번째 예제에서는 <code>synchronize</code> 키워드가 이 연결을 내부적으로 투명하게 수행합니다.</p>
<h2>기타 동기화 패턴 (Other Patterns that Synchronize)</h2>
<p><code>acquire</code> 및 <code>release</code> 메서드에 인수를 제공해야 하는 경우나 락이 한 코드 블록에서 획득되고 다른 블록에서 해제되는 경우에는 <code>synchronize</code> 및 <code>asynchronize</code> 키워드가 락 메서드의 사용을 완전히 대체할 수 없습니다. Zope의 <code>ZRendevous</code> 클래스 예시를 통해 이러한 경우를 보여줍니다. 이 클래스는 명시적인 락 풀(pool)을 사용하며, 락의 획득 및 해제가 다른 코드 블록에서 이루어져 <code>synchronize</code>를 직접 사용할 수 없는 경우도 있습니다.</p>
<p>초보 프로그래머에게 'synchronize'와 'asynchronize'는 여러 <code>try/finally</code> 블록에서 다른 락에 대한 여러 <code>acquire</code> 및 <code>release</code> 메서드를 처리할 때 발생하는 많은 문제를 제거할 수 있습니다.</p>
<h2>형식 문법 (Formal Syntax)</h2>
<p>제안된 동기화 문법은 다음과 같습니다.</p>
<pre><code class="language-bnf">synchronize_stmt: 'synchronize' [test] ':' suite
asynchronize_stmt: 'asynchronize' [test] ':' suite
compound_stmt: ... | synchronized_stmt | asynchronize_stmt
</code></pre>
<p>여기서 <code>test</code>는 동기화 대상 객체를 나타내며, <code>suite</code>는 동기화될 코드 블록입니다.</p>
<h2>제안된 구현 (Proposed Implementation)</h2>
<p>이 PEP의 작성자는 아직 구현을 탐색하지 않았습니다. 주요 구현 문제는 동기화 블록 동안 정확히 무엇이 락되고 언락되는지입니다. 대상 인수가 없는 'synchronize' 키워드 사용 시(unqualified synchronized block)에는 락이 생성되어 동기화된 코드 블록 객체와 연결될 수 있습니다. 'synchronize' 블록 내에서 'asynchronize' 키워드가 발생하면 내부 블록이 실행되기 전에 코드 블록 락이 해제되고, 내부 블록이 종료될 때 다시 락이 설정됩니다. 특정 동기화 대상이 지정되면 해당 객체에 락이 연결됩니다.</p>
<h2>하위 호환성 (Backward Compatibility)</h2>
<p>하위 호환성은 <code>from __future__</code> 구문 (PEP 236)과 새로운 경고 프레임워크 (PEP 230)를 통해 해결됩니다. 개발자는 <code>from __future__ import threadsync</code>와 같은 문장을 사용하여 이 새로운 문법을 사용할 수 있습니다. 또한, 'synchronize' 또는 'asynchronize'를 식별자로 사용하는 모든 코드는 Python에서 경고를 발행합니다.</p>
<h2>PEP 310 신뢰할 수 있는 획득/해제 쌍 (<code>with</code> 키워드)</h2>
<p>PEP 310은 <code>synchronize</code>와 동일한 기능을 수행할 수 있는 <code>with</code> 키워드를 제안합니다 (단, <code>asynchronize</code> 기능은 없습니다). 다음 패턴:</p>
<pre><code class="language-python">initialize_lock()
with the_lock:
    change_shared_data()
</code></pre>
<p>는 제안된 <code>synchronize the_lock: change_shared_data()</code>와 동일합니다. PEP 310의 <code>with</code> 문은 기존 락에 대해 동기화해야 하는 반면, 이 PEP는 자격 없는(unqualified) <code>synchronize</code> 문이 전역적, 내부적, 투명한 락에 대해서도 동기화할 것을 제안합니다. <code>with</code> 문은 락 초기화를 요구하는 반면, <code>synchronize</code> 문은 락을 포함한 모든 대상 객체에 대해 동기화할 수 있습니다.</p>
<p><code>with</code> 문은 동기화보다 더 추상적이며 다양한 목적(예: 트랜잭션 관리)에 사용될 수 있습니다. 반면, <code>synchronize</code> 및 <code>asynchronize</code> 키워드는 스레드 동기화 외의 다른 일반적인 획득/해제 패턴에는 사용될 수 없습니다.</p>
<h2>Java는 어떻게 하는가 (How Java Does It)</h2>
<p>Java는 'synchronized' 키워드를 정의합니다 (이 PEP의 'synchronize'와 문법적 시제 차이에 유의). 이 키워드는 모든 객체에 대해 한정되어야 하며, <code>synchronized (Expression) Block</code>과 같은 문법을 가집니다. <code>Expression</code>은 <code>Block</code>이 동기화될 유효한 객체를 생성해야 합니다.</p>
<h2>Jython은 어떻게 하는가 (How Jython Does It)</h2>
<p>Jython은 정적 메서드 'make_synchronized'를 가진 'synchronize' 클래스를 사용합니다. 이 메서드는 하나의 호출 가능한 인수를 받아 인수를 감싸는 새롭고 동기화된 호출 가능한 "래퍼(wrapper)"를 반환합니다.</p>
<h2>Python에 제안된 변경 사항 요약 (Summary of Proposed Changes to Python)</h2>
<p>언어에 새로운 'synchronize' 및 'asynchronize' 키워드를 추가하는 것이었습니다.</p>
<h2>위험 (Risks)</h2>
<ul>
<li>이 PEP는 Python 언어에 두 개의 키워드를 추가하여 기존 코드를 손상시킬 수 있었습니다.</li>
<li>테스트할 구현이 없었습니다.</li>
<li>당시 Python 프로그래머가 직면한 가장 중요한 문제는 아니었습니다 (비록 꽤 악명 높은 문제이기는 했습니다).</li>
<li>Java의 유사 키워드는 과거 분사형 'synchronized'였지만, 이 PEP는 Python의 정신에 더 가깝게 현재 시제 'synchronize'를 제안했습니다.</li>
</ul>

<p>이 PEP는 python-dev 메일링 리스트에서 논의되지 않았습니다.</p>
