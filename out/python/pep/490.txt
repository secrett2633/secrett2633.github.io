<h1 id="pep-490-c-수준에서-예외-연결-chain-exceptions-at-c-level"><a href="#pep-490-c-수준에서-예외-연결-chain-exceptions-at-c-level">PEP 490 – C 수준에서 예외 연결 (Chain exceptions at C level)</a></h1>
<ul>
<li><strong>작성자:</strong> Victor Stinner <vstinner at python.org></li>
<li><strong>상태:</strong> Rejected (거부됨)</li>
<li><strong>유형:</strong> Standards Track</li>
<li><strong>생성일:</strong> 2015년 3월 25일</li>
<li><strong>Python 버전:</strong> 3.6</li>
</ul>
<h2 id="요약-abstract"><a href="#요약-abstract">요약 (Abstract)</a></h2>
<p>이 PEP는 Python 수준에서 이미 예외가 연결(chained)되는 것처럼, C 수준에서도 예외를 연결하는 것을 제안합니다.</p>
<h2 id="도입-배경-rationale"><a href="#도입-배경-rationale">도입 배경 (Rationale)</a></h2>
<p>Python 3은 기본적으로 예외를 연결(chain)하는 새로운 기능을 도입했습니다 (PEP 3134).</p>
<p>예를 들어, 다음 Python 코드를 보면:</p>
<pre><code class="language-python">try:
    raise TypeError("err1")
except TypeError:
    raise ValueError("err2")
</code></pre>
<p>출력은 다음과 같습니다.</p>
<pre><code>Traceback (most recent call last):
  File "test.py", line 2, in &#x3C;module>
    raise TypeError("err1")
TypeError: err1
During handling of the above exception, another exception occurred:
Traceback (most recent call last):
  File "test.py", line 4, in &#x3C;module>
    raise ValueError("err2")
ValueError: err2
</code></pre>
<p>Python 코드에서는 예외가 기본적으로 연결되지만, C로 작성된 확장 모듈에서는 그렇지 않습니다.</p>
<p>새로운 비공개 함수 <code>_PyErr_ChainExceptions()</code>는 Python 3.4.3 및 3.5에서 예외를 연결하기 위해 도입되었습니다. 현재 이 함수는 예외를 연결하기 위해 명시적으로 호출해야 하며, 그 사용법이 간단하지 않습니다.</p>
<p><code>zipimport</code> 모듈에서 이전 <code>OSError</code>를 새로운 <code>ZipImportError</code> 예외에 연결하기 위해 <code>_PyErr_ChainExceptions()</code>를 사용하는 예시는 다음과 같습니다.</p>
<pre><code class="language-c">PyObject *exc, *val, *tb;
PyErr_Fetch(&#x26;exc, &#x26;val, &#x26;tb);
PyErr_Format(ZipImportError, "can't open Zip file: %R", archive);
_PyErr_ChainExceptions(exc, val, tb);
</code></pre>
<p>이 PEP는 일관성을 유지하고 디버깅에 도움이 되는 더 많은 실패 정보를 제공하기 위해 C 수준에서도 예외를 자동으로 연결할 것을 제안합니다. 위의 예시는 단순히 다음과 같이 될 수 있습니다.</p>
<pre><code class="language-c">PyErr_Format(ZipImportError, "can't open Zip file: %R", archive);
</code></pre>
<h2 id="제안-proposal"><a href="#제안-proposal">제안 (Proposal)</a></h2>
<h3 id="pyerr_-함수들을-수정하여-예외-연결"><a href="#pyerr_-함수들을-수정하여-예외-연결"><code>PyErr_*()</code> 함수들을 수정하여 예외 연결</a></h3>
<p>Python C API에서 예외를 발생시키는 C 함수들(<code>PyErr_SetString()</code>, <code>PyErr_Format()</code>, <code>PyErr_SetNone()</code> 등)을 수정하여 예외를 자동으로 연결하도록 제안합니다.</p>
<h3 id="예외를-연결하지-않는-함수-수정"><a href="#예외를-연결하지-않는-함수-수정">예외를 연결하지 않는 함수 수정</a></h3>
<p>새로운 예외가 이전 예외의 정보를 포함하거나 더 많은 정보를 포함하는 경우, 특히 두 예외의 유형이 같은 경우에는 이전 예외를 유지하는 것이 항상 유용하지 않을 수 있습니다.</p>
<p><code>int(str)</code>을 사용한 불필요한 예외 연결의 예시는 다음과 같습니다.</p>
<pre><code>TypeError: a bytes-like object is required, not 'type'
During handling of the above exception, another exception occurred:
Traceback (most recent call last):
  File "&#x3C;stdin>", line 1, in &#x3C;module>
TypeError: int() argument must be a string, a bytes-like object or a number, not 'type'
</code></pre>
<p>새로운 <code>TypeError</code> 예외가 이전 예외보다 더 많은 정보를 포함하고 있으므로, 이전 예외는 숨겨져야 합니다.</p>
<p>새로운 예외와 현재 예외를 연결하지 않으려면, 새로운 예외를 발생시키기 전에 <code>PyErr_Clear()</code> 함수를 호출하여 현재 예외를 지울 수 있습니다.</p>
<h3 id="예외를-연결하는-함수-수정"><a href="#예외를-연결하는-함수-수정">예외를 연결하는 함수 수정</a></h3>
<p>일부 함수는 현재 예외를 저장한 다음 복원합니다. 새로운 예외가 발생하면, 현재 해당 예외는 함수에 따라 <code>sys.stderr</code>에 표시되거나 무시됩니다. 이러한 함수 중 일부는 대신 예외를 연결하도록 수정되어야 합니다.</p>
<p>새로운 예외를 무시하는 함수의 예시:</p>
<ul>
<li><code>ptrace_enter_call()</code>: 예외 무시</li>
<li><code>subprocess_fork_exec()</code>: <code>enable_gc()</code>에 의해 발생하는 예외 무시</li>
<li><code>_thread</code> 모듈의 <code>t_bootstrap()</code>: 부트스트랩 함수를 <code>sys.stderr</code>에 표시하려다 발생하는 예외 무시</li>
<li><code>PyDict_GetItem()</code>, <code>_PyDict_GetItem_KnownHash()</code>: 딕셔너리에서 키를 찾을 때 발생하는 예외 무시</li>
<li><code>_PyErr_TrySetFromCause()</code>: 예외 무시</li>
<li><code>PyFrame_LocalsToFast()</code>: <code>dict_to_map()</code>에 의해 발생하는 예외 무시</li>
<li><code>_PyObject_Dump()</code>: 예외 무시 (디버깅 용도로 Python 상태를 수정해서는 안 됨)</li>
<li><code>Py_ReprLeave()</code>: 예외 무시 (보고할 방법이 없기 때문)</li>
<li><code>type_dealloc()</code>: <code>remove_all_subclasses()</code>에 의해 발생하는 예외 무시</li>
<li><code>PyObject_ClearWeakRefs()</code>: 예외 무시</li>
<li><code>call_exc_trace()</code>, <code>call_trace_protected()</code>: 예외 무시</li>
<li><code>remove_importlib_frames()</code>: 예외 무시</li>
<li><code>do_mktuple()</code> (예: <code>Py_BuildValue()</code>에 사용되는 헬퍼): 예외 무시</li>
<li><code>flush_io()</code>: 예외 무시</li>
<li><code>sys_write()</code>, <code>sys_format()</code>: 예외 무시</li>
<li><code>_PyTraceback_Add()</code>: 예외 무시</li>
<li><code>PyTraceBack_Print()</code>: 예외 무시</li>
</ul>
<p>새로운 예외를 <code>sys.stderr</code>에 표시하는 함수의 예시:</p>
<ul>
<li><code>atexit_callfuncs()</code>: <code>PyErr_Display()</code>로 예외를 표시하고 최신 예외를 반환 (여러 콜백을 호출하고 최신 예외만 반환)</li>
<li><code>sock_dealloc()</code>: <code>ResourceWarning</code> 예외를 <code>PyErr_WriteUnraisable()</code>로 로깅</li>
<li><code>slot_tp_del()</code>: <code>PyErr_WriteUnraisable()</code>로 예외 표시</li>
<li><code>_PyGen_Finalize()</code>: <code>gen_close()</code> 예외를 <code>PyErr_WriteUnraisable()</code>로 표시</li>
<li><code>slot_tp_finalize()</code>: <code>__del__()</code> 메서드에 의해 발생하는 예외를 <code>PyErr_WriteUnraisable()</code>로 표시</li>
<li><code>PyErr_GivenExceptionMatches()</code>: <code>PyType_IsSubtype()</code>에 의해 발생하는 예외를 <code>PyErr_WriteUnraisable()</code>로 표시</li>
</ul>
<h2 id="하위-호환성-backward-compatibility"><a href="#하위-호환성-backward-compatibility">하위 호환성 (Backward compatibility)</a></h2>
<p>예외를 연결하는 것의 부작용 중 하나는 예외가 트레이스백(traceback) 객체를 저장하고, 트레이스백 객체는 프레임(frame) 객체를 저장하며, 프레임 객체는 지역 변수를 저장한다는 것입니다. 지역 변수는 예외에 의해 살아남게 됩니다. 일반적인 문제는 지역 변수와 예외 사이에 순환 참조(reference cycle)가 발생하는 것입니다. 즉, 예외가 지역 변수에 저장되고, 프레임이 예외에 간접적으로 저장되는 경우입니다. 이 순환 참조는 예외를 저장하는 애플리케이션에만 영향을 미칩니다.</p>
<p>이러한 순환 참조는 Python 3.5에 도입된 새로운 <code>traceback.TracebackException</code> 객체로 해결될 수 있습니다. 이 객체는 지역 변수를 저장하지 않고도 전체 텍스트 트레이스백을 포맷하는 데 필요한 정보를 저장합니다.</p>
<p><code>asyncio</code> 모듈은 순환 참조 문제의 영향을 받습니다. 이 모듈은 Python 3.3용 버전을 출시하기 위해 Python 표준 라이브러리 외부에서도 유지 관리됩니다. <code>traceback.TracebackException</code>은 순환 참조 문제를 해결하기 위해 비공개 <code>asyncio</code> 모듈로 백포트될 수 있습니다.</p>
<h2 id="대안-alternatives"><a href="#대안-alternatives">대안 (Alternatives)</a></h2>
<h3 id="변경-없음-no-change"><a href="#변경-없음-no-change">변경 없음 (No change)</a></h3>
<p>새로운 비공개 함수 <code>_PyErr_ChainExceptions()</code>만으로도 예외를 수동으로 연결하기에 충분합니다. 예외는 의미 있는 곳에서만 명시적으로 연결될 것입니다.</p>
<h3 id="새로운-예외-연결-헬퍼-함수-new-helpers-to-chain-exceptions"><a href="#새로운-예외-연결-헬퍼-함수-new-helpers-to-chain-exceptions">새로운 예외 연결 헬퍼 함수 (New helpers to chain exceptions)</a></h3>
<p><code>PyErr_SetString()</code>과 같은 함수는 예외를 자동으로 연결하지 않습니다. <code>_PyErr_ChainExceptions()</code>의 사용을 더 쉽게 만들기 위해 다음과 같은 새로운 비공개 함수들이 추가될 수 있습니다.</p>
<ul>
<li><code>_PyErr_SetStringChain(exc_type, message)</code></li>
<li><code>_PyErr_FormatChain(exc_type, format, ...)</code></li>
<li><code>_PyErr_SetNoneChain(exc_type)</code></li>
<li><code>_PyErr_SetObjectChain(exc_type, exc_value)</code></li>
</ul>
<p><code>_PyErr_SetKeyError(key)</code> 또는 <code>PyErr_SetImportError(message, name, path)</code>와 같은 특정 예외를 발생시키는 헬퍼 함수들은 예외를 연결하지 않습니다. 이 헬퍼 함수들과 예외를 연결하려면 일반적인 <code>_PyErr_ChainExceptions(exc_type, exc_value, exc_tb)</code> 함수를 사용해야 합니다.</p>
