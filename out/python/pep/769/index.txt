<h2>요약</h2>
<p>이 제안은 <code>operator</code> 모듈의 <code>attrgetter</code>, <code>itemgetter</code>, <code>getitem</code> 함수에 <code>default</code> 키워드 인자를 추가하여 기능을 개선하는 것을 목표로 했습니다. 이 인자가 추가되면, 지정된 속성 (attribute) 또는 항목 (item)이 없을 때 특정 기본값 (default value)을 반환하여 예외 발생을 방지하고 선택적 속성 또는 항목을 처리하는 코드를 간소화할 수 있습니다.</p>
<h2>동기</h2>
<p>현재 <code>attrgetter</code>와 <code>itemgetter</code>는 지정된 속성이나 항목이 존재하지 않으면 예외를 발생시킵니다. 이러한 한계 때문에 개발자는 추가적인 오류 처리를 구현해야 했고, 이는 코드를 더 복잡하고 가독성이 떨어지게 만들었습니다.</p>
<p><code>default</code> 매개변수를 도입하면 선택적 속성이나 항목과 관련된 작업을 간소화하고, 반복적인 코드 (boilerplate code)를 줄이며, 코드의 명확성을 높일 수 있습니다.</p>
<p><code>getitem</code>의 경우에도 비슷한 상황이 발생하는데, 이 경우 <code>default</code> 값을 지정할 수 있게 하면 내장 함수인 <code>getattr()</code>와 오랫동안 존재했던 비대칭성을 해결할 수 있다는 추가적인 이점이 있습니다.</p>
<h2>근거</h2>
<p>주요 설계 결정은 지정된 모든 속성 또는 항목에 적용되는 단일 <code>default</code> 매개변수를 도입하는 것이었습니다.</p>
<p>이러한 접근 방식은 단순성을 유지하고 여러 속성 또는 항목에 개별 기본값을 할당하는 복잡성을 피합니다. 다중 기본값을 허용하는 것에 대한 논의도 있었지만, 복잡성 증가와 혼란 가능성 때문에 모든 경우에 단일 기본값을 선호하게 되었습니다 (자세한 내용은 아래의 "거절된 아이디어" 참조).</p>
<h2>사양 (Specification)</h2>
<p>제안된 동작은 다음과 같습니다:</p>
<ul>
<li><strong><code>attrgetter</code></strong> : <code>f = attrgetter("name", default=XYZ)</code>를 사용한 다음 <code>f(obj)</code>를 호출하면, 속성이 존재할 경우 <code>obj.name</code>을 반환하고, 그렇지 않으면 <code>XYZ</code>를 반환합니다.</li>
<li><strong><code>itemgetter</code></strong> : <code>f = itemgetter(2, default=XYZ)</code>를 사용한 다음 <code>f(obj)</code>를 호출하면, <code>obj[2]</code>가 유효할 경우 이를 반환하고, 그렇지 않으면 <code>XYZ</code>를 반환합니다.</li>
<li><strong><code>getitem</code></strong> : <code>getitem(obj, k, XYZ)</code> 또는 <code>getitem(obj, k, default=XYZ)</code>를 사용하면, <code>obj[k]</code>가 유효할 경우 이를 반환하고, 그렇지 않으면 <code>XYZ</code>를 반환합니다.</li>
</ul>
<p>처음 두 가지 경우에 이 개선 사항은 단일 및 다중 속성/항목 검색에 모두 적용되며, 누락된 속성이나 항목에 대해 <code>default</code> 값이 반환됩니다.</p>
<p>추가적인 <code>default</code> (키워드) 인자가 사용되지 않는 경우에는 어떤 경우에도 기능 변경이 없습니다.</p>
<h3><code>attrgetter</code> 예시</h3>
<h4>현재 동작 (변경 없음):</h4>
<pre><code class="language-python">>>> class C:
...     class D:
...         class X:
...             pass
...     class E:
...         pass
...
>>> attrgetter("D")(C)
&#x3C;class '__main__.C.D'>
>>> attrgetter("badname")(C)
Traceback (most recent call last):
File "&#x3C;stdin>", line 1, in &#x3C;module>
AttributeError: type object 'C' has no attribute 'badname'
>>> attrgetter("D", "E")(C)
(&#x3C;class '__main__.C.D'>, &#x3C;class '__main__.C.E'>)
>>> attrgetter("D", "badname")(C)
Traceback (most recent call last):
File "&#x3C;stdin>", line 1, in &#x3C;module>
AttributeError: type object 'C' has no attribute 'badname'
>>> attrgetter("D.X")(C)
&#x3C;class '__main__.C.D.X'>
>>> attrgetter("D.badname")(C)
Traceback (most recent call last):
File "&#x3C;stdin>", line 1, in &#x3C;module>
AttributeError: type object 'D' has no attribute 'badname'
</code></pre>
<h4>이 PEP를 적용한 제안된 <code>default</code> 키워드 사용 시:</h4>
<pre><code class="language-python">>>> attrgetter("D", default="noclass")(C)
&#x3C;class '__main__.C.D'>
>>> attrgetter("badname", default="noclass")(C)
'noclass'
>>> attrgetter("D", "E", default="noclass")(C)
(&#x3C;class '__main__.C.D'>, &#x3C;class '__main__.C.E'>)
>>> attrgetter("D", "badname", default="noclass")(C)
(&#x3C;class '__main__.C.D'>, 'noclass')
>>> attrgetter("D.X", default="noclass")(C)
&#x3C;class '__main__.C.D.X'>
>>> attrgetter("D.badname", default="noclass")(C)
'noclass'
</code></pre>
<h3><code>itemgetter</code> 예시</h3>
<h4>현재 동작 (변경 없음):</h4>
<pre><code class="language-python">>>> obj = ["foo", "bar", "baz"]
>>> itemgetter(1)(obj)
'bar'
>>> itemgetter(5)(obj)
Traceback (most recent call last):
File "&#x3C;stdin>", line 1, in &#x3C;module>
IndexError: list index out of range
>>> itemgetter(1, 0)(obj)
('bar', 'foo')
>>> itemgetter(1, 5)(obj)
Traceback (most recent call last):
File "&#x3C;stdin>", line 1, in &#x3C;module>
IndexError: list index out of range
</code></pre>
<h4>이 PEP를 적용한 제안된 <code>default</code> 키워드 사용 시:</h4>
<pre><code class="language-python">>>> itemgetter(1, default="XYZ")(obj)
'bar'
>>> itemgetter(5, default="XYZ")(obj)
'XYZ'
>>> itemgetter(1, 0, default="XYZ")(obj)
('bar', 'foo')
>>> itemgetter(1, 5, default="XYZ")(obj)
('bar', 'XYZ')
</code></pre>
<h3><code>getitem</code> 예시</h3>
<h4>현재 동작 (변경 없음):</h4>
<pre><code class="language-python">>>> obj = ["foo", "bar", "baz"]
>>> getitem(obj, 1)
'bar'
>>> getitem(obj, 5)
Traceback (most recent call last):
File "&#x3C;stdin>", line 1, in &#x3C;module>
IndexError: list index out of range
</code></pre>
<h4>이 PEP를 적용한 제안된 <code>default</code> 인자 (위치 인자 또는 키워드 인자) 사용 시:</h4>
<pre><code class="language-python">>>> getitem(obj, 1, "XYZ")
'bar'
>>> getitem(obj, 5, "XYZ")
'XYZ'
>>> getitem(obj, 1, default="XYZ")
'bar'
>>> getitem(obj, 5, default="XYZ")
'XYZ'
</code></pre>
<h2>구현 가능성에 대한 고려사항</h2>
<p><code>attrgetter</code>의 구현은 비교적 간단합니다. <code>getattr</code>를 사용하고 발생할 수 있는 <code>AttributeError</code>를 캐치하는 것을 의미합니다. 따라서 <code>attrgetter("name", default=XYZ)(obj)</code>는 다음과 유사하게 동작할 것입니다:</p>
<pre><code class="language-python">try:
    value = getattr(obj, "name")
except AttributeError:
    value = XYZ
</code></pre>
<p>참고로, 속성 체인 (예: <code>attrgetter("foo.bar.baz", default=XYZ)</code>)이 지정될 때 각 단계에서 무엇이 반환되었는지 구별할 수 없으므로, <code>getattr</code>에 기본값을 사용하여 구현할 수는 없습니다.</p>
<p><code>itemgetter</code>와 <code>getitem</code>의 구현은 그리 쉽지 않습니다. 가장 직관적인 방법은 <code>__getitem__</code>을 시도하고 가능한 예외를 캐치하는 것입니다. 이렇게 하면 <code>itemgetter(123, default=XYZ)(obj)</code> 또는 <code>getitem(obj, 123, default=XYZ)</code>는 다음 코드와 동일하게 동작할 것입니다:</p>
<pre><code class="language-python">try:
    value = obj[123]
except (IndexError, KeyError):
    value = XYZ
</code></pre>
<p>그러나 성능상의 이유로 구현은 다음과 같이 보일 수 있으며, 이는 정확히 동일한 동작을 합니다:</p>
<pre><code class="language-python">if type(obj) == dict:
    value = obj.get(123, XYZ)
else:
    try:
        value = obj[123]
    except (IndexError, KeyError):
        value = XYZ
</code></pre>
<p>여기서 확인은 <code>isinstance</code>를 사용하지 않고 정확한 <code>type</code>에 대해 이루어집니다. 이는 <code>dict</code>를 상속하지만 <code>get</code> 메서드를 덮어쓰는 사용자 정의 객체인 경우 정확한 동작을 보장하기 위함입니다.</p>
<p>이러한 방식은 성능이 더 좋지만, 이는 구현 세부 사항일 뿐이므로 동작 방식에 대한 원래 설명을 유지할 수 있습니다.</p>
<p>캡처할 예외와 관련하여, <code>__getitem__</code>은 <code>IndexError</code>, <code>KeyError</code>, 또는 <code>TypeError</code>를 발생시킬 수 있지만 (참조), 컨테이너에 지정된 키나 인덱스가 없는 경우에만 처음 두 가지가 발생할 수 있습니다. <code>TypeError</code>는 코드의 버그를 나타낼 가능성이 높으므로 기본 동작을 트리거하기 위해 캡처하지 않습니다.</p>
<h2>코너 케이스 (Corner Cases)</h2>
<p>기본값 옵션은 항목/속성 접근이 일반적인 경우 실패할 때만 작동합니다. 즉, 접근하는 객체 자체가 기본값을 처리해서는 안 됩니다.</p>
<p>예를 들어, <code>defaultdict</code>는 항목에 접근할 때 결코 오류를 발생시키지 않으므로 다음 코드는 중복되거나 혼란스러울 수 있습니다:</p>
<pre><code class="language-python">>>> from collections import defaultdict
>>> from operator import itemgetter
>>> dd = defaultdict(int)
>>> itemgetter("foo", default=-1)(dd)
0
</code></pre>
<p><code>__getitem__</code> 또는 <code>__getattr__</code>을 오버로드하여 자체 대체 (fallback) 로직을 구현하는 모든 사용자 정의 객체에도 동일하게 적용됩니다.</p>
<h2>거절된 아이디어</h2>
<h3>다중 기본값 (Multiple Default Values)</h3>
<p>여러 속성 또는 항목에 대해 여러 기본값을 허용하는 아이디어가 고려되었습니다.</p>
<p>두 가지 대안이 논의되었습니다. 하나는 <code>attrgetter</code> / <code>itemgetter</code>에 전달된 매개변수의 개수와 동일한 수의 항목을 가져야 하는 이터러블 (iterable)을 사용하는 것이었고, 다른 하나는 <code>attrgetter</code> / <code>itemgetter</code>에 전달된 이름과 일치하는 키를 가진 딕셔너리를 사용하는 것이었습니다.</p>
<p>여기서 해결하기 정말 복잡한 문제 (기능을 설명하기 어렵게 만들고 혼란스러운 코너 케이스를 유발하는)는 이터러블이나 딕셔너리가 모든 항목에 대해 실제로 원하는 기본값일 경우 어떤 일이 발생하느냐였습니다. 예를 들어:</p>
<pre><code class="language-python">>>> itemgetter("a", default=(1, 2))({})
(1, 2)
>>> itemgetter("a", "b", default=(1, 2))({})
((1, 2), (1, 2))
</code></pre>
<p><code>default</code>를 사용하여 "다중 기본값"을 허용한다면, 위 예시의 첫 번째 경우는 <code>default</code>에 이름보다 많은 항목이 있기 때문에 예외가 발생하고, 두 번째 경우는 <code>((1, 2), (1, 2))</code>를 반환할 것입니다. 이 때문에 다중 기본값에 대해 다른 이름 (예: <code>defaults</code> - 표현력이 좋지만 <code>default</code>와 너무 비슷하여 오류 발생 가능성이 있음)을 사용하는 가능성을 고려했습니다.</p>
<p>다중 기본값을 가능하게 하는 또 다른 제안은 <code>attrgetter</code>와 <code>itemgetter</code>의 조합을 허용하는 것이었습니다. 예를 들어:</p>
<pre><code class="language-python">>>> ig_a = itemgetter("a", default=1)
>>> ig_b = itemgetter("b", default=2)
>>> ig_combined = itemgetter(ig_a, ig_b)
>>> ig_combined({"a": 999})
(999, 2)
>>> ig_combined({})
(1, 2)
</code></pre>
<p>그러나 <code>itemgetter</code> 또는 <code>attrgetter</code>를 조합하는 것은 완전히 새로운 동작이며 정의하기 매우 복잡합니다. 불가능하지는 않지만, 이 PEP의 범위를 벗어납니다.</p>
<p>결론적으로, 다중 기본값을 갖는 것은 지나치게 복잡하고 잠재적으로 혼란스러울 수 있다고 판단되었으며, 단순성과 예측 가능성을 위해 단일 <code>default</code> 매개변수가 선호되었습니다.</p>
<h3>튜플 반환 일관성 (Tuple Return Consistency)</h3>
<p>또 다른 거절된 제안은 주어진 키/이름/인덱스의 개수와 상관없이 항상 튜플을 반환하도록 하는 플래그를 추가하는 것이었습니다. 예를 들어:</p>
<pre><code class="language-python">>>> letters = ["a", "b", "c"]
>>> itemgetter(1, return_tuple=True)(letters)
('b',)
>>> itemgetter(1, 2, return_tuple=True)(letters)
('b', 'c')
</code></pre>
<p>이것은 다중 기본값 일관성에 거의 도움이 되지 않으며, 추가적인 논의가 필요하고, 이 PEP의 범위를 벗어납니다.</p>
<h2>오픈 이슈</h2>
<p>현재 열려 있는 이슈는 없습니다.</p>
<h2>교육 방법</h2>
<p>기본 동작이 수정되지 않으므로, <code>attrgetter</code>와 <code>itemgetter</code>를 처음 가르칠 때는 이 새로운 <code>default</code> 기능을 생략할 수 있습니다. 기능이 필요할 때만 소개될 수 있습니다.</p>
<h2>하위 호환성</h2>
<p>제안된 변경 사항은 하위 호환성을 가집니다. <code>default</code> 매개변수는 선택 사항이며, 이 매개변수 없이 기존 코드는 이전과 동일하게 작동할 것입니다. 새로운 <code>default</code> 매개변수를 명시적으로 사용하는 코드만 새로운 동작을 보일 것이며, 현재 구현에 어떤 방해도 주지 않습니다.</p>
<h2>보안 영향</h2>
<p><code>default</code> 매개변수를 도입하는 것은 본질적으로 보안 취약점을 유발하지 않습니다.</p>
