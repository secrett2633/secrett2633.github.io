<p><strong>원문 링크:</strong> <a href="https://peps.python.org/pep-0347/">PEP 347 - Migrating the Python CVS to Subversion</a></p>
<p><strong>상태:</strong> Final | <strong>유형:</strong> Process | <strong>작성일:</strong> 14-Jul-2004</p>
</blockquote>
<p>PEP 347 – Python CVS를 Subversion으로 마이그레이션</p>
<h2>개요</h2>
<p>이 문서는 Python 소스 코드를 SourceForge.net의 CVS 저장소에서 svn.python.org의 Subversion 저장소로 이전할 것을 제안합니다.</p>
<h2>도입 배경 (Rationale)</h2>
<p>이번 변경은 두 가지 측면을 가집니다: CVS에서 Subversion으로의 전환, 그리고 SourceForge에서 python.org로의 전환입니다. 각각의 배경은 다음과 같습니다.</p>
<h3>Subversion으로의 전환 (Moving to Subversion)</h3>
<p>CVS는 Subversion에서 개선된 여러 가지 한계를 가지고 있었습니다. Python 개발에 있어 가장 주목할 만한 개선 사항들은 다음과 같습니다.</p>
<ul>
<li><strong>파일 및 디렉터리 이름 변경 및 삭제 이력 유지:</strong> 파일 및 디렉터리 이름을 변경하거나 디렉터리를 삭제할 때도 해당 파일들의 이력을 계속 유지할 수 있습니다.</li>
<li><strong>Change Sets 지원:</strong> 전역 리비전 번호를 통해 여러 파일에 대한 연관된 변경 사항(change sets)을 지원합니다. Change sets는 트랜잭션 방식으로 처리됩니다.</li>
<li><strong>원자적이고 빠른 태그 지정 (Tagging):</strong> CVS 태그 작업은 몇 분이 걸릴 수 있었지만, Subversion 태그( <code>svn cp</code> )는 빠르고 원자적으로 완료됩니다. 브랜치(Branch) 작업 또한 매우 효율적입니다.</li>
<li><strong>오프라인 Diff 지원:</strong> 패치를 생성할 때 유용한 오프라인 Diff 기능을 지원합니다.</li>
</ul>
<h3>python.org로의 전환 (Moving to python.org)</h3>
<p>SourceForge는 지난 몇 년간 중요한 인프라를 제공해 주었습니다. 하지만 SourceForge에 대한 관심은 과거에 반복적인 과부하 상황을 야기했으며, SourceForge 운영자들은 항상 제때 대응할 수 없었습니다. 특히 CVS의 경우, 익명 접근을 위한 두 번째 읽기 전용 CVS 서버를 도입하여 주 CVS 서버의 부하를 줄여야 했습니다. 이 서버는 주기적으로 동기화되지만, 읽기-쓰기 CVS 저장소와 동기화 사이에 지연이 발생했습니다. 결과적으로 커밋 권한이 없는 사용자들은 최근 저장소 변경 사항을 지연 후에야 볼 수 있었습니다.</p>
<p>python.org에서는 저장소를 익명으로 접근할 수 있도록 만들 수 있을 것입니다.</p>
<h2>마이그레이션 절차 (Migration Procedure)</h2>
<p>Python CVS 저장소를 이전하기 위해 다음 단계들이 실행되어야 합니다. 각 단계는 아래 섹션에서 더 자세히 설명됩니다.</p>
<ol>
<li><strong>SSH 키 수집:</strong> 현재 모든 커미터(committer)들의 SSH 키와 커밋 메시지에 표시될 사용자 이름을 수집합니다.</li>
<li><strong>SourceForge 저장소 폐쇄 공지:</strong> 마이그레이션 시작 시 SourceForge 저장소 폐쇄를 공지합니다.</li>
<li><strong>CVS 저장소 다운로드:</strong> 마지막 커밋으로부터 24시간 후에 CVS 저장소를 다운로드합니다.</li>
<li><strong>CVS 저장소 변환:</strong> CVS 저장소를 Subversion 저장소로 변환합니다.</li>
<li><strong>저장소 공개:</strong> 커미터들에게는 쓰기 권한을, 익명 사용자들에게는 읽기 전용 접근 권한을 부여하여 저장소를 공개합니다.</li>
<li><strong>SourceForge의 CVS 접근 비활성화:</strong> SourceForge에서 CVS 접근을 비활성화합니다.</li>
</ol>
<h3>SSH 키 수집 (Collect SSH keys)</h3>
<p><code>svn+ssh</code> 가 저장소에 대한 쓰기 접근을 위한 최적의 방법으로 선택되었습니다. 개발자들은 기존 SSH 키를 계속 사용할 수 있지만, python.org에 설치되어야 합니다.</p>
<p>각 개발자마다 새로운 Unix 사용자를 생성하는 것을 피하기 위해, <code>authorized_keys</code> 파일에 <code>command=</code> 속성을 가진 단일 계정을 사용해야 합니다.</p>
<p><code>authorized_keys</code> 파일의 라인 예시는 다음과 같습니다 (가독성을 위해 줄 바꿈).</p>
<pre><code>command="/usr/bin/svnserve --root=/svnroot -t --tunnel-user='&#x3C;username>'",no-port-forwarding, no-X11-forwarding,no-agent-forwarding,no-pty ssh-dss &#x3C;key> &#x3C;comment>
</code></pre>
<p>사용자 이름으로는 SourceForge 계정 이름 대신 실제 이름을 사용하여 로그 메시지에서 사람들을 더 잘 식별할 수 있도록 합니다.</p>
<h3>관리자 접근 (Administrator Access)</h3>
<p><code>pythondev</code> 계정에 대한 관리자 접근 권한은 현재 Python SourceForge 프로젝트의 모든 관리자에게 부여되어야 합니다. 셸(shell) 로그인과 <code>svnserve</code> 로그인을 구별하기 위해 관리자는 두 개의 키를 유지해야 합니다. OpenSSH를 사용하여 두 번째 키를 생성하는 절차는 다음과 같습니다.</p>
<pre><code class="language-bash">cd .ssh
ssh-keygen -t DSA -f pythondev -C &#x3C;user>@pythondev
vi config
</code></pre>
<p><code>config</code> 파일에 다음 줄을 추가해야 합니다.</p>
<pre><code>Host pythondev
Hostname dinsdale.python.org
User pythondev
IdentityFile ~/.ssh/pythondev
</code></pre>
<p>그러면 " <code>ssh pythondev</code> "를 통해 셸 로그인이 가능해집니다.</p>
<h3>CVS 저장소 다운로드 (Downloading the CVS Repository)</h3>
<p>CVS 저장소는 다음 URL에서 다운로드할 수 있습니다.</p>
<p><code>http://cvs.sourceforge.net/cvstarballs/python-cvsroot.tar.bz2</code></p>
<p>이 <code>tarball</code>은 하루에 한 번만 생성되므로, 저장소 동결(repository freeze) 후에 <code>tarball</code>을 가져오기까지 시간이 좀 걸릴 수 있습니다. <code>python-commits</code> 메일링 리스트에 기록된 마지막 커밋이 <code>tarball</code>에 실제로 포함되었는지 확인해야 합니다.</p>
<p>변환 후 변환된 CVS <code>tarball</code>은 <code>www.python.org/archive/python-cvsroot-&#x3C;date>.tar.bz2</code> 에 영구적으로 보관해야 합니다.</p>
<h3>CVS 저장소 변환 (Converting the CVS Repository)</h3>
<p>Python CVS 저장소는 <code>distutils</code> 와 <code>python</code> 두 개의 모듈을 포함합니다. <code>python</code> 모듈은 다시 <code>dist</code> 와 <code>nondist</code> 로 구성되며, <code>dist</code> 는 <code>src</code> (실제 Python 코드)만 포함합니다. <code>nondist</code> 에는 다양한 하위 디렉터리가 있습니다.</p>
<p>이들은 <code>&#x3C;project>/{trunk,tags,branches}</code> 구조를 따라 더 짧은 URL을 얻기 위해 Subversion 저장소에서 재구성되어야 합니다. 각 <code>nondist</code> 디렉터리, <code>src</code> ( <code>python</code> 이라고 명명), 그리고 <code>distutils</code> 에 대해 프로젝트가 생성될 것입니다. 저장소 재구성은 아래와 같이 CVS 트리에서 수행하는 것이 가장 좋습니다.</p>
<p>저장소 형식으로 <code>fsfs</code> 백엔드(Subversion 1.1 필요)를 사용해야 합니다. <code>fsfs</code> 백엔드는 <code>dump</code> 명령을 실행할 필요 없이 점진적 저장소 백업을 허용하므로 백업에 더 유리하다는 장점이 있습니다.</p>
<p>변환은 <code>cvs2svn</code> 유틸리티(예: <code>cvs2svn</code> Debian 패키지에서 사용 가능)를 사용하여 수행해야 합니다. <code>cvs2svn</code> 은 현재 <code>project/trunk</code> 구조를 지원하지 않으므로, 각 프로젝트는 개별적으로 변환되어야 합니다. 각 변환 결과를 대상 저장소의 별도 디렉터리에 넣으려면 <code>svnadmin load</code> 를 사용해야 합니다.</p>
<p>Subversion은 이진 파일과 텍스트 파일에 대해 CVS와 다른 관점을 가집니다. CVS 의미론을 올바르게 전달하기 위해, CVS에서 이진 파일로 표시되지 않은 모든 파일에 <code>svn:eol-style</code> 을 <code>native</code> 로 설정해야 합니다.</p>
<p>요약하자면, 변환 스크립트는 다음과 같습니다.</p>
<pre><code class="language-sh">#!/bin/sh
rm cvs2svn-*
rm -rf python py.new
tar xjf python-cvsroot.tar.bz2
rm -rf python/CVSROOT
svnadmin create --fs-type fsfs py.new
mv python/python python/orig
mv python/orig/dist/src python/python
mv python/orig/nondist/* python # nondist/nondist is empty
rmdir python/nondist
rm -rf python/orig
for a in python/*
do
  b=`basename $a`
  cvs2svn -q --dump-only --encoding=latin1 --force-branch=cnri-16-start \
    --force-branch=descr-branch --force-branch=release152p1-patches \
    --force-tag=r16b1 $a
  svn mkdir -m"Conversion to SVN" file:///`pwd`/py.new/$b
  svnadmin load -q --parent-dir $b py.new &#x3C; cvs2svn-dump
  rm cvs2svn-dump
done
</code></pre>
<p>이 변환의 샘플 결과는 <code>http://www.dcl.hpi.uni-potsdam.de/pysvn/</code> 에서 확인할 수 있습니다.</p>
<h3>저장소 공개 (Publish the Repository)</h3>
<p>저장소는 <code>http://svn.python.org/projects</code> 에 공개되어야 합니다. 모든 현재 SourceForge 커미터에게는 <code>svn+ssh://pythondev@svn.python.org/</code> 를 통해 읽기-쓰기 접근 권한을 부여하고, 웹DAV(WebDAV)를 통한 읽기 전용 익명 접근 권한도 부여해야 합니다.</p>
<p>선택적으로 <code>websvn</code> (예: Debian <code>websvn</code> 패키지에서 사용 가능)을 제공할 수 있습니다. 하지만 테스트 설치에서는 <code>websvn</code> 이 메모리 부족으로 인해 작동이 중단되었습니다.</p>
<p>현재 SourceForge 프로젝트 관리자는 <code>pythondev</code> 계정의 <code>authorized_keys2</code> 파일에 대한 쓰기 권한을 얻어야 합니다.</p>
<h3>CVS 비활성화 (Disable CVS)</h3>
<p>CVS를 완전히 비활성화하는 것은 불가능한 것으로 보입니다. 프로젝트 페이지에서 사용자 인터페이스만 제거할 수 있으며, 저장소 자체는 계속 사용할 수 있습니다. 필요하다면, <code>python</code> 및 <code>distutils</code> 모듈에 대한 쓰기 접근은 CVS <code>commitinfo</code> 항목을 통해 비활성화할 수 있습니다.</p>
<h2>논의 (Discussion)</h2>
<p>위 절차에 대한 몇 가지 대안이 제안되었습니다. 거부된 대안들은 여기서 간략하게 논의됩니다.</p>
<ul>
<li><strong>다중 저장소 생성:</strong> <code>python</code> 과 <code>distutils</code> 에 대해 각각 하나의 저장소를 생성하는 방안이 있었습니다. 이는 URL을 더 짧게 만들 수 있었겠지만, 단일 저장소가 프로젝트 간 코드 이동을 지원하므로 거부되었습니다.</li>
<li><strong>표준 <code>cvs2svn</code> 및 이름 변경을 통한 <code>project/trunk</code> 구조 생성:</strong> 이는 이전 리비전이 최근 리비전과 다른 경로 이름을 사용한다는 단점이 있었습니다. <code>dump</code> 파일을 통한 제안된 접근 방식은 이름 변경 없이 작동합니다.</li>
<li><strong>python.org 호스팅에 대한 관리 부담:</strong> 여러 사람이 python.org에 저장소를 호스팅하는 것이 <code>pydotorg</code> 관리자에게 발생시킬 관리 오버헤드에 대해 우려를 표했습니다. <code>BerliOS</code> 가 특정 대안으로 제안되기도 했습니다. <code>pydotorg</code> 관리자들 자신은 추가 작업량에 반대하지 않았으며, 과중한 업무가 발생할 경우 저장소를 다시 마이그레이션하는 것이 대안으로 제시되었습니다.</li>
<li><strong>다른 인증 전략:</strong> <code>svn+ssh</code> 의 대안으로 다음과 같은 것들이 제안되었습니다.
<ul>
<li><strong>SSL 및 기본 인증을 사용한 웹DAV(WebDAV)를 통한 Subversion:</strong> <code>pydotorg</code> 에서 생성된 비밀번호를 사용자에게 메일로 보내는 방식이었습니다. 사람들이 이 접근 방식을 좋아하지 않았는데, 비밀번호를 기억할 수 없어 디스크에 저장해야 했기 때문이며, 이는 보안 위험이 됩니다.</li>
<li><strong>SSL 클라이언트 인증서를 사용한 웹DAV(WebDAV)를 통한 Subversion:</strong> 작동은 하겠지만, 인증 기관을 관리해야 하는 부담이 있었습니다.</li>
</ul>
</li>

<li>오프사이트 미러링/백업을 위해 저장소 <code>tarball</code> 다운로드 대신 <code>rsync</code> 사용을 제안했습니다.</li>
