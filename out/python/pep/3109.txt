<p><strong>원문 링크:</strong> <a href="https://peps.python.org/pep-3109/" target="_blank" rel="noopener noreferrer">PEP 3109 - Raising Exceptions in Python 3000</a></p>

<h2 id="pep-3109-python-3000에서의-예외-발생-raising-exceptions"><a href="#pep-3109-python-3000에서의-예외-발생-raising-exceptions">PEP 3109 – Python 3000에서의 예외 발생 (Raising Exceptions)</a></h2>
<ul>
<li><strong>작성자:</strong> Collin Winter</li>
<li><strong>상태:</strong> Final (최종)</li>
<li><strong>유형:</strong> Standards Track</li>
<li><strong>생성일:</strong> 2006년 1월 19일</li>
<li><strong>Python 버전:</strong> 3.0</li>
</ul>
<h3 id="초록-abstract"><a href="#초록-abstract">초록 (Abstract)</a></h3>
<p>이 PEP는 Python의 예외 발생(raising exceptions) 메커니즘에 대한 변경 사항을 소개합니다. 이 변경은 코드의 불필요한 복잡성(<code>line noise</code>)을 줄이고 언어의 크기를 축소하는 것을 목표로 합니다.</p>
<h3 id="배경-rationale"><a href="#배경-rationale">배경 (Rationale)</a></h3>
<p>Python의 핵심 원칙 중 하나는 "이를 수행하는 명확하고, 가급적이면 오직 하나의 방법만 존재해야 한다"는 것입니다. Python 2.x의 <code>raise</code> 문은 이 원칙을 위반하여 동일한 의도를 표현하는 여러 방법을 허용했습니다.</p>
<p>예를 들어, 다음 두 문장은 동일한 기능을 합니다.</p>
<pre><code class="language-python">raise E, V
raise E(V)
</code></pre>
<p>또한, 예외에 임의의 트레이스백(traceback)을 첨부할 수 있는 세 번째 <code>raise</code> 문 형식도 있었습니다.</p>
<pre><code class="language-python">raise E, V, T
</code></pre>
<p>여기서 <code>T</code>는 트레이스백입니다. PEP 344에 명시된 대로, Python 3.x의 예외 객체는 <code>__traceback__</code> 속성을 가지게 되며, 세 표현식 <code>raise</code> 문은 다음과 같이 번역될 수 있습니다.</p>
<p><code>raise E, V, T</code>는 다음과 같이 변환됩니다.</p>
<pre><code class="language-python">e = E(V)
e.__traceback__ = T
raise e
</code></pre>
<p>이러한 변환을 통해 <code>raise</code> 문은 네 가지 형식에서 두 가지 형식으로 줄어들 수 있습니다.</p>
<ol>
<li><strong><code>raise</code> (인수 없음):</strong> <code>except</code> 스위트(suite) 내에서 현재 활성화된 예외를 다시 발생시킬 때 사용됩니다.</li>
<li><strong><code>raise EXCEPTION</code>:</strong> 새로운 예외를 발생시킬 때 사용됩니다. 이 형식에는 두 가지 하위 변형이 있습니다. <code>EXCEPTION</code>은 예외 클래스 또는 예외 클래스의 인스턴스일 수 있습니다. 유효한 예외 클래스는 <code>BaseException</code>과 그 하위 클래스입니다 (PEP 352). 만약 <code>EXCEPTION</code>이 하위 클래스인 경우, 인수 없이 호출되어 예외 인스턴스를 얻습니다.</li>
</ol>
<p>그 외의 방식으로 예외를 발생시키려 하면 에러가 발생합니다.</p>
<p>이러한 통합을 통해 A.M. Kuchling이 언급한 더 실질적인 이점도 있습니다.</p>
<p>PEP 8은 두 가지 <code>raise</code> 문 형식 중 어떤 것을 선호하는지 명시하지 않았습니다.</p>
<pre><code class="language-python">raise ValueError, 'blah'
raise ValueError("blah")
</code></pre>
<p>두 번째 형식이 더 좋습니다. 그 이유는 예외 인수가 길거나 문자열 포매팅을 포함하는 경우, 괄호 때문에 줄 연속 문자(line continuation characters)를 사용할 필요가 없기 때문입니다.</p>
<p>BDFL(Benevolent Dictator For Life, 귀도 반 로섬)은 이러한 여러 <code>raise</code> 형식의 통합에 동의하고 지지했습니다.</p>
<h3 id="문법-변경-grammar-changes"><a href="#문법-변경-grammar-changes">문법 변경 (Grammar Changes)</a></h3>
<p>Python 3에서는 <code>raise</code> 문의 문법이 다음과 같이 변경됩니다.</p>
<p>Python 2의 문법:</p>
<pre><code>raise_stmt: 'raise' [test [',' test [',' test]]]
</code></pre>
<p>Python 3의 문법:</p>
<pre><code>raise_stmt: 'raise' [test]
</code></pre>
<h3 id="내장-타입-변경-changes-to-builtin-types"><a href="#내장-타입-변경-changes-to-builtin-types">내장 타입 변경 (Changes to Builtin Types)</a></h3>
<p>예외 발생과 관련하여, 제너레이터(generator) 객체의 <code>throw()</code> 메서드 시그니처도 변경되어 선택적 두 번째 및 세 번째 매개변수가 삭제됩니다. 이에 따라 시그니처는 (PEP 342에 따라) 다음과 같이 변경됩니다.</p>
<p>Python 2의 <code>generator.throw()</code> 시그니처:</p>
<pre><code class="language-python">generator.throw(E, [V, [T]])
</code></pre>
<p>Python 3의 <code>generator.throw()</code> 시그니처:</p>
<pre><code class="language-python">generator.throw(EXCEPTION)
</code></pre>
<p>여기서 <code>EXCEPTION</code>은 <code>BaseException</code>의 하위 클래스이거나 <code>BaseException</code>의 하위 클래스 인스턴스입니다.</p>
<h3 id="의미론적-변경-semantic-changes"><a href="#의미론적-변경-semantic-changes">의미론적 변경 (Semantic Changes)</a></h3>
<p>Python 2에서는 다음 <code>raise</code> 문이 유효했습니다.</p>
<pre><code class="language-python">raise ((E1, (E2, E3)), E4), V
</code></pre>
<p>인터프리터는 튜플의 첫 번째 요소를 예외 타입으로 (재귀적으로) 처리하여 위 코드가 다음 코드와 완전히 동일하게 작동하도록 했습니다.</p>
<pre><code class="language-python">raise E1, V
</code></pre>
<p>Python 3.0부터는 이와 같은 튜플을 예외로 발생시키는 기능이 지원되지 않습니다. 이 변경은 <code>raise</code> 문을 이미 이러한 방식(튜플 사용)을 허용하지 않는 제너레이터 객체의 <code>throw()</code> 메서드와 일치시킵니다.</p>
<h3 id="호환성-문제-compatibility-issues"><a href="#호환성-문제-compatibility-issues">호환성 문제 (Compatibility Issues)</a></h3>
<p>두 개 또는 세 개의 표현식을 사용하는 모든 <code>raise</code> 문과 제너레이터의 <code>throw()</code> 호출은 수정이 필요합니다. 다행히도, 이 경우 Python 2.x에서 Python 3.x로의 변환은 간단하며, Guido van Rossum의 <code>2to3</code> 유틸리티가 <code>raise</code> 및 <code>throw</code> fixers ()를 사용하여 기계적으로 처리할 수 있습니다.</p>
<p>다음과 같은 변환이 수행됩니다.</p>
<ul>
<li>0개 또는 1개의 표현식을 사용하는 <code>raise</code> 문은 그대로 유지됩니다.</li>
<li>두 개의 표현식을 사용하는 <code>raise</code> 문은 다음처럼 변환됩니다.
<pre><code class="language-python">raise E, V
# ->
raise E(V)
</code></pre>
</li>
<li>두 개의 표현식을 사용하는 <code>throw()</code> 호출은 다음처럼 변환됩니다.
<pre><code class="language-python">generator.throw(E, V)
# ->
generator.throw(E(V))
</code></pre>
(이 변환에는 주의사항이 있습니다. 섹션 #5 참조.)</li>
<li>세 개의 표현식을 사용하는 <code>raise</code> 문은 다음처럼 변환됩니다.
<pre><code class="language-python">raise E, V, T
# ->
e = E(V)
e.__traceback__ = T
raise e
</code></pre>
</li>
<li>세 개의 표현식을 사용하는 <code>throw()</code> 호출은 다음처럼 변환됩니다.
<pre><code class="language-python">generator.throw(E, V, T)
# ->
e = E(V)
e.__traceback__ = T
generator.throw(e)
</code></pre>
(이 변환에도 주의사항이 있습니다. 섹션 #5 참조.)</li>
</ul>
<p><code>E</code>가 튜플 리터럴(tuple literal)인 두 개 또는 세 개의 표현식 <code>raise</code> 문은 <code>2to3</code>의 <code>raise</code> fixer를 사용하여 자동으로 변환될 수 있습니다. 그러나 <code>E</code>가 함수 호출 결과와 같이 리터럴이 아닌 튜플인 <code>raise</code> 문은 수동으로 변환해야 합니다. <code>E</code>가 예외 클래스이고 <code>V</code>가 예외 인스턴스인 두 개 또는 세 개의 표현식 <code>raise</code> 문은 특별한 주의가 필요합니다. 이러한 경우는 두 가지로 나눌 수 있습니다.</p>
<ol>
<li><strong>인수 없는 <code>raise</code> 문의 확장된 버전으로 <code>raise E, V</code>를 사용하는 경우.</strong>
예시: <code>F</code>가 <code>E</code>의 하위 클래스라고 가정합니다.
<pre><code class="language-python">try:
    something()
except F as V:
    raise F(V)
except E as V:
    handle(V)
</code></pre>
이것은 다음과 같이 표현하는 것이 더 좋습니다.
<pre><code class="language-python">try:
    something()
except F:
    raise
except E as V:
    handle(V)
</code></pre>
</li>
<li><strong><code>raise E, V</code>를 사용하여 예외를 다른 클래스로 "캐스팅(casting)"하는 방법.</strong>
<code>distutils.compiler.unixcompiler</code>의 예시를 들어보겠습니다.
<pre><code class="language-python">try:
    self.spawn(pp_args)
except DistutilsExecError as msg:
    raise CompileError(msg)
</code></pre>
이것은 <code>PEP 344</code>에서 도입된 <code>raise ... from ...</code> 구문을 사용하여 다음과 같이 표현하는 것이 더 좋습니다.
<pre><code class="language-python">try:
    self.spawn(pp_args)
except DistutilsExecError as msg:
    raise CompileError from msg
</code></pre>
</li>
</ol>
