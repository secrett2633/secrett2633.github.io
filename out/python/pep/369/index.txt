<h2>PEP 369 – Post import hooks</h2>
<ul>
<li><strong>작성자</strong> : Christian Heimes </li>
<li><strong>상태</strong> : 철회됨 (Withdrawn)</li>
<li><strong>유형</strong> : Standards Track</li>
<li><strong>생성일</strong> : 2008년 1월 2일</li>
<li><strong>Python 버전</strong> : 2.6, 3.0</li>
<li><strong>이력</strong> : 2012년 12월 2일 (Post-History)</li>
</ul>
<h3>철회 공지 (Withdrawal Notice)</h3>
<p>이 PEP는 Python 3.3에서 <code>importlib</code>로의 마이그레이션 이후 많은 세부 설계가 더 이상 유효하지 않게 되어, 작성자에 의해 철회되었습니다.</p>
<h3>초록 (Abstract)</h3>
<p>이 PEP는 <code>import</code> 메커니즘에 <code>post import hook</code>을 추가하기 위한 개선 사항을 제안합니다. 이는 주로 Python 3.0에서 예상되는 추상 기본 클래스(Abstract Base Classes, ABCs)의 광범위한 사용을 지원하기 위한 것입니다.</p>
<p>이 PEP는 원래 지연 임포트(lazy imports)와 <code>post import hook</code>을 결합한 형태로 시작되었습니다. <code>python-dev</code> 메일링 리스트에서의 논의 끝에, PEP는 두 개의 개별 PEP로 분리되었습니다.</p>
<h3>배경 (Rationale)</h3>
<p>Python은 모듈이 성공적으로 로드된 후 코드를 실행하기 위한 <code>import</code> 메커니즘에 <code>hook</code>을 걸 수 있는 API를 제공하지 않습니다. PEP 302의 <code>import hook</code>은 모듈을 찾고 로드하는 것에 관한 것이었지만, <code>post import hook</code>으로 설계된 것은 아닙니다.</p>
<h3>사용 사례 (Use cases)</h3>
<p><code>post import hook</code>의 한 가지 사용 사례는 모듈 임포트 시 콜백(callbacks)에 대한 Alyssa (Nick) Coghlan의 초기 게시물에서 언급되었습니다. 이는 Python 3.0과 그 ABC를 개발하는 동안 발견되었습니다. <code>decimal.Decimal</code>과 같은 클래스를 ABC에 등록하고 싶었지만, 모든 인터프리터 시작 시 모듈이 임포트되어서는 안 된다는 점이 문제였습니다. Alyssa는 다음과 같은 예시를 제시했습니다.</p>
<pre><code class="language-python">@imp.when_imported('decimal')
def register(decimal):
    Inexact.register(decimal.Decimal)
</code></pre>
<p><code>register</code> 함수는 'decimal'이라는 모듈에 대한 콜백으로 등록됩니다. <code>decimal</code>이 임포트될 때, 이 함수는 모듈 객체를 인자로 받아 호출됩니다.</p>
<p>이 특정 예시는 실제로는 필요하지 않지만 (왜냐하면 <code>decimal.Decimal</code>은 2.6 및 3.0에서 적절한 <code>abstract Number</code> 기본 클래스를 상속할 것이기 때문입니다), 여전히 그 원리를 잘 보여줍니다.</p>
<h3>기존 구현 (Existing implementations)</h3>
<p>PJE의 <code>peak.util.imports</code>는 <code>post load hook</code>을 구현합니다. 이 PEP의 구현은 PJE의 것과 많은 부분을 공유하며, 그의 아이디어에 부분적으로 기반을 두고 있습니다.</p>
<h3>Post import hook 구현 (Post import hook implementation)</h3>
<p><code>Post import hook</code>은 모듈이 로드된 후에 호출됩니다. 이 <code>hook</code>들은 하나의 인자, 즉 모듈 인스턴스를 받는 호출 가능한 객체(callable)입니다. 이들은 'os' 또는 'os.path'와 같은 모듈의 점(dotted) 이름으로 등록됩니다.</p>
<p>호출 가능한 객체들은 <code>sys.post_import_hooks</code> 딕셔너리에 저장되며, 이는 이름(문자열)에서 호출 가능한 객체 목록 또는 <code>None</code>으로 매핑됩니다.</p>
<h4>상태 (States)</h4>
<ul>
<li><strong>등록된 <code>hook</code>이 없음</strong> : <code>sys.post_import_hooks</code>에 해당 모듈에 대한 엔트리가 없습니다.</li>
<li><strong><code>hook</code>이 등록되었고 모듈이 아직 로드되지 않음</strong> : <code>import hook</code> 레지스트리에 <code>sys.post_import_hooks["name"] = [hook1]</code>과 같은 엔트리가 포함되어 있습니다.</li>
<li><strong>모듈이 성공적으로 로드됨</strong> : <code>import</code> 메커니즘은 <code>sys.post_import_hooks</code>가 새로 로드된 모듈에 대한 <code>post import hook</code>을 포함하는지 확인합니다. <code>hook</code>이 발견되면, 등록된 순서대로 모듈 인스턴스를 첫 번째 인자로 받아 호출됩니다. 메서드가 예외를 발생시키면 <code>hook</code> 처리가 중단됩니다. 마지막으로, 오류가 발생하더라도 모듈 이름에 대한 엔트리는 <code>None</code>으로 설정됩니다.
추가적으로, <code>hook</code> 내부에서 알림 메서드가 호출될 때 무한 재귀를 방지하기 위해 모듈 객체의 새로운 <code>__notified__</code> 슬롯(slot)은 <code>True</code>로 설정됩니다. <code>PyModule</code>을 서브클래싱하지 않는 객체의 경우, 대신 새로운 속성이 추가됩니다.</li>
<li><strong>모듈을 로드할 수 없음</strong> : <code>import hook</code>은 호출되지도, 레지스트리에서 제거되지도 않습니다. 나중에 모듈을 로드할 수 있습니다.</li>
<li><strong><code>hook</code>이 등록되었지만 모듈이 이미 로드됨</strong> : <code>hook</code>은 즉시 실행됩니다.</li>
</ul>
<h4>불변성 (Invariants)</h4>
<p><code>import hook</code> 시스템은 특정 불변성을 보장합니다. (XXX: 원문에 미완성 부분)</p>
<h4>Python 샘플 구현 (Sample Python implementation)</h4>
<p>Python 구현은 다음과 같을 수 있습니다.</p>
<pre><code class="language-python">def notify(name):
    try:
        module = sys.modules[name]
    except KeyError:
        raise ImportError("Module %s has not been imported" % (name,))
    if module.__notified__:
        return
    try:
        module.__notified__ = True
        if '.' in name:
            notify(name[:name.rfind('.')])
        for callback in post_import_hooks[name]:
            callback(module)
    finally:
        post_import_hooks[name] = None
    # XXX (원문에 미완성 부분)
</code></pre>
<h3>C API</h3>
<h4>새로운 C API 함수 (New C API functions)</h4>
<ul>
<li><code>PyObject* PyImport_GetPostImportHooks(void)</code>
<code>sys.post_import_hooks</code> 딕셔너리를 반환하거나 <code>NULL</code>을 반환합니다.</li>
<li><code>PyObject* PyImport_NotifyLoadedByModule(PyObject *module)</code>
모듈이 요청되었음을 <code>post import</code> 시스템에 알립니다. 오류가 발생하면 <code>NULL</code>을 반환하고, 그렇지 않으면 동일한 모듈 객체에 대한 빌린 참조(borrowed reference)를 반환합니다. 이 함수는 모듈 자체에 대한 <code>hook</code>만 호출하고 그 부모 모듈에 대한 <code>hook</code>은 호출하지 않습니다. 이 함수는 <code>import lock</code>이 획득된 상태에서 호출되어야 합니다.</li>
<li><code>PyObject* PyImport_NotifyLoadedByName(const char *name)</code>
<code>PyImport_NotifyLoadedByName("a.b.c")</code>는 <code>a</code>, <code>a.b</code>, <code>a.b.c</code>에 대해 이 특정 순서로 <code>PyImport_NotifyLoadedByModule()</code>을 호출합니다. 모듈은 <code>sys.modules</code>에서 검색됩니다. 모듈을 검색할 수 없으면 예외가 발생하고, 그렇지 않으면 <code>modname</code>에 대한 빌린 참조가 반환됩니다. <code>hook</code> 호출은 항상 최상위 부모 모듈부터 시작합니다. <code>PyImport_NotifyLoadedByName()</code>의 호출자는 <code>import lock</code>을 보유해야 합니다!</li>
<li><code>PyObject* PyImport_RegisterPostImportHook(PyObject *callable, PyObject *mod_name)</code>
<code>mod_name</code> 모듈에 대한 새로운 <code>hook</code> 호출 가능 객체 <code>callable</code>을 등록합니다.</li>
<li><code>int PyModule_GetNotified(PyObject *module)</code>
<code>__notified__</code> 슬롯/속성의 상태를 반환합니다.</li>
<li><code>int PyModule_SetNotified(PyObject *module, int status)</code>
<code>__notified__</code> 슬롯/속성의 상태를 설정합니다.</li>
</ul>
<p><code>PyImport_NotifyLoadedByModule()</code> 메서드는 <code>import_submodule()</code> 내부에서 호출됩니다. <code>import</code> 시스템은 <code>import lock</code>이 획득되었고 부모 모듈에 대한 <code>hook</code>이 이미 호출되었는지 확인합니다.</p>
<h3>Python API</h3>
<p><code>import hook</code> 레지스트리와 두 개의 새로운 API 메서드는 <code>sys</code> 및 <code>imp</code> 모듈을 통해 노출됩니다.</p>
<ul>
<li><code>sys.post_import_hooks</code>
이 딕셔너리에는 <code>post import hook</code>이 포함됩니다.
<code>{"name" : [hook1, hook2], ...}</code></li>
<li><code>imp.register_post_import_hook(hook: "callable", name: str)</code>
<code>name</code> 모듈에 대한 새로운 <code>hook</code>을 등록합니다.</li>
<li><code>imp.notify_module_loaded(module: "module instance") -> module</code>
모듈이 로드되었음을 시스템에 알립니다. 이 메서드는 기존의 지연/연기된 <code>import</code> 확장(lazy/deferred import extensions)과의 호환성을 위해 제공됩니다.</li>
<li><code>module.__notified__</code>
모듈 인스턴스의 슬롯입니다. (XXX: 원문에 미완성 부분)</li>
</ul>
<p><code>when_imported</code> 함수 데코레이터 또한 <code>imp</code> 모듈에 있으며, 이는 다음와 동일합니다.</p>
<pre><code class="language-python">def when_imported(name):
    def register(hook):
        register_post_import_hook(hook, name)
    return register

@imp.when_imported(name)
def hook(module):
    pass
</code></pre>
<h3>미해결 문제 (Open issues)</h3>
<ul>
<li><code>when_imported</code> 데코레이터는 아직 작성되지 않았습니다.</li>
<li>코드에는 여러 <code>XXX</code> 주석이 포함되어 있습니다. 이는 주로 엣지 케이스(edge cases)에서의 오류 처리에 관한 것입니다.</li>
</ul>
<h3>하위 호환성 (Backwards Compatibility)</h3>
<p>새로운 기능과 API는 Python의 기존 <code>import</code> 시스템과 충돌하지 않으며, 대부분의 소프트웨어에 대해 하위 호환성 문제를 일으키지 않습니다. 그러나 PEAK 및 Zope와 같이 자체적인 지연 <code>import</code> 매직을 구현하는 시스템은 일부 규칙을 따라야 합니다.</p>
<p><code>post import hook</code>은 기존의 지연(deferred) 및 <code>lazy import</code> 시스템과 협력하도록 신중하게 설계되었습니다. PEP 작성자는 자체적인 <code>on-load-hook</code>을 새로운 <code>hook</code> API로 대체할 것을 제안합니다. 대안적인 <code>lazy</code> 또는 <code>deferred import</code>는 여전히 작동하겠지만, 구현은 <code>imp.notify_module_loaded</code> 함수를 호출해야 합니다.</p>
<h3>참조 구현 (Reference Implementation)</h3>
<p>참조 구현은 이미 작성되었으며 <code>py3k-importhook</code> 브랜치에서 사용할 수 있습니다. 이는 여전히 일부 정리 작업, 문서 업데이트 및 추가 유닛 테스트가 필요합니다.</p>
