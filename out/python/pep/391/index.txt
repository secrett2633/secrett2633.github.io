<p><strong>원문 링크:</strong> <a href="https://peps.python.org/pep-0391/">PEP 391 - Dictionary-Based Configuration For Logging</a></p>
<p><strong>상태:</strong> Final | <strong>유형:</strong> Standards Track | <strong>작성일:</strong> 15-Oct-2009</p>
</blockquote>
<p><strong>PEP 391 – 로깅을 위한 딕셔너리 기반 설정</strong></p>
<h2>개요</h2>
<p>이 PEP(Python Enhancement Proposal)는 딕셔너리를 사용하여 로깅(logging) 설정을 구성하는 새로운 방법을 제안합니다.</p>
<h2>도입 배경 (Rationale)</h2>
<p>현재 Python의 <code>logging</code> 패키지를 설정하는 방법은 두 가지입니다. 하나는 로깅 API를 사용하여 프로그래밍 방식으로 설정하는 것이고, 다른 하나는 <code>ConfigParser</code> 기반 설정 파일을 사용하는 것입니다.</p>
<ol>
<li>
<p><strong>프로그래밍 방식 설정의 한계:</strong></p>
<ul>
<li>최대한의 제어는 가능하지만, 설정이 Python 코드 내에 고정됩니다.</li>
<li>런타임(runtime)에 설정을 쉽게 변경하기 어렵습니다.</li>
<li>결과적으로 애플리케이션의 특정 부분에 대한 로깅 상세도(verbosity)를 유연하게 조절하는 기능이 제한됩니다.</li>
<li>이는 문제 진단을 위한 로깅의 유용성을 떨어뜨립니다. 특히 프로덕션 환경에서는 로깅이 유일한 진단 도구가 될 수 있습니다.</li>
</ul>
</li>
<li>
<p><strong><code>ConfigParser</code> 기반 설정의 한계:</strong></p>
<ul>
<li>사용은 가능하지만, <code>logging</code> 패키지의 모든 측면을 설정할 수는 없습니다. 예를 들어, <code>Filter</code>는 이 시스템을 사용하여 설정할 수 없습니다.</li>
<li><code>ConfigParser</code> 형식 자체가 일부 사용자들 사이에서 강한 불만을 야기합니다.</li>
<li>당시 Python 표준 라이브러리에서 유일하게 지원되는 설정 형식이었기 때문에 선택되었지만, 많은 사람들이 이 형식을 '낡았거나(crufty)' '보기 싫다(ugly)'고 여깁니다.</li>
</ul>
</li>
</ol>
<p><strong>새로운 딕셔너리 기반 설정의 필요성:</strong>
최근 Python 버전에는 표준 라이브러리에 JSON 지원이 포함되어 있으며, 이는 설정 형식으로도 활용될 수 있습니다. Google App Engine과 같은 다른 환경에서는 YAML이 애플리케이션 설정에 사용되며, 로깅 설정도 애플리케이션 설정의 필수적인 부분으로 간주됩니다. 표준 라이브러리에는 현재 YAML 지원이 없지만, JSON과 YAML은 모두 Python 딕셔너리로 역직렬화(deserialization)될 수 있으므로 공통된 방식으로 지원을 제공할 수 있습니다.</p>
<p>딕셔너리를 통해 로깅 설정을 전달하는 방법을 제공함으로써, JSON 및/또는 YAML 사용자뿐만 아니라 사용자 정의 설정 방법을 사용하는 사용자도 원하는 설정을 설명하는 공통 형식을 통해 로깅을 더 쉽게 설정할 수 있게 됩니다.</p>
<p>기존 <code>ConfigParser</code> 기반 설정 시스템의 또 다른 단점은 증분(incremental) 설정을 지원하지 않는다는 것입니다. 새로운 설정은 기존 설정을 완전히 대체합니다. 다중 스레드 환경에서 증분 설정에 대한 완전한 유연성을 제공하기는 어렵지만, 새로운 설정 메커니즘은 제한적인 증분 설정 지원을 가능하게 할 것입니다.</p>
<h2>사양 (Specification)</h2>
<p>사양은 두 부분으로 구성됩니다: API와 설정 정보를 전달하는 데 사용되는 딕셔너리의 형식(즉, 준수해야 하는 스키마).</p>
<h3>명명 (Naming)</h3>
<p>역사적으로 <code>logging</code> 패키지는 PEP 8을 준수하지 않았습니다. 향후 PEP 8을 준수하도록 패키지 내의 메서드 및 함수 이름을 변경하여 수정될 예정입니다. 그러나 균일성을 위해 제안된 API 추가 사항은 현재 <code>logging</code>에서 사용되는 방식과 일관된 명명 체계를 따릅니다.</p>
<h3>API</h3>
<p><code>logging.config</code> 모듈에 다음 기능이 추가될 것입니다.</p>
<ul>
<li><code>dictConfig()</code> 함수: 설정 정보를 담고 있는 딕셔너리를 단일 인자로 받습니다. 딕셔너리 처리 중 오류가 발생하면 예외(Exception)가 발생합니다.</li>
<li>이 API는 사용자 정의가 가능합니다 (API 사용자 정의 섹션 참조). 증분 설정은 별도의 섹션에서 다룹니다.</li>
</ul>
<h3>딕셔너리 스키마 - 개요 (Dictionary Schema - Overview)</h3>
<p>스키마를 자세히 설명하기 전에 객체 연결, 사용자 정의 객체 지원, 외부 및 내부 객체 접근에 대해 간략히 설명합니다.</p>
<h4>객체 연결 (Object connections)</h4>
<p>이 스키마는 로거(loggers), 핸들러(handlers), 포매터(formatters), 필터(filters)와 같이 객체 그래프(object graph)로 서로 연결된 로깅 객체 집합을 설명하는 것을 목표로 합니다. 따라서 스키마는 객체 간의 연결을 나타내야 합니다. 예를 들어, 특정 로거에 특정 핸들러가 연결되어 있다고 가정합니다. 설정 딕셔너리에서는 각 대상 객체에 고유한 <code>id</code>를 부여하고, 이 <code>id</code>를 소스 객체의 설정에서 사용하여 소스와 대상 객체 간의 연결을 나타냅니다.</p>
<p><strong>예시 (YAML 스니펫):</strong></p>
<pre><code class="language-yaml">formatters:
  brief: # 'brief' id를 가진 포매터의 설정
  precise: # 'precise' id를 가진 포매터의 설정
handlers:
  h1: # 핸들러 id
    formatter: brief
  h2: # 다른 핸들러 id
    formatter: precise
loggers:
  foo.bar.baz:
    handlers: [h1, h2]
</code></pre>
<p>위 예시는 <code>foo.bar.baz</code> 로거에 <code>h1</code>과 <code>h2</code> 두 핸들러가 연결되어야 함을 나타냅니다. <code>h1</code>의 포매터는 <code>brief</code> id로, <code>h2</code>의 포매터는 <code>precise</code> id로 설명됩니다.</p>
<p>로거의 <code>id</code>는 <code>foo.bar.baz</code>와 같이 프로그래밍 방식으로 로거에 대한 참조를 얻는 데 사용되는 로거 이름입니다. <code>Formatter</code>와 <code>Filter</code>의 <code>id</code>는 어떤 문자열 값(예: <code>brief</code>, <code>precise</code>)이든 될 수 있으며, 이들은 설정을 처리하고 객체 간의 연결을 결정하는 데만 의미가 있고, 설정 호출이 완료된 후에는 어디에도 유지되지 않는 일시적인 값입니다.</p>
<p>핸들러 <code>id</code>는 특별히 처리됩니다 (아래 '핸들러 ID' 섹션 참조).</p>
<h4>사용자 정의 객체 (User-defined objects)</h4>
<p>스키마는 핸들러, 필터, 포매터를 위한 사용자 정의 객체를 지원해야 합니다. (로거는 인스턴스마다 다른 유형을 가질 필요가 없으므로, 설정에서 사용자 정의 로거 클래스에 대한 지원은 없습니다.)</p>
<p>설정될 객체는 일반적으로 세부 설정이 포함된 딕셔너리로 설명됩니다. 로깅 시스템은 일부 경우에 문맥(context)에서 객체가 어떻게 인스턴스화될지 추론할 수 있지만, 사용자 정의 객체가 인스턴스화될 때는 시스템이 이를 알지 못합니다. 사용자 정의 객체 인스턴스화에 대한 완전한 유연성을 제공하기 위해, 사용자는 '팩토리(factory)'를 제공해야 합니다. 팩토리는 설정 딕셔너리를 인자로 받아 인스턴스화된 객체를 반환하는 호출 가능(callable) 객체입니다. 이는 특수 키 <code>'()'</code> 아래에 팩토리로의 절대 임포트 경로(absolute import path)를 제공함으로써 이루어집니다.</p>
<p><strong>예시:</strong></p>
<pre><code class="language-yaml">formatters:
  brief:
    format: '%(message)s'
  default:
    format: '%(asctime)s %(levelname)-8s %(name)-15s %(message)s'
    datefmt: '%Y-%m-%d %H:%M:%S'
  custom:
    '()': my.package.customFormatterFactory # 사용자 정의 팩토리
    bar: baz
    spam: 99.9
    answer: 42
</code></pre>
<p>위 YAML 스니펫은 세 개의 포매터를 정의합니다.</p>
<ul>
<li><code>brief</code>와 <code>default</code> 포매터는 <code>'()'</code> 키를 포함하지 않으므로, 표준 <code>logging.Formatter</code> 인스턴스가 생성됩니다.</li>
<li><code>custom</code> 포매터는 특수 키 <code>'()'</code>를 포함하며, <code>my.package.customFormatterFactory</code>라는 사용자 정의 팩토리 호출 가능 객체를 사용합니다. 이 팩토리는 나머지 키-값 쌍(<code>bar</code>, <code>spam</code>, <code>answer</code>)을 키워드 인자로 받아 호출됩니다.</li>
</ul>
<p>키 <code>'()'</code>는 유효한 키워드 파라미터 이름이 아니므로 호출에 사용되는 키워드 인자 이름과 충돌하지 않으며, 해당 값이 <code>callable</code>이라는 것을 기억하기 쉽게 합니다.</p>
<h4>외부 객체 접근 (Access to external objects)</h4>
<p>설정이 <code>sys.stderr</code>와 같은 외부 객체를 참조해야 하는 경우가 있습니다. 설정 딕셔너리가 Python 코드로 구성되면 간단하지만, 설정이 텍스트 파일(예: JSON, YAML)을 통해 제공될 때 문제가 발생합니다. 텍스트 파일에서는 <code>sys.stderr</code>와 리터럴 문자열 <code>'sys.stderr'</code>를 구별하는 표준적인 방법이 없습니다.</p>
<p>이러한 구분을 용이하게 하기 위해, 설정 시스템은 문자열 값에서 특정 특수 접두사를 찾아 특별히 처리합니다. 예를 들어, <code>'ext://sys.stderr'</code>라는 리터럴 문자열이 설정에서 값으로 제공되면, <code>ext://</code>가 제거되고 나머지 값은 일반적인 임포트 메커니즘을 사용하여 처리됩니다.</p>
<p>이러한 접두사 처리는 프로토콜 처리와 유사한 방식으로 이루어집니다: <code>^(?P&#x3C;prefix>[a-z]+)://(?P&#x3C;suffix>.*)$</code> 정규 표현식과 일치하는 접두사(<code>prefix</code>)를 찾는 일반적인 메커니즘이 있으며, 접두사가 인식되면 접미사(<code>suffix</code>)는 접두사에 따라 처리되고 처리 결과가 문자열 값을 대체합니다.</p>
<p>구현은 <code>ext://</code>와 같은 표준 접두사 세트를 제공하지만, 이 메커니즘을 완전히 비활성화하거나 특별한 처리를 위해 추가 또는 다른 접두사를 제공할 수도 있습니다.</p>
<h4>내부 객체 접근 (Access to internal objects)</h4>
<p>외부 객체 외에도 설정 내의 객체를 참조해야 할 때도 있습니다. 이는 설정 시스템이 알고 있는 항목에 대해 암묵적으로 수행됩니다. 예를 들어, 로거 또는 핸들러의 레벨(level)에 대한 문자열 값 <code>'DEBUG'</code>는 자동으로 <code>logging.DEBUG</code> 값으로 변환되며, <code>handlers</code>, <code>filters</code>, <code>formatter</code> 항목은 객체 <code>id</code>를 받아 적절한 대상 객체로 확인됩니다.</p>
<p>그러나 <code>logging</code>에 알려지지 않은 사용자 정의 객체의 경우, 더 일반적인 메커니즘이 필요합니다. 예를 들어, 다른 핸들러에 위임하는 대상(target)을 인자로 받는 <code>logging.handlers.MemoryHandler</code> 인스턴스를 생각해봅시다. 시스템이 이 클래스를 이미 알고 있으므로, 설정에서 주어진 <code>target</code>은 관련 대상 핸들러의 객체 <code>id</code>이기만 하면 시스템은 <code>id</code>로부터 핸들러를 확인합니다.</p>
<p>하지만 사용자가 <code>alternate</code> 핸들러를 가진 <code>my.package.MyHandler</code>를 정의하는 경우, 설정 시스템은 <code>alternate</code>가 핸들러를 참조한다는 것을 알지 못합니다. 이를 위해 사용자가 다음과 같이 지정할 수 있는 일반적인 해결(resolution) 시스템이 제공됩니다.</p>
<pre><code class="language-yaml">handlers:
  file: # 파일 핸들러 설정
  custom:
    '()': my.package.MyHandler
    alternate: cfg://handlers.file # 설정 내의 객체 참조
</code></pre>
<p>리터럴 문자열 <code>'cfg://handlers.file'</code>은 <code>ext://</code> 접두사가 붙은 문자열과 유사한 방식으로 해결되지만, 임포트 네임스페이스 대신 설정 자체를 탐색합니다. 이 메커니즘은 <code>str.format</code>이 제공하는 방식과 유사하게 점(.) 또는 인덱스(index)를 통한 접근을 허용합니다.</p>
<h4>핸들러 ID (Handler Ids)</h4>
<p>일부 로깅 설정은 원하는 효과를 얻기 위해 핸들러 레벨(handler levels)을 사용해야 합니다. 그러나 이름으로 항상 식별할 수 있는 로거와 달리, 핸들러는 증분 설정 호출을 통해 레벨을 변경할 수 있는 영구적인 핸들(persistent handles)이 없습니다.</p>
<p>따라서 이 PEP는 핸들러에 선택적 <code>name</code> 속성을 추가할 것을 제안합니다. 이 속성이 사용되면, 이름과 핸들러를 매핑하는 딕셔너리에 항목이 추가됩니다. (핸들러가 닫히면 이 항목은 제거됩니다.) 증분 설정 호출이 이루어질 때, 설정 값에 따라 핸들러 레벨을 설정하기 위해 이 딕셔너리에서 핸들러가 조회됩니다. 자세한 내용은 증분 설정 섹션을 참조하십시오.</p>
<p>이러한 "영구 이름(persistent name)" 기능은 <code>Filter</code>와 <code>Formatter</code>에도 제공될 수 있지만, 이들을 증분적으로 설정할 수 있도록 하는 강력한 근거는 없습니다. 실용성이 순수성보다 중요하기 때문에 <code>Handler</code>에만 이 새로운 <code>name</code> 속성이 부여됩니다. 설정에서 핸들러의 <code>id</code>는 그 <code>name</code>이 됩니다.</p>
<p>핸들러 이름 조회 딕셔너리는 설정 용도로만 사용되며, 패키지의 공개 API의 일부가 되지 않을 것입니다.</p>
<h3>딕셔너리 스키마 - 상세 (Dictionary Schema - Detail)</h3>
<p><code>dictConfig()</code>에 전달되는 딕셔너리에는 다음 키가 포함되어야 합니다.</p>
<ul>
<li><code>version</code>: 스키마 버전을 나타내는 정수 값으로 설정됩니다. 현재 유효한 값은 1뿐이지만, 이 키를 통해 하위 호환성을 유지하면서 스키마를 발전시킬 수 있습니다.</li>
</ul>
<p>다른 모든 키는 선택 사항이지만, 존재하는 경우 아래 설명된 대로 해석됩니다. 아래에서 '설정 딕셔너리(configuring dict)'가 언급되는 모든 경우, 사용자 정의 인스턴스화가 필요한지 확인하기 위해 특수 키 <code>'()'</code>가 검사됩니다. 만약 그렇다면, 위에서 설명된 메커니즘이 인스턴스화에 사용되고, 그렇지 않으면 문맥을 사용하여 인스턴스화 방법을 결정합니다.</p>
<ul>
<li>
<p><code>formatters</code>: 해당 값은 각 키가 포매터 <code>id</code>이고 각 값이 해당 <code>Formatter</code> 인스턴스를 구성하는 방법을 설명하는 딕셔너리가 됩니다.</p>
<ul>
<li>설정 딕셔너리는 <code>format</code> 및 <code>datefmt</code> 키(기본값은 <code>None</code>)를 찾아서 <code>logging.Formatter</code> 인스턴스를 구성하는 데 사용됩니다.</li>
</ul>
</li>
<li>
<p><code>filters</code>: 해당 값은 각 키가 필터 <code>id</code>이고 각 값이 해당 <code>Filter</code> 인스턴스를 구성하는 방법을 설명하는 딕셔너리가 됩니다.</p>
<ul>
<li>설정 딕셔너리는 <code>name</code> 키(기본값은 빈 문자열)를 찾아서 <code>logging.Filter</code> 인스턴스를 구성하는 데 사용됩니다.</li>
</ul>
</li>
<li>
<p><code>handlers</code>: 해당 값은 각 키가 핸들러 <code>id</code>이고 각 값이 해당 <code>Handler</code> 인스턴스를 구성하는 방법을 설명하는 딕셔너리가 됩니다.</p>
<ul>
<li>설정 딕셔너리는 다음 키를 찾습니다:
<ul>
<li><code>class</code> (필수): 핸들러 클래스의 완전한 이름.</li>
<li><code>level</code> (선택 사항): 핸들러의 레벨.</li>
<li><code>formatter</code> (선택 사항): 이 핸들러의 포매터 <code>id</code>.</li>
<li><code>filters</code> (선택 사항): 이 핸들러의 필터 <code>id</code> 목록.</li>
</ul>
</li>
<li>다른 모든 키는 핸들러의 생성자에 키워드 인자로 전달됩니다.</li>
</ul>
</li>
<li>
<p><code>loggers</code>: 해당 값은 각 키가 로거 이름이고 각 값이 해당 <code>Logger</code> 인스턴스를 구성하는 방법을 설명하는 딕셔너리가 됩니다.</p>
<ul>
<li>설정 딕셔너리는 다음 키를 찾습니다:
<ul>
<li><code>level</code> (선택 사항): 로거의 레벨.</li>
<li><code>propagate</code> (선택 사항): 로거의 전파(propagation) 설정.</li>
<li><code>filters</code> (선택 사항): 이 로거의 필터 <code>id</code> 목록.</li>
<li><code>handlers</code> (선택 사항): 이 로거의 핸들러 <code>id</code> 목록.</li>
</ul>
</li>
<li>지정된 로거는 명시된 <code>level</code>, <code>propagation</code>, <code>filters</code>, <code>handlers</code>에 따라 구성됩니다.</li>
</ul>
</li>
<li>
<p><code>root</code>: 루트 로거(root logger)의 설정이 됩니다. 구성 처리는 다른 로거와 동일하게 이루어지며, <code>propagate</code> 설정은 적용되지 않습니다.</p>
</li>
<li>
<p><code>incremental</code>: 설정이 기존 설정에 대한 증분(incremental)으로 해석될지 여부를 나타냅니다. 이 값은 기본적으로 <code>False</code>이며, 이는 지정된 설정이 기존 <code>fileConfig()</code> API에서 사용되는 것과 동일한 의미로 기존 설정을 대체한다는 의미입니다.</p>
<ul>
<li>지정된 값이 <code>True</code>인 경우, 설정은 아래 '증분 설정' 섹션에 설명된 대로 처리됩니다.</li>
</ul>
</li>
<li>
<p><code>disable_existing_loggers</code>: 기존 로거를 비활성화할지 여부. 이 설정은 <code>fileConfig()</code>의 동명 파라미터와 동일합니다. 이 파라미터가 없으면 기본값은 <code>True</code>입니다. <code>incremental</code>이 <code>True</code>인 경우 이 값은 무시됩니다.</p>
</li>
</ul>
<h3>작동 예시 (A Working Example)</h3>
<p>다음은 YAML 형식의 실제 작동하는 설정 예시입니다 (이메일 주소는 가상의 것입니다).</p>
<pre><code class="language-yaml">version: 1 # 스키마 버전
formatters:
  brief:
    format: '%(levelname)-8s: %(name)-15s: %(message)s'
  precise:
    format: '%(asctime)s %(name)-15s %(levelname)-8s %(message)s'
filters:
  allow_foo:
    name: foo # 'foo' 이름의 필터
handlers:
  console:
    class: logging.StreamHandler
    formatter: brief
    level: INFO
    stream: ext://sys.stdout # 외부 객체 sys.stdout 참조
    filters: [allow_foo]
  file:
    class: logging.handlers.RotatingFileHandler
    formatter: precise
    filename: logconfig.log
    maxBytes: 1024
    backupCount: 3
  debugfile:
    class: logging.FileHandler
    formatter: precise
    filename: logconfig-detail.log
    mode: a
  email:
    class: logging.handlers.SMTPHandler
    mailhost: localhost
    fromaddr: my_app@domain.tld
    toaddrs:
    - support_team@domain.tld
    - dev_team@domain.tld
    subject: Houston, we have a problem.
loggers:
  foo:
    level: ERROR
    handlers: [debugfile]
  spam:
    level: CRITICAL
    handlers: [debugfile]
    propagate: no
  bar.baz:
    level: WARNING
root: # 루트 로거 설정
  level: DEBUG
  handlers: [console, file]
</code></pre>
<h3>증분 설정 (Incremental Configuration)</h3>
<p>증분 설정에 대한 완전한 유연성을 제공하기는 어렵습니다. 예를 들어, 필터(filters) 및 포매터(formatters)와 같은 객체는 익명이므로, 일단 설정이 완료되면 설정을 보강할 때 이러한 익명 객체를 참조할 수 없습니다.</p>
<p>또한, 설정이 완료된 후 런타임에 로거, 핸들러, 필터, 포매터의 객체 그래프를 임의로 변경할 강력한 필요성은 없습니다. 로거와 핸들러의 상세도는 레벨(level) 설정(및 로거의 경우 전파(propagation) 플래그)만으로 제어할 수 있습니다. 다중 스레드 환경에서 객체 그래프를 임의로 안전하게 변경하는 것은 문제가 있으며, 불가능하지는 않지만 구현에 추가되는 복잡성만큼의 이점이 없습니다.</p>
<p>따라서, 설정 딕셔너리의 <code>incremental</code> 키가 존재하고 <code>True</code>인 경우, 시스템은 모든 <code>formatters</code> 및 <code>filters</code> 항목을 완전히 무시하고, <code>handlers</code> 항목의 <code>level</code> 설정, 그리고 <code>loggers</code> 및 <code>root</code> 항목의 <code>level</code> 및 <code>propagate</code> 설정만 처리합니다.</p>
<p><code>dictConfig()</code>가 기본값이 <code>False</code>인 <code>incremental</code> 키워드 인자를 받도록 하는 등 다른 수단으로 증분 설정을 제공하는 것도 물론 가능합니다. 설정 딕셔너리 내의 값을 사용하도록 제안하는 이유는, 설정이 소켓 리스너(socket listener)로 피클링된(pickled) 딕셔너리 형태로 전송될 수 있도록 하기 위함입니다. 이를 통해 장시간 실행되는 애플리케이션의 로깅 상세도를 애플리케이션을 중지하고 다시 시작할 필요 없이 시간이 지남에 따라 변경할 수 있습니다.</p>
<p><strong>참고:</strong> 실제 경험을 바탕으로 한 증분 설정 요구 사항에 대한 피드백은 특히 환영합니다.</p>
<h3>API 사용자 정의 (API Customization)</h3>
<p><code>dictConfig()</code> API는 모든 사용 사례에 충분하지 않을 수 있습니다. API 사용자 정의를 위한 조치는 다음을 제공함으로써 이루어집니다.</p>
<ul>
<li><code>DictConfigurator</code> 클래스: 생성자에는 설정에 사용되는 딕셔너리가 전달되며, <code>configure()</code> 메서드를 가집니다.</li>
<li><code>dictConfigClass</code> 호출 가능 객체: (기본적으로) <code>DictConfigurator</code>로 설정됩니다. 이는 필요한 경우 <code>DictConfigurator</code>를 적절한 사용자 정의 구현으로 대체할 수 있도록 제공됩니다.</li>
</ul>
<p><code>dictConfig()</code> 함수는 지정된 딕셔너리를 전달하여 <code>dictConfigClass</code>를 호출한 다음, 반환된 객체의 <code>configure()</code> 메서드를 호출하여 실제로 설정을 적용합니다.</p>
<pre><code class="language-python">def dictConfig(config):
    dictConfigClass(config).configure()
</code></pre>
<p>이는 모든 사용자 정의 요구 사항을 충족시켜야 합니다. 예를 들어, <code>DictConfigurator</code>의 서브클래스(subclass)는 자체 <code>__init__()</code>에서 <code>DictConfigurator.__init__()</code>을 호출한 다음, 후속 <code>configure()</code> 호출에서 사용 가능한 사용자 정의 접두사를 설정할 수 있습니다. <code>dictConfigClass</code>는 서브클래스에 바인딩(bound)되고, <code>dictConfig()</code>는 기본, 사용자 정의되지 않은 상태와 똑같이 호출될 수 있습니다.</p>
<h3>소켓 리스너 구현 변경 (Change to Socket Listener Implementation)</h3>
<p>기존 소켓 리스너 구현은 다음과 같이 수정될 것입니다. 설정 메시지가 수신되면 <code>json</code> 모듈을 사용하여 딕셔너리로 역직렬화(deserialize)를 시도합니다. 이 단계가 실패하면 메시지는 <code>fileConfig</code> 형식으로 가정하고 이전과 같이 처리됩니다. 역직렬화가 성공하면 <code>dictConfig()</code>가 호출되어 결과 딕셔너리를 처리합니다.</p>
<h3>설정 오류 (Configuration Errors)</h3>
<p>설정 중에 오류가 발생하면 시스템은 적절하게 설명하는 메시지와 함께 <code>ValueError</code>, <code>TypeError</code>, <code>AttributeError</code> 또는 <code>ImportError</code>를 발생시킵니다. 다음은 오류를 발생시키는 (완전하지 않을 수 있는) 조건 목록입니다.</p>
<ul>
<li>문자열이 아니거나 실제 로깅 레벨에 해당하지 않는 문자열 레벨</li>
<li>부울(boolean)이 아닌 <code>propagate</code> 값</li>
<li>해당하는 대상이 없는 <code>id</code></li>
<li>증분 호출 중 발견된 존재하지 않는 핸들러 <code>id</code></li>
<li>유효하지 않은 로거 이름</li>
<li>내부 또는 외부 객체를 해결할 수 없는 경우</li>
</ul>

<p><em>최종 수정일: 2025-02-01 08:59:27 GMT</em>## PEP 391 – 로깅을 위한 딕셔너리 기반 설정</p>
<h2>개요</h2>
<p>이 PEP(Python Enhancement Proposal)는 딕셔너리를 사용하여 로깅(logging) 설정을 구성하는 새로운 방법을 제안합니다.</p>
<h2>도입 배경 (Rationale)</h2>
<p>현재 Python의 <code>logging</code> 패키지를 설정하는 방법은 크게 두 가지입니다: 로깅 API를 통한 프로그래밍 방식 설정과 <code>ConfigParser</code> 기반 설정 파일 사용입니다.</p>
<ol>
<li>
<p><strong>프로그래밍 방식 설정의 한계:</strong></p>
<ul>
<li>최대한의 제어를 제공하지만, 설정이 Python 코드 내에 고정되어 런타임(runtime)에 쉽게 변경하기 어렵습니다.</li>
<li>이로 인해 애플리케이션의 특정 부분에 대한 로깅 상세도(verbosity)를 유연하게 조절하는 기능이 제한되어, 문제 진단에 어려움을 겪을 수 있습니다.</li>
</ul>
</li>
<li>
<p><strong><code>ConfigParser</code> 기반 설정의 한계:</strong></p>
<ul>
<li>사용은 가능하지만, <code>Filter</code>와 같은 <code>logging</code> 패키지의 모든 요소를 설정할 수 없습니다.</li>
<li><code>ConfigParser</code> 형식 자체가 일부 사용자들 사이에서 불만을 야기하며, 당시 표준 라이브러리에서 유일하게 지원되는 형식이었지만, 많은 사람들이 이를 '낡았거나(crufty)' '보기 싫다(ugly)'고 여깁니다.</li>
</ul>
</li>
</ol>
<p><strong>새로운 딕셔너리 기반 설정의 필요성:</strong>
최근 Python 버전은 표준 라이브러리에 JSON 지원을 포함하며, 이는 설정 형식으로도 활용될 수 있습니다. Google App Engine과 같은 환경에서는 YAML이 애플리케이션 설정에 사용됩니다. JSON과 YAML은 모두 Python 딕셔너리로 역직렬화(deserialization)될 수 있으므로, 딕셔너리 기반 설정은 이러한 형식을 사용하는 사용자뿐만 아니라 사용자 정의 설정 방법을 사용하는 사용자에게도 공통적이고 유연한 설정 방식을 제공할 것입니다.</p>
<p>또한, 기존 <code>ConfigParser</code> 기반 시스템은 증분(incremental) 설정을 지원하지 않아 새로운 설정이 기존 설정을 완전히 대체하는 문제가 있었습니다. 새로운 메커니즘은 제한적인 증분 설정 지원을 가능하게 하여, 다중 스레드 환경에서도 로깅 설정을 보다 유연하게 변경할 수 있도록 합니다.</p>
<h2>사양 (Specification)</h2>
<p>이 PEP의 사양은 API와 설정 정보를 전달하는 데 사용되는 딕셔너리 형식(스키마)으로 구성됩니다.</p>
<h3>명명 (Naming)</h3>
<p><code>logging</code> 패키지는 역사적으로 PEP 8을 준수하지 않았지만, 제안된 API 추가 사항은 현재 <code>logging</code>에서 사용되는 명명 체계와 일관성을 유지합니다.</p>
<h3>API</h3>
<p><code>logging.config</code> 모듈에 다음 함수가 추가될 예정입니다.</p>
<ul>
<li><code>dictConfig(config)</code>: 설정 딕셔너리를 단일 인자로 받아 로깅 설정을 적용합니다. 처리 중 오류가 발생하면 예외(Exception)를 발생시킵니다.</li>
<li>이 API는 사용자 정의가 가능하며, 증분 설정도 지원합니다.</li>
</ul>
<h3>딕셔너리 스키마 - 개요 (Dictionary Schema - Overview)</h3>
<p>스키마는 로거, 핸들러, 포매터, 필터 등 서로 연결된 로깅 객체 그래프를 설명합니다. 객체 간의 연결은 고유한 <code>id</code>를 사용하여 정의됩니다.</p>
<h4>객체 연결 (Object connections)</h4>
<p>설정 딕셔너리에서는 각 대상 객체에 고유한 <code>id</code>를 부여하고, 이를 소스 객체의 설정에서 참조하여 객체 간의 연결을 나타냅니다.</p>
<p><strong>예시 (YAML):</strong></p>
<pre><code class="language-yaml">formatters:
  brief:
  precise:
handlers:
  h1:
    formatter: brief
  h2:
    formatter: precise
loggers:
  foo.bar.baz:
    handlers: [h1, h2]
</code></pre>
<p>위 예시에서 <code>foo.bar.baz</code> 로거는 <code>h1</code>, <code>h2</code> 핸들러에 연결되며, 각 핸들러는 <code>brief</code> 또는 <code>precise</code> 포매터를 사용합니다.</p>
<p>로거의 <code>id</code>는 로거 이름(<code>foo.bar.baz</code>)이고, <code>Formatter</code>와 <code>Filter</code>의 <code>id</code>는 임의의 문자열 값(<code>brief</code>, <code>precise</code>)으로, 설정 처리 중에만 유효한 일시적인 값입니다.</p>
<h4>사용자 정의 객체 (User-defined objects)</h4>
<p>스키마는 핸들러, 필터, 포매터를 위한 사용자 정의 객체를 지원합니다. 사용자 정의 객체 인스턴스화를 위해, 설정 딕셔너리에 특수 키 <code>'()'</code>와 함께 객체를 인스턴스화할 '팩토리(factory)'의 절대 임포트 경로를 제공해야 합니다. 팩토리는 설정 딕셔너리의 나머지 항목들을 키워드 인자로 받아 호출됩니다.</p>
<p><strong>예시:</strong></p>
<pre><code class="language-yaml">formatters:
  custom:
    '()': my.package.customFormatterFactory # 사용자 정의 팩토리
    bar: baz
</code></pre>
<p>여기서 <code>'()'</code> 키는 해당 값이 <code>callable</code> 객체임을 나타내며, 키워드 인자 이름과 충돌하지 않도록 선택되었습니다.</p>
<h4>외부 객체 접근 (Access to external objects)</h4>
<p><code>sys.stderr</code>와 같은 설정 외부의 객체를 참조해야 할 경우, <code>ext://</code>와 같은 특수 접두사를 사용합니다. 예를 들어, <code>'ext://sys.stderr'</code>는 <code>sys.stderr</code> 객체로 해석됩니다. <code>ext://</code> 접두사는 제거되고 나머지는 일반적인 임포트 메커니즘으로 처리됩니다.</p>
<h4>내부 객체 접근 (Access to internal objects)</h4>
<p>설정 내의 객체를 참조할 때는 <code>cfg://</code> 접두사를 사용합니다. 예를 들어, <code>'cfg://handlers.file'</code>은 설정 딕셔너리 내의 <code>handlers</code> 섹션에 있는 <code>file</code> 핸들러를 참조합니다. 이 메커니즘은 점(.) 또는 인덱스(index)를 통한 접근을 허용합니다.</p>
<h4>핸들러 ID (Handler Ids)</h4>
<p>핸들러의 레벨을 증분적으로 변경할 수 있도록, 핸들러에 선택적인 <code>name</code> 속성을 추가할 것을 제안합니다. 설정에서 핸들러의 <code>id</code>가 그 <code>name</code>이 되며, 이 이름은 증분 설정 호출 시 핸들러를 조회하는 데 사용됩니다.</p>
<h3>딕셔너리 스키마 - 상세 (Dictionary Schema - Detail)</h3>
<p><code>dictConfig()</code>에 전달되는 딕셔너리는 다음 키를 포함해야 합니다.</p>
<ul>
<li><code>version</code>: 스키마 버전을 나타내는 정수(현재는 1).</li>
<li><code>formatters</code>: 포매터 <code>id</code>와 해당 <code>Formatter</code> 인스턴스 설정을 담는 딕셔너리. <code>format</code>, <code>datefmt</code> 키를 사용합니다.</li>
<li><code>filters</code>: 필터 <code>id</code>와 해당 <code>Filter</code> 인스턴스 설정을 담는 딕셔너리. <code>name</code> 키를 사용합니다.</li>
<li><code>handlers</code>: 핸들러 <code>id</code>와 해당 <code>Handler</code> 인스턴스 설정을 담는 딕셔너리. <code>class</code> (필수), <code>level</code>, <code>formatter</code>, <code>filters</code> 키를 포함합니다.</li>
<li><code>loggers</code>: 로거 이름과 해당 <code>Logger</code> 인스턴스 설정을 담는 딕셔너리. <code>level</code>, <code>propagate</code>, <code>filters</code>, <code>handlers</code> 키를 포함합니다.</li>
<li><code>root</code>: 루트 로거(root logger)의 설정. <code>propagate</code> 설정은 적용되지 않습니다.</li>
<li><code>incremental</code>: <code>True</code>로 설정되면 증분 설정으로 처리됩니다(기본값은 <code>False</code>).</li>
<li><code>disable_existing_loggers</code>: 기존 로거를 비활성화할지 여부(기본값은 <code>True</code>, <code>incremental</code>이 <code>True</code>이면 무시됨).</li>
</ul>
<h3>작동 예시 (A Working Example)</h3>
<p>PEP 391 문서에는 YAML 형식으로 작성된 상세한 로깅 설정 예시가 포함되어 있습니다. 이 예시는 <code>formatters</code>, <code>filters</code>, <code>handlers</code>, <code>loggers</code>, <code>root</code> 섹션을 사용하여 로깅의 다양한 측면(예: 콘솔 출력, 파일 로테이션, 이메일 알림 등)을 구성하는 방법을 보여줍니다.</p>
<h3>증분 설정 (Incremental Configuration)</h3>
<p><code>incremental</code> 키가 <code>True</code>인 경우, 시스템은 <code>formatters</code>와 <code>filters</code> 항목을 무시하고, <code>handlers</code> 항목의 <code>level</code> 설정, 그리고 <code>loggers</code> 및 <code>root</code> 항목의 <code>level</code> 및 <code>propagate</code> 설정만 처리합니다.
이는 장시간 실행되는 애플리케이션의 로깅 상세도를 애플리케이션을 중지하고 다시 시작할 필요 없이 런타임에 변경할 수 있도록 합니다.</p>
<h3>API 사용자 정의 (API Customization)</h3>
<p><code>dictConfig()</code> API의 사용자 정의를 위해 <code>DictConfigurator</code> 클래스와 <code>dictConfigClass</code> 호출 가능 객체가 제공됩니다. <code>dictConfigClass</code>를 사용자 정의 구현으로 대체하여 <code>dictConfig()</code>의 동작을 확장할 수 있습니다.</p>
<h3>소켓 리스너 구현 변경 (Change to Socket Listener Implementation)</h3>
<p>기존 소켓 리스너 구현은 설정 메시지를 수신했을 때 <code>json</code> 모듈을 사용하여 딕셔너리로 역직렬화를 시도하고, 성공하면 <code>dictConfig()</code>를 호출하여 처리하도록 수정될 것입니다.</p>
<h3>설정 오류 (Configuration Errors)</h3>
<p>설정 중에 유효하지 않은 레벨, 부울이 아닌 <code>propagate</code> 값, 존재하지 않는 <code>id</code>, 유효하지 않은 로거 이름, 객체 해결 실패 등 오류가 발생하면 <code>ValueError</code>, <code>TypeError</code>, <code>AttributeError</code> 또는 <code>ImportError</code>가 발생합니다.</p>
<h3>커뮤니티 논의 및 참조 구현 (Discussion in the community &#x26; Reference implementation)</h3>
<p>이 PEP는 <code>python-dev</code> 및 <code>python-list</code> 메일링 리스트에서 논의되었으며, 제안에 대한 전반적인 반대는 없었습니다.
변경 사항에 대한 참조 구현은 <code>dictconfig.py</code> 모듈과 단위 테스트로 제공됩니다.</p>
