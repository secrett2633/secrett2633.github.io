<h1>PEP 571 – manylinux2010 플랫폼 태그</h1>
<ul>
<li><strong>작성자:</strong> Mark Williams, Geoffrey Thomas, Thomas Kluyver</li>
<li><strong>BDFL-Delegate:</strong> Alyssa Coghlan</li>
<li><strong>토론:</strong> Distutils-SIG list</li>
<li><strong>상태:</strong> Superseded (교체됨)</li>
<li><strong>유형:</strong> Informational (정보성)</li>
<li><strong>주제:</strong> Packaging (패키징)</li>
<li><strong>작성일:</strong> 2018년 2월 5일</li>
<li><strong>Superseded-By:</strong> PEP 600</li>
<li><strong>결의:</strong> Distutils-SIG 메시지</li>
</ul>
<h2>개요 (Abstract)</h2>
<p>이 PEP는 PEP 513에 의해 도입된 <code>manylinux1</code> 태그를 계승하는 <code>manylinux2010</code> 플랫폼 태그의 생성을 제안합니다. 또한, PyPI와 pip가 호환 가능한 플랫폼에서 <code>manylinux2010</code> 배포판의 업로드, 다운로드 및 설치를 지원하도록 업데이트될 것을 제안합니다.</p>
<h2>배경 (Rationale)</h2>
<p>이름 그대로, <code>manylinux1</code> 플랫폼 태그는 많은 Linux 시스템에서 바이너리 확장 모듈 (binary extension modules) 설치를 현실로 만들었습니다. <code>cryptography</code> 및 <code>numpy</code>와 같은 라이브러리는 일반적인 아키텍처에서 설치가 취약한 개발 환경 및 빌드 툴체인에 의존하지 않게 되면서 Python 개발자들에게 더 쉽게 접근할 수 있게 되었습니다.</p>
<p><code>manylinux1</code> wheels는 포함된 확장 모듈이 하위 호환성 (backwards-compatibility) 정책의 혜택을 받을 만큼 오래된 버전 심볼 (versioned symbols)을 내보내는 소수의 시스템 레벨 공유 라이브러리 (system-level shared libraries)에만 링크되도록 허용함으로써 이식성을 달성합니다. 예를 들어, <code>manylinux1</code> wheel 내의 확장 모듈이 <code>glibc</code>에 의존하는 경우, 버전 2.5 이하로 빌드되어야 하며, 그 후에 필요한 심볼을 버전 2.5로 계속 내보내는 최신 <code>glibc</code> 버전을 제공하는 시스템에서 실행될 수 있습니다.</p>
<p>PEP 513은 작성 당시 가장 오래된 지원 CentOS 릴리스였던 CentOS 5.11에서 화이트리스트 (whitelisted) 공유 라이브러리와 해당 심볼 버전을 가져왔습니다. 아쉽게도 CentOS 5.11은 2017년 3월 31일에 지원이 종료되었고, 지속적인 사용에 대한 명확한 경고가 있었습니다. 이는 패키지가 구식 버전으로 남아 <code>manylinux1</code> Docker 이미지를 사용하는 Python 소프트웨어 패키지 작성자들의 노력을 방해한다는 것을 의미합니다.</p>
<p>이제 CentOS 6는 가장 오래된 지원 CentOS 릴리스이며, 2020년 11월 30일까지 유지보수 업데이트를 받습니다. 우리는 CentOS 6에서 파생된 새로운 PEP 425 스타일의 플랫폼 태그인 <code>manylinux2010</code>을 제안하고, <code>manylinux</code> 툴체인, PyPI 및 pip가 이를 지원하도록 업데이트될 것을 제안합니다.</p>
<p>이것은 원래 <code>manylinux2</code>로 제안되었지만, 버전 관리는 달력 연도 (Calendar Versioning 또는 CalVer)를 사용하도록 변경되었습니다. 이는 <code>manylinux2014</code> 이전에 가상의 <code>manylinux2017</code> 표준이 새로운 PEP를 통해 정의되거나, 이 PEP보다 오래되었지만 <code>manylinux1</code>보다 새로운 시스템을 대상으로 하는 <code>manylinux2007</code> 표준이 정의될 수 있는 등, 향후 <code>manylinux</code> 태그를 순서에 관계없이 정의하기 쉽게 만듭니다.</p>
<p>달력 버전 관리는 또한 어떤 Linux 배포판 버전이 어떤 태그를 지원하는지에 대한 대략적인 아이디어를 제공합니다. <code>manylinux2010</code>은 2010년 이후 출시된 대부분의 배포판 버전에서 작동할 것입니다. 그러나 이것은 단지 근사치일 뿐이며, 실제 호환성 규칙은 아래에 정의되어 있으며 일부 최신 배포판은 이를 충족하지 못할 수 있습니다.</p>
<h2>manylinux2010 정책</h2>

<h2>호환 Wheel 컴파일 (Compilation of Compliant Wheels)</h2>
<p><code>manylinux1</code>과 마찬가지로, <code>auditwheel</code> 도구는 <code>manylinux2010</code> Docker 컨테이너에서 <code>pip wheel</code> 또는 <code>bdist_wheel</code>로 빌드된 Linux wheel에 <code>manylinux2010</code> 플랫폼 태그를 추가합니다.</p>
<h3>Docker 이미지 (Docker Image)</h3>
<p>CentOS 6를 기반으로 하는 두 가지 <code>manylinux2010</code> Docker 이미지가 <code>manylinux2010</code> wheel로 안정적으로 변환될 수 있는 바이너리 Linux wheel을 빌드하기 위해 제공됩니다. x86_64 및 i686 이미지에는 새로운 컴파일러 스위트 (devtoolset-8의 <code>gcc</code>, <code>g++</code>, <code>gfortran</code>)와 최신 Python 및 pip 릴리스가 설치되어 있습니다.</p>
<h3>vsyscall이 없는 커널과의 호환성 (Compatibility with kernels that lack vsyscall)</h3>
<p>Docker 컨테이너는 사용자랜드 (userland)가 호스트 커널과 호환된다고 가정합니다. 불행히도, 점차 흔해지는 커널 구성은 x86_64 CentOS 6 Docker 이미지의 이 가정을 깨뜨립니다.</p>
<p><code>glibc</code> 버전 2.14 이하는 x86_64에서 <code>vsyscall</code>로 알려진 구식 시스템 호출 최적화를 커널이 제공하도록 요구합니다. 이 최적화를 위해 커널은 자주 호출되는 시스템 호출 (가장 주목할 만한 것은 <code>time(2)</code>)의 읽기 전용 페이지를 고정된 메모리 위치에 각 프로세스에 매핑합니다. <code>glibc</code>는 <code>vsyscall</code> 페이지 내의 적절한 오프셋으로 함수 포인터를 역참조하고 이를 호출함으로써 이러한 시스템 호출을 호출합니다. 이는 일반적인 시스템 호출 호출과 관련된 커널 호출 오버헤드를 피합니다. <code>vsyscall</code>은 커널이 최적화된 시스템 호출을 포함하는 재배치 가능한 가상 공유 객체인 <code>vDSO</code> (virtual dynamic shared object)를 대신 각 프로세스에 매핑하는 동등한 메커니즘을 선호하여 오랫동안 사용 중단되었습니다.</p>
<p><code>vsyscall</code> 페이지는 주소 공간 배치 무작위화 (ASLR, Address Space Layout Randomization)에 참여하지 않으므로 심각한 보안 문제를 내포합니다. 예측 가능한 위치와 내용은 반환 지향형 프로그래밍 (ROP, return-oriented programming) 공격에 사용되는 가젯 (gadgets)의 유용한 소스가 됩니다. 동시에, <code>vsyscall</code>에 의존하는 <code>glibc</code> 버전이 존재하지 않는 페이지로 시스템 호출 포인터를 역참조하려고 할 때 세그먼트 오류 (segmentation faults)가 발생하므로, <code>vsyscall</code>의 제거는 x86_64 ABI를 깨뜨립니다. 타협책으로 Linux 3.1은 실행 가능한 코드, 즉 ROP 가젯 재료를 줄인 "에뮬레이트된" <code>vsyscall</code>을 구현했습니다. <code>vsyscall=emulated</code>는 수년 동안 대부분의 배포판 커널에서 기본 구성이었습니다.</p>
<p>불행히도 <code>vsyscall</code> 에뮬레이션은 여전히 예측 가능한 코드를 신뢰할 수 있는 메모리 위치에 노출하며, 반환 지향형 프로그래밍에 계속 유용합니다. 대부분의 배포판이 이제 <code>vsyscall</code>에 의존하지 않는 <code>glibc</code> 버전으로 업그레이드되었기 때문에, <code>vsyscall</code>을 전혀 지원하지 않는 커널을 배포하기 시작했습니다.</p>
<p>CentOS 5.11과 6은 모두 <code>vsyscall</code> 페이지에 의존하는 <code>glibc</code> 버전 (각각 2.5 및 2.12.2)을 포함하므로, 이들을 기반으로 하는 컨테이너는 많은 배포판의 향후 릴리스와 함께 제공되는 커널에서 실행될 수 없습니다. 예를 들어, Travis CI가 <code>vsyscall</code> 인터페이스를 제공하지 않는 커널에서 작업을 실행하기 시작하면, Python 패키지 작성자는 <code>manylinux</code> wheels를 빌드하기 위해 Docker 이미지를 사용할 수 없게 됩니다.</p>
<p>우리는 <code>glibc</code> git 저장소에서 <code>vsyscall</code>에 대한 모든 의존성을 <code>manylinux2010</code> 이미지에 포함된 <code>glibc</code> 버전으로 백포트 (backport)하는 패치를 도출했습니다. <code>glibc</code>를 재빌드하고, 따라서 <code>manylinux2010</code> 이미지 자체를 빌드하려면 여전히 <code>vsyscall</code> 메커니즘을 제공하는 호스트 커널이 필요하지만, 결과 이미지는 이를 제공하는 호스트와 그렇지 않은 호스트 모두에서 실행될 수 있습니다. <code>vsyscall</code> 인터페이스는 실행 중인 프로세스에만 적용되는 최적화이므로, 이 수정된 이미지로 빌드된 <code>manylinux2010</code> wheels는 수정되지 않은 CentOS 6 시스템에서 빌드된 것과 동일해야 합니다. 또한 <code>vsyscall</code> 문제는 x86_64에만 적용되며, i686 ABI의 일부가 아닙니다.</p>
<h2>Auditwheel</h2>
<p><code>auditwheel</code> 도구는 <code>manylinux2010</code> wheels를 생성하도록 업데이트되었습니다. 그 동작과 목적은 PEP 513 이후로 변경되지 않았습니다.</p>
<h2>설치 프로그램의 플랫폼 감지 (Platform Detection for Installers)</h2>
<p>플랫폼은 PEP 513에 설명된 <code>_manylinux</code> 모듈에 <code>manylinux2010_compatible</code> 부울 (boolean) 속성을 정의할 수 있습니다. 해당 속성이 <code>False</code>인 경우 플랫폼은 <code>manylinux2010</code>과 호환되지 않는 것으로 간주됩니다.</p>
<p><code>_manylinux</code> 모듈을 찾을 수 없거나 <code>manylinux2010_compatible</code> 속성이 없는 경우, 도구는 <code>glibc</code> 검사로 대체될 수 있습니다. 플랫폼에 <code>glibc</code> 2.12 이상이 있는 경우, <code>_manylinux</code> 모듈이 다른 내용을 명시하지 않는 한 호환되는 것으로 가정합니다.</p>
<p>특히, 우리가 제안하는 알고리즘은 다음과 같습니다.</p>
<pre><code class="language-python">def is_manylinux2010_compatible():
    # Only Linux, and only x86-64 / i686
    from distutils.util import get_platform
    if get_platform() not in ["linux-x86_64", "linux-i686"]:
        return False

    # Check for presence of _manylinux module
    try:
        import _manylinux
        return bool(_manylinux.manylinux2010_compatible)
    except (ImportError, AttributeError):
        # Fall through to heuristic check below
        pass

    # Check glibc version. CentOS 6 uses glibc 2.12.
    # PEP 513 contains an implementation of this function.
    return have_compatible_glibc(2, 12)
</code></pre>
<h2>manylinux1 wheels와의 하위 호환성 (Backwards compatibility with manylinux1 wheels)</h2>
<p>PEP 513에서 설명했듯이, <code>manylinux1</code> 화이트리스트 라이브러리에 대해 지정된 심볼 버전은 상한선을 구성합니다. 동일한 원칙이 이 PEP에서 <code>manylinux2010</code>에 대해 정의된 심볼 버전에도 적용됩니다. 결과적으로 <code>manylinux1</code> wheels는 <code>manylinux2010</code> wheels로 간주됩니다. 따라서 <code>manylinux2010</code> 플랫폼 태그를 인식하는 <code>pip</code>는 <code>manylinux2010</code> wheels를 사용할 수 없을 때 (명시적으로 설정된 경우에도) <code>manylinux2010</code> 플랫폼용으로 <code>manylinux1</code> wheels를 설치할 것입니다.</p>
<h2>PyPI 지원 (PyPI Support)</h2>
<p>PyPI는 <code>manylinux1</code>을 허용하는 것과 동일한 방식으로 <code>manylinux2010</code> 플랫폼 태그를 포함하는 wheels의 업로드를 허용해야 합니다. <code>manylinux2010</code> wheels의 호환성을 확인하려고 시도해서는 안 됩니다.</p>
<h2>PEP 571 변경 요약 (Summary of changes to PEP 571)</h2>
<p>승인 후 받은 피드백을 바탕으로 이 PEP에 다음 변경 사항이 적용되었습니다.</p>
<ul>
<li><code>libgcc_s</code>의 최대 버전 심볼이 GCC_4.3.0에서 GCC_4.5.0으로 업데이트되어 32비트 CentOS 6 문제를 해결했습니다. 이는 x86_64의 <code>libgcc_s</code>가 GCC_4.3.0에서 GCC_4.5.0으로 추가 심볼을 가지고 있지 않으므로 x86_64에는 영향을 미치지 않습니다.</li>
</ul>
